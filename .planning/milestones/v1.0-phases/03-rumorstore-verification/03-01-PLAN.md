<objective>
Add the `RumorStore` convenience class to `applesauce-core` and prove the reused generic infrastructure (verification, kind-5 delete, models/timeline/filter) works end-to-end over unsigned `Rumor` events with focused unit tests (RUMOR-03, RUMOR-04, RUMOR-05).

This is the store half of the Part A gate; the cast half (RUMOR-06) is Plans 02-03.

Output: `packages/core/src/event-store/rumor-store.ts` (new `RumorStore`), its public re-export, a `rumor-store.test.ts` covering accept/reject + streams + kind-5 delete, and a `minor` changeset.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller-supplied rumor → `RumorStore` memory | `RumorStore.add(rumor)` accepts an unsigned `Rumor` from a caller (typically the output of an upstream gift-wrap unwrap). The store's ONLY integrity check is `verifyRumor` (`getEventHash(rumor) === rumor.id`); authorization/decryption is a different protocol layer's job (migration doc "Deletion Policy" rationale), already performed before the rumor reaches this store. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering | `RumorStore` accepting a rumor whose `id` does not match its serialized contents | medium | mitigate | `verifyRumor` is wired as the non-overridable default verifier via `super({ ...options, verifyEvent: verifyRumor })`; the RUMOR-03 reject test (`add` returns null for a mismatched `id`) is the acceptance proof. |
| T-03-02 | Tampering | Caller silently disabling verification with `new RumorStore({ verifyEvent: undefined })` | low | mitigate | Structurally impossible: the constructor type is `Omit<EventStoreOptions<Rumor>, "verifyEvent">`, so `verifyEvent` cannot be passed to `RumorStore` at all — a compile error, not a runtime bypass. |
| T-03-03 | Spoofing / Information Disclosure | `RumorStore` treated as an authorization boundary (it is not) | low | accept | Documented, accepted design (migration doc): the local store verifier only checks hash integrity; authorization is verified upstream by the layer that produced the rumor. No new mitigation in this phase; recorded so downstream consumers do not over-trust store membership. |

No package installs occur in this plan (no `T-03-SC` supply-chain checkpoint). Highest severity is medium (below the block-on:high threshold), so no blocking human checkpoint is required.
</threat_model>
