<objective>
Close the one residual CHAN-02 / ROADMAP Success Criterion 2 gap from 07-VERIFICATION.md. The typed `MissingChannelKeyError` guard (`requireChannelKey`) shipped in 07-03 is wired only into `sendMessage` (community.ts:849) and `sendEvent` (community.ts:835). Five other channel-plane write entry points — `sendThread` (:887), `replyToThread` (:894), `react` (:901), `editMessage` (:910), `deleteMessage` (:916) — open with `const epoch = this.channelEpoch(channelId);` and go straight to `bindToChannel(...)` + `publishToPlane({ plane: "channel", channelId }, ...)` with no guard. For a known-but-keyless private channel they therefore surface `planeKeyFor`'s generic, non-`instanceof`-catchable `Error("unknown channel")` — the exact "indistinguishable from a truly-unknown id" failure mode CHAN-02/D-06 exists to close.

This is a typed-error / DX gap, NOT a security regression: `deriveConcordKeys` still produces no `keys.channels` entry for a keyless private channel on any path, so no wrong-plane send or key leak occurs either way. The only defect is that a consuming app cannot `instanceof`-catch `MissingChannelKeyError` to gate reactions/edits/deletes/thread-replies the way it already can for the composer's `sendMessage`. The fix (07-REVIEW.md WR-01, already review-approved) is to add `this.requireChannelKey(channelId);` as the first statement of the five methods, mirroring the proven `sendMessage`/`sendEvent` guard exactly.

Purpose: Make every channel-plane write path throw the same typed, catchable rejection, so a client that lost access to a private channel can uniformly disable reactions/edits/deletes/thread-replies.
Output: Five guard call-sites + one regression test locking the whole surface.
</objective>
