<objective>
Settle the Phase 2 carry-forward WR-01: restore `castEvent`'s compile-time signature guard (dropped by Phase 2's necessary widening to `StoreEvent`) by splitting the strict public `castEvent` from a loose `@internal performCast`, gated on whether the cast's own declared event type requires a signature.

Purpose: After Phase 2, `castEvent(rumor, SignedOnlyCast, store)` compiles even though a signed-only cast will read `.sig` on an unsigned rumor at runtime. This plan reintroduces the guardrail as a strictly-safer, sig-based conditional (`CastEventInput<T>`) â€” NOT the naive exact-`T` conditional, which RESEARCH proved over-tightens and breaks concord's real narrowed-kind rumor cast. The stream operators keep their loose signatures by calling the internal `performCast`. This is the enabling change for RUMOR-06 (its test lives in Plan 03).

Output: `cast.ts` with `CastEventInput<T>` + `@internal performCast` + a sig-gated `castEvent`; `cast-stream.ts` rewired to `performCast`; a `minor` changeset. Zero runtime behavior change â€” only the compile-time input type of `castEvent` narrows.
</objective>
