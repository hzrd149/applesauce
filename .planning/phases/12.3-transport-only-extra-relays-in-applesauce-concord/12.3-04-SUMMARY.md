# Phase 12.3 Plan 04: ConcordCommunity transport-only extras Summary

**`ConcordCommunity` — the phase's primary site — gains `extraRelays` plus a private `transport()` merge helper wired into every non-reactive transport call site, reactive `connected$`/`authenticated$`/live-subscription derivation off the extras stream, a split invite-link dual-meaning local (URL stays protocol-only, bundle publish targets the merged set), and a refounding quorum that publishes wide while its majority threshold and ack attribution stay bound to the protocol relay set.**

## Accomplishments
- Added `ConcordCommunityOptions.extraRelays?: ExtraRelaysOption` (imported from plan 01's shared `helpers/relays.ts`), constructed a `private readonly extras: ExtraRelays` BEFORE any observable derivation, and added a private `transport(): string[]` immediately below `relays()` — the class's sole merge point; `relays()`'s body is untouched
- Rewired six non-reactive transport call sites to `this.transport()`: `syncContext()`'s `relays:` field, the fresh-public-channel catch-up's `ensureAuth` call, the channel-rotation rekey-wrap publish loop, the invite-link revoke publish, the channel-key (Direct Invite) grant publish, and `publishToPlane`'s publish
- `spawnPrivateChannel` now threads the community's ORIGINAL (unresolved) `extraRelays` option through to `ConcordPrivateChannel`'s option bag via a new `extraRelaysOption` private field, keeping `relays: this.relays()` untouched as the sub-engine's protocol base (D-13)
- Made `connected$`/`authenticated$` reactive: both `switchMap` over `this.extras.relays$` and route through `this.transport()`, with D-07 any-of/all-of caveats and the D-10 shared-source recommendation documented directly in JSDoc; `relay-auth.ts` itself was not modified
- Widened `openLive()`'s churn guard to a `${authors}|${targets}` signature computed from a single `targets = this.transport()` local reused for the guard, `registerStreamKeys`/`ensureAuth`, and the subscription target/wait-for-auth list; added a reactive `extrasSub` (guarded on an existing `liveSub`, mirroring plan 02/03's construction-timing precedent) that re-invokes `openLive()` on later extras emissions, disposed alongside the holder in `dispose()`
- Split `createInvite`'s single dual-meaning local into `protocolRelays` (feeds `buildInviteLink`) and `transportRelays` (feeds the bundle publish) with a comment naming the D-05/T-12.3-15 leak surface at the split; `refreshInviteBundles`'s shared local (which only ever fed a publish) became `transportRelays` outright
- `refound()` now publishes every wrap — gated root-roll/channel-rekey plus the unconditional compaction/snapshot wraps — to a `transportRelays` local, while the majority threshold and a new ack-attribution filter (`protocolRelaySet` built via `normalizeRelayUrl`) both read only the `relays` (protocol) local, so an always-acking extra can neither satisfy nor dilute the discoverability quorum (D-06); the ROTATE-09 block comment was extended to record this split and its rationale
- Full 49-file/303-test package suite and the full 18-package workspace build stay green with no changeset (D-15)

## Decisions Made
- `PublishResponse`'s relay-URL field is `from` (per `packages/relay/src/types.ts:78`), not `origin` as the plan's `<read_first>` pointer named it — the ack-attribution filter reads `r.from`, consistent with `community.test.ts`'s existing `okResponses` fixture shape
- `extrasSub` is guarded with `if (this.liveSub)` — identical to plan 02/03's precedent — since `ExtraRelays`'s internal `BehaviorSubject` always emits once synchronously at construction; an unguarded subscribe would open the live socket before `start()` ever runs, a timing regression D-14 forbids even though the relay-set *content* would still match with no extras configured
- `connected$`/`authenticated$` route their merge through `this.transport()` inside the `switchMap` body rather than the switchMap's own emitted value, keeping `transport()` the class's one literal merge point (D-04), mirroring plans 02/03
- `openLive()` computes the merged target set into a single `targets` local reused for the guard signature, `registerStreamKeys`/`ensureAuth`, and the subscription call/`waitForAuth`, so the guard can never disagree with what was actually dialled
- `refreshInviteBundles`'s shared local needed no protocol/transport split (verified it has no URL-builder consumer) — renamed straight to `transportRelays` rather than introducing an unused protocol local
