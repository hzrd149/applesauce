<objective>
Land the public API surface: one `extraRelays` option on `ConcordClientOptions`
(D-13's single injection point), threaded unresolved into every sub-engine option
bag, plus the client's own three transport call sites and the second dual-meaning
local — the invite-link join path, where one local feeds both the bundle request
(transport, merge) and the relay set that becomes the joined community's material
relays (protocol, must not merge).

The client also carries the phase's user-facing documentation burden (D-10): the
option's JSDoc is where an app learns that a not-yet-emitted Observable resolves to
the empty set rather than blocking, that each engine subscribes its own source so a
hot/shared source is recommended, and that extras are additive only — an app wanting
local-only traffic sets the existing fallback relays option instead (D-14).

Runs in wave 4 because it passes `extraRelays` into option bags declared by plans
03 and 04.

Purpose: make the feature reachable and correctly documented at the API boundary.
Output: the public option plus the client-level merges and the join-path split.

No changeset (D-15).
</objective>

  - what it is: additional relay endpoints used purely as network targets for all Concord-managed traffic — community and channel sync, live subscriptions, publishes, community-list and invite-list reads and writes, direct-invite watching, invite-bundle fetch/publish/revoke, and relay auth (D-12);

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app → public SDK option | The only place an app injects extras; everything downstream inherits from here |
| join path → persisted JoinMaterial | The relay set carried onward from a join becomes long-lived protocol state persisted and republished in the user's community list |
| community list publish | An encrypted, user-published document whose entries carry each community's material relays |
