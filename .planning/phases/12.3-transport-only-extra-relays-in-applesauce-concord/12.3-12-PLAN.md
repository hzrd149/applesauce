<objective>
Close the four findings the post-12.3-11 re-review recorded in

hardened, and two of them are defects in that hardening's own reach:

- **CR-01 (BLOCKER)** — `validateInviteBundle` caps `channels` on the raw array
  (correct) but leaves `held_roots` and each `channels[].held` array
  count-unbounded. `held_roots` flows into `JoinMaterial` (`client.ts:651`) ->
  `buildChain` (an `O(N²)` rebuild) -> `syncEpochs`, which performs one
  **sequential networked epoch sync per entry** (`sync.ts:331-333`). ~800 entries
  fit inside a NIP-44-capped bundle. The validator's own doc comment already
  claims the cap exists "to refuse an unbounded-allocation link", so the code
  contradicts its stated contract. `channels[].held` has the same omission, with
  one X25519 derivation per entry inside `deriveChannelKeys`.
- **CR-02 (BLOCKER)** — the bundle's top-level `name` is unvalidated. The
  `isValidChannelEntry` rationale ("`name` is display-only") is true of a
  *channel* name's key-derivation role but false of the document-size role of
  either name: `joinFromBundle` writes `bundle.name` into `JoinMaterial.name`,
  which is serialized **twice per entry** (`seed` + `current`) into the kind-13302
  document, hard-capped at 65535 bytes. One ~40KB name pushes the document over
  the cap, after which `saveCommunityList` returns without publishing —
  permanently, because `leave()` only appends a tombstone and never prunes
  `this.list`. `label`/`creator_npub` reach a published Guestbook Join by the
  same route.
- **WR-01 (WARNING)** — `community.ts:372` subscribes to the app's `extraRelays`
  source, then line 374 throws on malformed material, so `extras.dispose()` is
  never reached. 12.3-11's own skip-and-continue reconcile converted this from a
  once-per-reconcile leak into **one permanent subscriber per Community List
  emission**, on exactly the shared-`BehaviorSubject` shape this package's docs
  recommend. A warning made materially worse by the immediately preceding plan is
  worth closing in the plan that follows it.
- **WR-02 (WARNING)** — 12.3-11's defence-in-depth filter guards
  `fallbackRelays`, already safe on both callers, while `joinByBundle` — a public
  API — accepts a `bundle` with no validation at all. The added gate fires only
  on values that were already safe; the one branch carrying unvalidated attacker
  input is ungated.

**Why one plan, not two or four.** The findings' file sets intersect pairwise:
CR-01 and CR-02 both rewrite `validateInviteBundle`'s field-check block in
`invite-bundle.ts`; CR-02's recoverability half, WR-02 and WR-01's reconcile half
all edit `client.ts`. Same-wave plans must have zero `files_modified` overlap, so
any split produces serialized plans editing the same two functions back-to-back

immediately adjacent to CR-02's string guards) and inside `joinFromBundle`/
`joinByLink` (WR-02 moves the very line CR-02's join path reads). Splitting buys
no parallelism and adds ordering risk. One plan, three tasks, each with its own
independently-failing regression tests.

Purpose: every finding here is a defect in a boundary this phase exists to make
trustworthy — the one surface in this package where a stranger's bytes become the
user's persisted, published protocol state and the user's outbound network work.

Output: count bounds and length bounds on every attacker-controlled array and
string in a bundle, a validating `joinByBundle`, a relay gate relocated to the
branch that is actually untrusted, a recoverable Community List, a self-cleaning
`ConcordCommunity` constructor, and a reconcile loop that fails a bad entry once
rather than every emission.

### CR-02 fix shape (the question the planning guidance asks to answer explicitly)

The bound goes **at the validator**, not at `joinFromBundle`, for three reasons:
the validator is where this phase's established boundary-predicate pattern puts
untrusted-input rules (12.3-11's `patterns-established`); after Task 2's WR-02
fix both public join entry points route through it, so one bound covers both; and
`joinFromBundle` builds `JoinMaterial` from already-normalized fields, so a bound
there would be a second, drift-prone copy of the rule.

The unrecoverable-wedge property **does** need its own remediation and gets one
(Task 2's `leave()` prune). Bounding is necessary but not sufficient: a client
that already clicked such a link is wedged with the poisoned bytes sitting in a
purely local `this.list` — the entry never reached a relay precisely because the
publish is what fails, so it never propagated to another device, which means a
local prune fully recovers it. `leave()` is reachable for such an entry (a large
`name` does not break engine construction, so `this.communities.get(cid)` is
populated and `leave()` does not early-return). The prune is liveness-safe under
CORD-02 §8: the tombstone `leave()` already appends keeps the membership dead
even if another device later re-merges the entry's bytes, because
`mergeCommunity` keeps the older `added_at` and `isCommunityLive` compares it
against the newer `removed_at`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| encrypted invite bundle -> `validateInviteBundle` | A bundle decrypted with a token the attacker also supplied. The `community_id` owner proof is self-certifying and trivially satisfiable by an attacker who generates their own `owner`/`owner_salt`, so it authenticates NOTHING about array cardinality or string length. |
| validated bundle -> `JoinMaterial` -> `syncEpochs` | The point at which a stranger's chosen array length becomes the joining client's outbound network work: one sequential authenticated epoch sync per `held_roots` entry. |
| validated bundle -> `JoinMaterial` -> serialized kind-13302 document | The point at which a stranger's chosen string length becomes bytes inside a document with a hard 65535-byte publish cap, duplicated across `seed` and `current`. |
| `joinByBundle(bundle)` public API -> `joinFromBundle` | A public method whose argument is documented as pre-validated but, before this plan, enforced nothing — every field bypasses the validator. |
| application-supplied `extraRelays` Observable -> `ConcordCommunity` constructor | A long-lived object the HOST application owns; the package must never accumulate subscribers on it, and must never let a failed construction retain one. |
| another device's / an older client's Community List -> `reconcileCommunities` | Entries re-enter this client on every cast emission, including entries this client version cannot construct. |
