<objective>
Close two of the three verification gaps that live entirely inside
`packages/concord/src/helpers/relays.ts`: the crash-on-errored-source defect
(VERIFICATION.md derived robustness truth / REVIEW.md CR-03) and the false
"byte-identical when omitted" guarantee (VERIFICATION.md D-14 gap / REVIEW.md
WR-02). Also fix the two comparator/diagnostics warnings on the same file
(WR-01, WR-03).

Purpose: `extraRelays` is public API that accepts an arbitrary app-supplied
Observable. Today an error on that stream takes down the host process once per
engine and freezes every holder's snapshot with no signal. Separately, every
engine's transport target set changes shape even for a client constructed with
no `extraRelays` at all, contradicting the roadmap Success Criterion and five
JSDoc claims.

Output: a fail-soft, correctly-comparing, identity-preserving `ExtraRelays`
plus regression tests that fail against the current implementation, and the
two test fixtures this phase mutated restored to their pre-phase form.
</objective>

## D-14 resolution — decision and rationale

VERIFICATION.md offers two resolutions for the D-14 gap. **This plan takes
option (a): a genuine identity fast path in `ExtraRelays.merge`.**

Rationale:

1. **Smaller behavioral surface.** Option (a) touches exactly one method. Option
   (b) requires editing the ROADMAP.md Phase 12.3 Success Criteria, five JSDoc
   sites, and auditing every downstream consumer that keys on exact
   `pool.status$` / relay-target strings — a larger blast radius for a weaker
   guarantee.
2. **It restores the roadmap criterion as written**, so no ROADMAP.md Success
   Criteria edit is required and re-verification can match the failed truth
   statement verbatim.
3. **It is exactly pre-phase behavior.** Pre-phase, `this.relays()` was handed
   straight to `pool.publish` / `pool.subscription`. Returning `base` unchanged
   when the extras snapshot is empty reproduces that byte-for-byte, including
   the array reference semantics.
4. **It removes WR-03's silent-narrowing risk for the default configuration**
   entirely: with no extras configured, malformed protocol relays are no longer
   silently dropped from transport targets, because no merge runs at all.

Consequence accepted and handled in Task 3: two test fixtures this phase edited
to the newly-normalized form must be restored to the un-normalized form, since
those edits existed only to accommodate the behavior this task reverts.

**WR-09 explicitly deferred.** REVIEW.md WR-09 proposes that `ConcordClient`
hand its sub-engines `this.extras.relays$` instead of the raw option, collapsing
N subscriptions on the app's source to one. Deferred, with rationale: it changes
the value threaded through every sub-engine option bag (a D-13 threading change),
is a pure optimization rather than a correctness fix, and its stated risk — a
cold source re-executing per engine — is already documented in the public option
JSDoc and is fully mitigated for the crash case by Task 1 of this plan. It is a
follow-up candidate, not a gap.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host app -> `extraRelays` Observable | An arbitrary app-supplied stream crosses into library-owned subscription lifecycle; its error channel is app-controlled |
| protocol relay list -> transport target set | `material.relays` (attacker-reachable via an invite bundle) flows into `merge()`'s `base` argument |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-12.3-08-01 | Denial of Service | `ExtraRelays` constructor / `toRelaysObservable` | high | mitigate | Task 1: `catchError` substitutes an empty relay set and an explicit subscribe error handler logs and degrades, so an app-side stream failure can no longer raise N uncaught exceptions (one per engine) in the host process |
| T-12.3-08-02 | Denial of Service | `ExtraRelays.current` freeze after source error | medium | mitigate | Task 1: the internal subject is never errored or completed, so `connected$`, `authenticated$`, and the live-reopen subscriptions stay responsive instead of going permanently inert with no signal |
| T-12.3-08-03 | Tampering | `sameRelaySet` suppressing a genuine extras change | medium | mitigate | Task 1: deduplicated-set equality means an app narrowing or replacing its extras set can no longer have that change silently dropped, which would otherwise leave the engine dialling a de-configured endpoint |
