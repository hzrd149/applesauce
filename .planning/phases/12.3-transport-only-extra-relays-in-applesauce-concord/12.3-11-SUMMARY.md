# Phase 12.3 Plan 11: Invite/Join Path Gap Closure (CR-01, CR-02) Summary

**Closed the two BLOCKER gaps from 12.3-VERIFICATION.md: one shared `isSafeInviteRelayURL` predicate now gates every relay string an invite can carry into `JoinMaterial.relays` (CR-01/WR-01), and `validateInviteBundle`/`recordJoin`/`reconcileCommunities` now bound every field that reaches `hexToBytes` or epoch arithmetic, with construction ordered before document mutation so a rejected join leaves no residue (CR-02).**

## Accomplishments

- `isSafeInviteRelayURL` (new export, `helpers/invite-bundle.ts`) is the single scheme/shape gate applied at both places a stranger's bytes reach `JoinMaterial.relays` from an invite: `decodeFragment`'s terminal filter and `validateInviteBundle`'s relay filter — plus a defence-in-depth second gate on `joinFromBundle`'s fallback-relay branch in `client.ts`. Encrypted `wss://` passes the existing `isSafeRelayURL` shape check; plaintext `ws://` is accepted only for a loopback host (`localhost`, `127.0.0.1`, `[::1]`), closing WR-01 while preserving this project's own local-cache-relay use case.
- `validateInviteBundle` now rejects a bundle whose `community_root` isn't 64-char hex or whose `root_epoch` isn't a non-negative safe integer — the exact fields `baseKeysFor` feeds into `hexToBytes`/epoch arithmetic and the exact repro the verifier used by hand. Each `channels[]` entry (`id`/`key`/`epoch`, and any `held` keys) is shape-checked the same way and dropped individually if malformed; an optional `held_roots` is validated identically but rejects the WHOLE bundle when malformed, since `buildInviteBundle` never emits that field.
- `ConcordClient.recordJoin` now constructs the engine (`addCommunity`) BEFORE mutating `this.list`, so a synchronous throw during key derivation (a malformed bundle that reached `joinByBundle` directly, bypassing `validateInviteBundle`) leaves the document byte-identical to what it was — nothing phantom for `saveMirror()`/`saveCommunityList()` to persist or publish. `reconcileCommunities` wraps its `addCommunity` call in try/catch, logging and skipping an unconstructable Community List entry instead of aborting the loop or raising an unhandled rejection from the fire-and-forget `void this.reconcileCommunities()` call site.
- 23 new regression tests added across the two test files (10 in Task 1's scope, 9 in Task 2's, 4 in Task 3's), each observed FAILING against the pre-fix source before the corresponding fix landed (see Non-Vacuity Evidence below). Full suite: 354 pre-existing tests + 23 new = 377, all passing. Build (`tsc`) clean.

## Decisions Made

- WR-01's loopback carve-out is checked independently of `isSafeRelayURL` (not gated behind it) because `isSafeRelayURL`'s general hostname regex does not accept the bracketed IPv6 loopback literal (`[::1]`) this carve-out must admit — verified empirically (`isSafeRelayURL("ws://[::1]:4869")` returns `false`). `isSafeInviteRelayURL` therefore checks the anchored loopback regex FIRST, falling through to `entry.startsWith("wss://") && isSafeRelayURL(entry)` only when it doesn't match.
- A malformed `channels[]` entry (or a malformed entry inside its `held` array) is DROPPED — the containing bundle still validates minus that one channel — mirroring the existing junk-relay precedent (T-12.3-09-04) that a junk entry isn't grounds to reject the whole bundle.
- A malformed `held_roots` entry rejects the WHOLE bundle (not per-entry dropped), because `buildInviteBundle` deliberately never emits this field — any bundle carrying one is, by construction, not one this codebase minted, so there's no "legitimate partial" reading to preserve.
- Confirmed by reading `addCommunity`/`ConcordCommunity`'s constructor that constructing the engine does NOT read `this.list`: the `onMaterialChange` callback that eventually writes `this.list` (via `refreshCommunity`) is only invoked later by the running engine's own fold/sync logic, never during construction (`this.keys = deriveConcordKeys(options.material, [])` is the only synchronous work in the constructor that touches `material`). This makes the `recordJoin` reordering (construct-then-mutate) safe with no other call-site changes needed.
- `reconcileCommunities` skips (never prunes) an unconstructable entry from `this.list` — it may be legitimate material this client version simply cannot build yet (e.g., a newer field this version doesn't understand), and silently tombstoning another device's membership would be a worse failure than leaving it un-started.

## Issues Encountered

- The Task 3 "leaves no residue" test was initially written checking `storage.getItem()` immediately after the rejected `joinByBundle` call. On the first pre-fix run this assertion passed even without the fix (vacuous), because `saveMirror()` is only called from `joinFromBundle` AFTER `recordJoin` returns — and `recordJoin` throws before returning either way, so the mirror write never happens on that call in either the pre-fix or post-fix ordering. The test was corrected to force an explicit `client.saveCommunityList()` call after the failed join (surfacing whatever is left in the in-memory `this.list`, exactly as a later legitimate operation naturally would) and assert on the resulting published event instead of the mirror storage. Re-run against the reverted source confirmed this version genuinely fails pre-fix (phantom entry present in the published document) and passes post-fix. No production code was affected by this correction — test-authoring only.
