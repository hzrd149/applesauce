<objective>
Thread the transport-only extras through the two invite engines. Both are pure
transport consumers with no protocol writes of their own, so every relay set they
build is a legitimate merge target (D-12 explicitly names both: the revoke path's
bootstrap relays and the watcher's discovered inboxes).

The invite manager holds no socket, so every site here is a per-operation snapshot
(D-11). The invite watcher is a hybrid: its `refresh`/auth paths are per-operation,
but its live gift-wrap subscription is continuous and must react to later extras
emissions without churning on no-ops (D-09).

Runs in wave 2 alongside the private-channel work: neither engine touches the other's
files, and both must declare their new option before `client.ts` (wave 4) passes it.

Purpose: a cache relay should also cache the join/invite path, which is where it
helps most (CONTEXT.md motivating setup).
Output: `extraRelays` on both option bags plus a private merge helper in each class.

No changeset (D-15).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app-supplied extras → invite/DM inbox sockets | An app-local relay now receives the user's gift-wrap inbox requests and the user's NIP-42 AUTH |
| watcher discovery surface → app UI | The public discovered-inbox observable is consumed as protocol truth by apps |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-12.3-10 | Information Disclosure | extras appearing in the watcher's public discovered-inbox observable | high | mitigate | Explicit prohibition plus source assertion: only the relay-resolution method's output is ever pushed to that subject. Merging happens exclusively inside `transport(base)` at pool call sites, which have no path back into the subject. |
| T-12.3-11 | Spoofing | the user's identity-key NIP-42 AUTH performed against an app-supplied extras relay | medium | mitigate | D-03 rules extras into auth deliberately, and the watcher's user-auth path is already gated by the existing auto-authenticate flag (default off) — this plan changes the relay list, not the gate. An app that has not opted into user auth is unaffected. |
| T-12.3-12 | Information Disclosure | the user's inbox filter (their pubkey) disclosed to an app-local relay | low | accept | The app chose the extras; the same pubkey is already broadcast to the user's real inbox relays. No new secret is revealed. |
| T-12.3-13 | Denial of Service | live-socket churn on repeated extras emissions | low | mitigate | Content-de-duplicated `relays$` plus a reopen guard keyed on the merged set. |
| T-12.3-14 | Tampering | a hostile extras relay injecting bogus invite events | medium | mitigate | Unchanged existing controls: inbound events are validated/self-certified before use (the coordinate re-enforcement and bundle validation already in place), so widening the read set cannot promote unvalidated content. Confirm during implementation that no validation is bypassed on the merged path. |
