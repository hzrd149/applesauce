# Phase 12.3 Plan 03: Invite manager and invite watcher transport-only extras Summary

**`ConcordInviteManager` and `InviteWatcher` both gain `extraRelays` plus a private `transport()` merge helper — the manager merges per-operation onto its own set or (for revoke) the link's own bootstrap relays, and the watcher additionally merges into its per-relay user-AUTH loop and gains a reactive, churn-guarded live subscription that reopens on genuine `extraRelays` changes without leaking transport targets into its public discovery subject.**

## Accomplishments
- Added `ConcordInviteManagerOptions.extraRelays?: ExtraRelaysOption` and a private `transport(base?: string[])` helper; rewired the invite-list request, the invite-list publish, and the revoke-bundle publish — the revoke path's base stays the parsed link's own bootstrap relays (falling back to the manager's default set) with the base-vs-merged split named explicitly in the diff (D-12)
- Added `InviteWatcherOptions.extraRelays?: ExtraRelaysOption` (JSDoc'd as distinct from both `relays` and `inboxRelays`) and a private `transport(base: string[])` helper; rewired the historical invite request, the live subscription, and the per-relay NIP-42 user-AUTH loop so a gating extras relay also receives the user's AUTH (D-03)
- Widened `needsAuth$` to recompute over the merged set (triggered by `extras.relays$` alongside the existing sources) without ever pushing a merged value onto the public `relays$` discovered-inbox subject — that subject is fed only by `resolveRelays()`'s output, both before and after this plan
- Made the watcher's live subscription reactive per D-09: a new `extrasSub` re-invokes `openLive()` on later extras emissions (guarded to only fire once a live subscription already exists, mirroring plan 02's construction-timing guard), and `openLive()` itself gained a `pubkey|sorted-merged-relay-set` signature guard so a no-op re-emission cannot tear down and reopen the socket
- Full 49-file/303-test package suite and package build stay green with no changeset (D-15); one pre-existing test's mock `pool.status$` fixture key was corrected to the normalized URL form to match the real `RelayPool.relay()` normalization now flowing through the merged/normalized transport set

## Decisions Made
- `ConcordInviteManager` has no genuine dispose/teardown path (`stop()` is a restartable soft-stop, not a one-way lifecycle end), so per the plan's own fallback instruction the extras holder is left for GC rather than inventing a new lifecycle hook
- `InviteWatcher.stop()` DOES dispose the extras holder and unsubscribe `extrasSub`, per the plan's explicit instruction for this class — no test exercises a stop()-then-restart cycle, so this doesn't regress any covered behavior
- The revoke path's base selection is captured in two named locals (`bootstrapRelays`, `base`) before being wrapped in `this.transport(base)`, keeping the existing bootstrap-relay preference visibly unchanged in the diff
- `needsAuth$`'s `combineLatest` gained `this.extras.relays$` purely as a recomputation trigger; the `map()` ignores that positional value and calls `this.transport(relays)` instead, so `transport()` stays the class's one literal merge point (mirrors plan 02's identical decision on `connected$`/`authenticated$`)
- `openLive()`'s reopen-guard signature is `pubkey|sorted-merged-relay-set` — this engine has no "authors" list the way `ConcordPrivateChannel` does (its live filter targets one recipient pubkey, not a rotating authors set), so the pubkey stands in as the closest analogue to plan 02's guard shape

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Corrected a test fixture URL to its normalized form**
- **Found during:** Task 2 verification (`pnpm --filter applesauce-concord test -- invite-watcher`)
- **Issue:** Routing `needsAuth$` and `authenticateUser()`'s per-relay auth loop through `transport()` (as the plan requires) causes `mergeRelaySets` to normalize every relay URL — even with no `extraRelays` configured, since `mergeRelaySets` always normalizes both of its input sets. `new URL("wss://relay.example").toString()` always serializes with a trailing slash. The `autoAuthenticate:false — authenticateUser satisfies needsAuth$` test's mock `pool.status$` fixture used the literal `"wss://relay.example"` (no trailing slash) as its status-object key, which no longer matched the now-normalized URL passed through `transport()`, so the status lookup (`statuses[normalizeURL(url)] ?? statuses[url]`) missed entirely and `needsAuth$` read `false` where the test expected `true`.
- **Fix:** Confirmed this is a fixture inaccuracy, not a defect in the new code: `RelayPool.relay(url)` (`packages/relay/src/pool.ts`) always calls `normalizeURL(url)` before constructing/looking up a `Relay`, so a real `Relay.status$`'s `.url` field is always the normalized form. The test's hand-rolled mock previously worked only because the pre-phase code path never normalized the URL before comparing against the mock. Updated the mock's `pool.status$` key (and the nested `url` field) to `"wss://relay.example/"` to match what production `RelayPool` would actually produce.
- **Files modified:** `packages/concord/src/client/__tests__/invite-watcher.test.ts`
- **Commit:** `0779a154`
