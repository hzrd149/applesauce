# Phase 12.3 Plan 08: ExtraRelays fail-soft + D-14 byte-identity Summary

**`packages/concord/src/helpers/relays.ts` hardened against an errored app-supplied `extraRelays` Observable and given a genuine empty-extras identity fast path, closing CR-03/WR-01/WR-02/WR-03 with 10 new regression tests.**

## Performance

- **Duration:** ~25 min
- **Tasks:** 3
- **Files modified:** 3 (2 source, plus 2 test files — one of the test files was in the same file as a source module)

## Accomplishments

- `sameRelaySet` rewritten as deduplicated-Set equality — a duplicate entry on either side can no longer mask a genuine membership difference (previously `["x","y"]` vs `["x","x"]` was misreported as equal).
- `toRelaysObservable` is fail-soft: `catchError` ahead of `startWith` substitutes the empty relay set when an app-supplied source errors, logged through a new module-scope debug logger (`logger.extend("extra-relays")`, D-16). `ExtraRelays`'s constructor subscribe gains a matching explicit `error` handler as defense in depth — it degrades the internal subject to `[]` without ever erroring or completing it, so downstream consumers stay live.
- `ExtraRelays.merge` gained a genuine identity fast path: when the current extras snapshot is empty, `merge(base)` returns `base` by reference — no normalization, no deduplication, no new array — restoring the roadmap's literal "omitted extraRelays produces byte-identical transport targets" guarantee (D-14). With extras present, `merge` is unchanged: routed through `mergeRelaySets` as the module's sole merge primitive (D-02).
- `merge()` also emits a WR-03 diagnostic debug line (result unchanged) when the merge silently dropped unparseable protocol relay URLs, making that narrowing observable instead of silent.
- The one test fixture (`private-channel.test.ts`) that broke as a consequence of the identity fast path was restored to compare the no-extras subscription target directly against the configured relay constant, with no trailing-slash normalization mapping. `invite-watcher.test.ts`'s `pool.status$` fixture needed no change — confirmed empirically by re-running the full suite.
