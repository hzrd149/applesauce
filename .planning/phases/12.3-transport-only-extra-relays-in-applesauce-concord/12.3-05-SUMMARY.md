# Phase 12.3 Plan 05: ConcordClient public extraRelays option Summary

**`ConcordClientOptions.extraRelays` — the phase's single public API surface (D-13) — threaded unresolved into every sub-engine option bag, plus a client-level `transport()` merge point wired into the community-list read/publish and a split invite-link join-path local so the bundle fetch widens while the joined community's material relays stay unmerged (D-05).**

## Accomplishments
- Added `ConcordClientOptions.extraRelays?: ExtraRelaysOption`, positioned directly after the existing fallback `relays` option, with the phase's primary user-facing JSDoc covering scope (D-12), the never-written-to-protocol-state guarantee (D-01), additive-only semantics (D-14), the Observable/hot-source contract (D-10), and the D-07 `connected$`/`authenticated$` status consequences
- Stored the raw option unresolved on `extraRelaysOption` and threaded it into the invite manager's constructor call, `ensureInviteWatcher()`'s `InviteWatcher` construction, and `addCommunity()`'s `ConcordCommunity` construction — each sub-engine's own `relays:` field left untouched
- Added a private `transport(base?: string[])` helper (`this.extras.merge(base ?? this.defaultRelays)`) as the client's sole merge point, backed by a new `private readonly extras: ExtraRelays` constructed in the constructor
- Split `joinByLink`'s D-05 dual-meaning local: `relays` (bootstrap-or-default) stays unmerged and flows into `joinFromBundle` as the joined community's material relays; a new `requestRelays = this.transport(relays)` feeds only the invite-bundle fetch
- Routed the community-list read (`fetchList`) and publish (`saveCommunityList`) through `this.transport()`; the publish site computes the merged target set once into a `targets` local shared by a new D-16 trace log and the `pool.publish()` call
- Left `joinByBundle`'s `this.defaultRelays` protocol argument and `createNewCommunity`'s caller-supplied-or-default material relays argument textually unchanged
- Full 49-file/303-test package suite and package build stay green with no changeset (D-15)

## Decisions Made
- `ConcordClient.extras` is not disposed in `stop()` — `stop()` is a restartable soft-stop (`started = false`, `start()` can run again) and `extras` is a constructor-scoped `readonly` field with no re-construction path on restart, so disposing it would silently break a legitimate stop-then-start cycle. This mirrors plan 03's identical decision for `ConcordInviteManager`, which has the same restartable-stop shape (as opposed to `InviteWatcher`'s one-way `stop()`, which does dispose).
- The community-list publish site's `targets` local is computed once and reused for both the D-16 trace and the `pool.publish()` call, so the logged count can never disagree with what was actually dialled — mirrors `community.ts`'s `openLive()` targets-local convention from plan 04
- Task 1 deliberately did not construct the `extras: ExtraRelays` holder or the `transport()` method — that is Task 2's explicit responsibility per the plan's own split. Building the file after Task 1's edit alone surfaced a TS6133 unused-field/unused-import error, which confirmed (rather than corrected) that the field was properly deferred to Task 2, not a defect to fix within Task 1

## Deviations from Plan

None — plan executed exactly as written. The TS6133 unused-declaration errors encountered mid-Task-1 were an artifact of drafting both tasks' field declarations together before splitting the constructor logic per-task; resolved by moving the `extras`/`ExtraRelays` construction to Task 2 as the plan specifies, not a deviation from the plan's own scope.

## Issues Encountered
- Drafting Task 1 initially added both the `extraRelaysOption` field and the `extras: ExtraRelays` field/construction together (getting ahead of the plan's task split), which failed `pnpm --filter applesauce-concord build` with `noUnusedLocals` TS6133 errors on the then-unused `extras` field and `ExtraRelays` import. Corrected by deferring the `extras` field, its constructor line, and the `ExtraRelays` value import to Task 2, keeping Task 1 scoped to exactly the raw-option field and pass-through the plan specifies.
