<objective>
Close the two BLOCKER gaps recorded in `12.3-VERIFICATION.md`, both reproduced
by the verifier through direct execution:

- **CR-01** — `decodeFragment`'s `lead === 0xff` branch pushes an arbitrary
  decoded string onto `ParsedInvite.bootstrapRelays` with no URL validation, and
  `joinByLink`/`joinFromBundle` promote that array to `JoinMaterial.relays`
  whenever `bundle.relays` is empty. Plan 12.3-09's own `isSafeRelayURL` filter on
  `bundle.relays` made the empty-`bundle.relays` condition reliably
  attacker-triggerable, so this is a regression from this phase's diff.
- **CR-02** — `validateInviteBundle` never validates `community_root`,
  `root_epoch`, or `channels[]` entry shape, and `recordJoin` mutates `this.list`
  BEFORE calling `addCommunity`, which throws synchronously inside
  `hexToBytes(material.community_root)`. The result is a phantom membership that
  is published in the user's kind-13302 list and re-throws on every subsequent
  `reconcileCommunities` run.

**Why one plan, not two.** CR-01 and CR-02 both edit
`packages/concord/src/helpers/invite-bundle.ts` and
`packages/concord/src/client/client.ts` — the file sets are identical, so a split
could never share a wave (same-wave plans must have zero `files_modified`
overlap) and would produce two serialized plans editing the same two functions
back-to-back, with a real merge/ordering hazard inside
`validateInviteBundle` (CR-01 rewrites its relay filter; CR-02 adds field checks
immediately above it). Splitting therefore buys no parallelism and adds risk.
One plan, three tasks, each with its own independently-failing regression test.

Purpose: both gaps are untrusted-input defects on the invite/join path — the one
surface in this package where a stranger's bytes become the user's persisted and
published protocol state.

Output: a single shared relay predicate applied at every boundary, a complete
bundle field validator, an atomic `recordJoin`, a fault-tolerant reconcile loop,
and four regression tests each observed failing before its fix.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| invite-link fragment bytes -> `decodeFragment` | A URL fragment authored entirely by whoever shares the link; never seen by a server, never signed, never bounded before this phase's fix |
| encrypted invite bundle -> `validateInviteBundle` | A bundle decrypted with a token the attacker also supplied; the `community_id` owner proof is self-certifying and trivially satisfiable by an attacker who generates their own `owner`/`owner_salt`, so it authenticates nothing about the remaining fields |
| validated bundle -> `JoinMaterial` -> local mirror + published kind-13302 | The point at which a stranger's bytes become the user's own persisted and republished protocol state, and the refounding quorum's denominator |
| another device's Community List -> `reconcileCommunities` | Entries authored by a different client version (or by an earlier, unhardened join) re-enter this client on every list emission |
