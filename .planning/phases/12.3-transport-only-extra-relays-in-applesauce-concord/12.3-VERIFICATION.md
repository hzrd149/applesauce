### Gaps Summary

The phase's headline architectural goal — the transport/protocol split (D-01, D-02, D-04, D-05, D-12, D-13) — is genuinely and robustly implemented. In particular, the D-05 dual-meaning-local split is proven non-vacuous by an actual revert-and-observe test methodology recorded in plan 06's summary, which is stronger evidence than most phases produce. The reactivity/churn work (D-07/D-08/D-09/D-11) was initially deferred without behavioral tests across plans 02-04, but plan 07 closed every one of those deferrals with passing tests exercising a live `BehaviorSubject` through two emissions — this is a genuine, not superficial, closure.

However, three concrete bugs — independently reproduced in this verification via direct execution against the built code, matching the 12.3-REVIEW.md findings exactly — mean the phase's stated Success Criteria are not fully met:

1. **D-06/ROTATE-09 (refounding quorum) is broken on realistic inputs.** The majority threshold and the ack-attribution set are computed from differently-normalized views of the same relay list, and the normalization step itself throws on any malformed entry. Since `material.relays` is attacker-reachable via an invite bundle whose entries are never validated as URLs, a hostile or merely malformed invite link can either crash Refounding outright or make its quorum permanently unsatisfiable. This is the CORD-06 ban/exclusion mechanism — a security-critical operation — and the bug is new in this diff, not pre-existing.
2. **The roadmap's own "behavior byte-identical when the option is omitted" Success Criterion (D-14) is false.** `mergeRelaySets` always normalizes and dedupes, even against an empty extras array, so relay-target strings and `status$` keys change shape for every engine even when no `extraRelays` is configured at all — a fact the phase's own plan summaries inadvertently document by having to correct pre-existing test fixtures to the newly-normalized form.
3. **`ExtraRelays` crashes the host application if the app-supplied Observable errors**, and freezes its snapshot permanently with no signal. This not one of the enumerated D-01…D-16 decisions verbatim, but it directly undermines the basic safety contract of accepting an app-supplied stream as public API, and was independently confirmed to fire an uncaught exception.

None of these three represent the extras/protocol leak the phase was principally guarding against — that guarantee holds. But they are concrete, reproducible defects in security-critical (refounding) and basic-safety (crash-on-error) code paths introduced by this phase, matching the code review's own "critical" classification, and they block a clean pass of the phase's own stated Success Criteria (D-06 and D-14 verbatim).

**This looks like it needs closure, not an override.** These are implementation bugs with clear, narrow fixes already spelled out in 12.3-REVIEW.md (CR-01/CR-02/CR-03), not a deliberate design deviation — an override is not appropriate here.
