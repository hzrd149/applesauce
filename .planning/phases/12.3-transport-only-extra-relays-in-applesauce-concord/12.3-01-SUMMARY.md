# Phase 12.3 Plan 01: extraRelays resolution + merge helper Summary

**Shared `ExtraRelaysOption`/`sameRelaySet`/`toRelaysObservable`/`ExtraRelays` primitive in `packages/concord/src/helpers/relays.ts`, exported from the concord helpers barrel — a continuous (never `take(1)`) resolver plus a synchronous snapshot holder that merges exclusively through `applesauce-core`'s `mergeRelaySets`.**

## Accomplishments
- Created `toRelaysObservable`, a continuous `startWith([])` + `distinctUntilChanged(sameRelaySet)` resolver for `string[] | Observable<string[]>` that deliberately never applies a first-value-only operator, so long-lived Concord engines react to every later emission on an app-supplied extras Observable (D-10/D-11)
- Created `ExtraRelays`, a per-engine holder giving a synchronous `.current` snapshot, a shared `.relays$` hot stream, and a `.merge(base)` that is the sole merge entry point, routed through `mergeRelaySets` (D-02)
- Exported the module from the concord helpers barrel with no collision and no new changeset (D-15), package build and full 303-test suite stay green

## Decisions Made
- `sameRelaySet` mirrors the existing `sameSet`/`sameChannelViews` content-comparator convention already used in `packages/concord/src/client/community.ts`, rather than inventing a new comparison style
- The JSDoc explaining why `toRelaysObservable` omits a first-value-only operator was worded to avoid the literal substrings `take(1)` and `unwrap` (paraphrased as "first-value-only operator" / "one-shot resolution helper"), so the plan's own `grep -c "take(1)"` / `grep -c "unwrap"` acceptance checks pass cleanly against the file's prose as well as its code — an initial draft failed this grep on its own explanatory comment and was reworded
- Test fixtures for relay URLs use the already-normalized (trailing-slash) form as the baseline constant, with a separate no-trailing-slash variant reserved for the normalization-collapse assertion, since `new URL("wss://host")` always serializes with a trailing `/` — an initial draft's merge test asserted against the pre-normalization string and failed until corrected

## Deviations from Plan

None - plan executed exactly as written. The JSDoc wording adjustment and test-fixture normalization fix above were corrections made during the plan's own RED/GREEN verification loop (Task 1's `<behavior>`/`<acceptance_criteria>` already specified these exact checks), not new scope.

## Issues Encountered
- First draft of `toRelaysObservable`'s JSDoc literally contained the words `take(1)` and `unwrap` while explaining what the resolver deliberately avoids, which the module's own no-`take(1)`/no-`unwrap` grep acceptance criteria then flagged; reworded to prose with no literal substring match
- First draft of the merge-dedup test asserted against un-normalized relay URL strings; `mergeRelaySets` normalizes every entry (bare-host `wss://` URLs always gain a trailing `/` from `URL`'s own serialization), so the test's expected literals were corrected to the normalized form
