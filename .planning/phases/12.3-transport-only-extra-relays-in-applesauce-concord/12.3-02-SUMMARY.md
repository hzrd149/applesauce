# Phase 12.3 Plan 02: ConcordPrivateChannel transport-only extras Summary

**`ConcordPrivateChannel` gains `extraRelays` plus a private `transport()` merge helper, routing ensureAuth/live-subscription/sync-context through the merged set and re-deriving `connected$`/`authenticated$` reactively, with the existing author-signature churn guard widened (not replaced) to also key on the merged relay set.**

## Accomplishments
- Added `ConcordPrivateChannelOptions.extraRelays?: ExtraRelaysOption`, imported from plan 01's shared `packages/concord/src/helpers/relays.ts`, JSDoc'd as additive-only and never a substitute for `relays` (D-14)
- Added a private `transport(): string[]` method (`this.extras.merge(this.opts.relays)`) — the class's single merge point (D-04) — and rewired `openLive()`'s `ensureAuth` call, its `pool.subscription` target, and `syncContext()`'s `relays:` field to all read `this.transport()`
- Made `connected$`/`authenticated$` reactive: both now `switchMap` over `this.extras.relays$` and call `relayAuth`'s unchanged plain-`string[]` methods with `this.transport()`, with the D-07 any-of/all-of caveats spelled out in JSDoc directly on the two observables
- Preserved `openLive()`'s existing author-signature dedupe guard and widened its key to also cover the sorted merged transport set (D-09/Pitfall 4), and added a guarded `extrasSub` subscription that re-invokes `openLive()` on later extras emissions only once a live subscription already exists — so the extras holder's synchronous construction-time emission cannot prematurely open a socket before `start()`
- Disposal now unsubscribes the new extras-driven subscription and calls `this.extras.dispose()` alongside the existing `liveSub`/`authDrivers` teardown; full 49-file/303-test package suite stays green with no changeset (D-15)

## Decisions Made
- The extras-driven `openLive()` re-invocation subscription is guarded with `if (this.liveSub)` rather than firing unconditionally, because `ExtraRelays`'s internal `BehaviorSubject` always emits once synchronously at construction (even with no `extraRelays` configured) — an unguarded subscribe would open the live socket at construction time, before `start()`/`walk()` ever runs, which is a genuine timing regression the plan's D-14 byte-identical requirement forbids even though the relay-set *content* would still match
- `connected$`/`authenticated$` route their merge through `this.transport()` inside the `switchMap`, rather than calling `mergeRelaySets(options.relays, extra)` directly with the switchMap's emitted value, keeping `transport()` the literal single merge point in the class (D-04) even though both expressions are equal at that instant
- `transport()` is called separately at each of `openLive()`'s four sites (signature, `ensureAuth`, subscription target, log) rather than cached once in a local variable, so the source text literally reads `this.transport()` at each call site per the plan's acceptance-criteria wording
