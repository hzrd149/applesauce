## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** `.relays` keeps its exact current meaning everywhere in concord (protocol: `material.relays` → `options.relays` → `STOCK_RELAYS`). It is never widened. `extraRelays` is carried as an additional, separate field alongside it, mirroring the applesauce-loaders convention (`loaders/tag-value-loader.ts:36,73-81`, `address-loader.ts:154`, `event-loader.ts:111`): the option is declared as `extraRelays?: string[] | Observable<string[]>`, rides alongside the pointer's own relays, and is merged only at the point of use via `mergeRelaySets`. This is the structural guarantee: because `relays()` never returns an extra, an app-local endpoint can only appear where someone explicitly wrote `mergeRelaySets(...)`. Every protocol write is safe by default. (Superseded an earlier proposal to rename to `protocolRelays()`/`transportRelays()` — that rename is dropped. Field names stay `relays`.)
- **D-02:** Merge helper is `mergeRelaySets` from `applesauce-core` (`packages/core/src/helpers/relays.ts:48`) — already a concord dependency, and it normalizes URLs and dedupes for free.
- **D-03:** `extraRelays` do participate in NIP-42 auth. `ensureAuth` runs over the merged set at `community.ts:657,682`, `sync.ts:137,191`, `channel-sync.ts:49,96`, `private-channel.ts:289`, so a gating cache relay still receives stream-key AUTH and can serve private-channel traffic.
- **D-04:** The `mergeRelaySets` call lives in a private per-class helper (e.g. `ConcordCommunity.transport()`) rather than being spelled out inline at each of ~20 pool call sites. One place resolves the extras snapshot and dedupes; call sites stay one-liners. This is a deliberate concord-scale adaptation of the loaders' inline style, justified by the site count.
- **D-05:** Regression guard is canary + targeted, both.
  - Canary: construct a client with `extraRelays: ["wss://canary.invalid"]`, drive create-community → create-invite-link → `joinByLink` → metadata edit → refound, and assert the canary string appears in zero published event payloads and zero returned artifacts. Catches leak paths nobody enumerated, including ones added in later phases.
  - Targeted: a spec-derived assertion at each enumerated protocol write (below).
  - Critical: these assert on event content and returned artifacts, never on publish targets — extras should appear as publish destinations; that is the feature.
  - TEST-01 (standing) applies: expectations derived independently, non-vacuity proven.
- **The two dual-meaning locals (the actual leak surface):**
  - `community.ts:1154-1169` (`createInviteLink`): `inviteRelays` feeds both `pool.publish` (transport → merge) and `buildInviteLink` (PROTOCOL → must not merge).
  - `client.ts:481-514` (`joinByLink`): `relays` feeds both `pool.request` (transport → merge) and `joinFromBundle` → `material.relays` (PROTOCOL, `:546`).
  - Other protocol writes to keep clean: `buildInviteBundle(this.material, ...)` (`community.ts:1142`), `createNewCommunity` → `material.relays` (`client.ts:465`, explicit caller argument — already safe), the metadata edit path (`admin.ts:157`), and anything surfaced on `state$.metadata`. Concord publishes no NIP-65/10050 itself, so the roadmap's "user-published relay lists" clause is reached only via `material.relays` inside the kind-13302 community list (`client.ts:878`) — covered by the canary.
- **D-06 (Refounding quorum, ROTATE-09 must not move):** Publish to the merged set, count protocol responses only. `refound()` at `community.ts:1358-1364` publishes each gated wrap to the merged set (so a cache relay does receive rekey/compaction/snapshot wraps), but the gate filters responses by `from ∈ relays()` (URL-normalized — `PublishResponse = { ok, message?, from }`, `packages/relay/src/types.ts:78`) and keeps `majorityThreshold = Math.ceil((relays().length + 1) / 2)`. Rationale: publish acks are the only evidence anyone else can discover the new epoch. Counting extras would let a user satisfy discoverability against their own always-OK localhost relay while no community relay accepted the wrap; inflating the denominator would raise a 2-of-3 bar to 3-of-4 and could make refounding permanently impossible. The arithmetic must stay bit-for-bit what Phase 8 settled on.
- **D-07 (Connection/auth status observables):** `connected$` and `authenticated$` derive from the merged set (user ruling, made with consequences below in view — document in JSDoc, don't mitigate):
  - `connected$` is `.some()` (`relay-auth.ts:88`) — an always-up local relay pins it true, so every community relay can be down while the UI still reads "live".
  - `authenticated$` is `.every()` over connected relays (`relay-auth.ts:109-112`). An extra that gates nothing short-circuits true at line 110 (harmless), but since D-03 has extras participating in AUTH, an extra that does gate and rejects our stream keys pins it false indefinitely.
- **D-08:** Those two are built once at construction from a snapshot (`community.ts:360-361`; `RelayAuth.connected$/authenticated$` take a plain `string[]` and close over it), so they must be re-derived reactively: switchMap at the community — `extraRelays$.pipe(switchMap(extra => relayAuth.connected$(mergeRelaySets(this.relays(), extra))))`. `RelayAuth`'s public signatures stay plain `string[]`; only the construction sites change.
- **D-09:** Live subscriptions are rebuilt via switchMap over the extras observable: `extraRelays$.pipe(switchMap(extra => pool.subscription(merged, ...)))`. Consistent with D-08. Constraint: `openLive()`'s existing dedupe guard (`community.ts:645-648`, `if (sig === this.liveAuthors && this.liveSub) return`) keys on the author signature only — its churn-avoidance intent must be preserved in the reactive form (e.g. `distinctUntilChanged` over authors + merged set), or a no-op emission will needlessly tear down and reopen sockets.
- **D-10:** Consumption shape is `unwrap` + `startWith([])`, following `packages/loaders/src/helpers/loaders.ts:18-20`. `startWith([])` means a not-yet-emitted Observable resolves to the empty set and never blocks Concord traffic (a `ConcordClient` is long-lived, unlike a one-shot loader). Known consequence to document: a cold Observable's producer is re-subscribed per operation and per sub-engine — apps should pass a hot/shared source (`BehaviorSubject`, or `shareReplay(1)`).
- **D-11:** `startWith([])` applies everywhere; `take(1)` applies only to one-shot operations. Applying `take(1)` at a long-lived site would freeze it at the first value and silently undo D-08/D-09.
  - Continuous (no `take(1)`): community live sub (`community.ts:659`), private-channel live sub (`private-channel.ts:292`), invite-watcher live sub (`invite-watcher.ts:364`), and `connected$`/`authenticated$` (`community.ts:360-361`, `private-channel.ts:80-82`).
  - Per-operation snapshot: publishes, `pool.request`, `syncAuthors`/`syncEpoch` backfill, invite-bundle fetch.
  - Rule: anything holding a socket or emitting over time is continuous.
- **D-12:** Extras union onto every pool call, no exceptions — including relay sets Concord did not derive from community material: invite-link bootstrap fetch (`client.ts:492`), invite-manager revoke to the link's own bootstrap relays (`invite-manager.ts:206-207`), user-list reads/writes on `defaultRelays` (`client.ts:743,878`), and invite-watcher's discovered NIP-17/NIP-65 inboxes (`invite-watcher.ts:213,364`). Matches the roadmap's "all Concord-managed traffic"; a cache relay then also caches the join path, where it helps most.
- **D-13:** Client-level configuration only. A single `extraRelays` on `ConcordClientOptions`, threaded down as a pass-through field into every community, private-channel, invite-manager and invite-watcher option bag. One source of truth, one leak surface to reason about. Per-community override is deferred.
- **D-14:** extras are purely additive, always — they never suppress or substitute for anything, and the fallback chain (`material.relays` → `options.relays` → `STOCK_RELAYS`) is untouched. An app wanting local-only traffic passes `relays: ["ws://localhost:4869"]` as the default set, which is the existing supported mechanism.
- **D-15:** concord is unreleased — no changeset, and breaking-change caution does not apply. See `[[concord-unreleased-no-changesets]]`. Carried from Phase 12.2 D-10.
- **D-16:** The namespaced `debug` logging landed in Phase 12.2 (`applesauce:concord:*`) is available; new relay-merge plumbing should log under it where useful. Derive/store the `Debugger` once — never `.extend()` inline at a log call site. See `[[debug-logger-hoist-extend]]`.

## Summary

This phase adds one option, `extraRelays?: string[] | Observable<string[]>`, to `ConcordClientOptions`, threads it as a pass-through field into every sub-engine (`ConcordCommunity`, `ConcordPrivateChannel`, `ConcordInviteManager`, `InviteWatcher`), and unions it into every transport-facing `pool.request`/`pool.subscription`/`pool.publish`/`ensureAuth`/`relayAuth.connected$`/`relayAuth.authenticated$` call across the package — while leaving every place that writes relay URLs into signed/published content (`buildInviteLink`, `material.relays`, `buildInviteBundle`, the kind-13302 community list) reading only from the existing `.relays`/`defaultRelays` accessors, untouched.

The codebase already carries the two building blocks this phase needs, both already concord dependencies: `mergeRelaySets` (`applesauce-core/helpers/relays.ts:48`, normalizes + dedupes) for the merge itself, and two *different* precedents for consuming a `T | Observable<T>` option — `applesauce-loaders`' `unwrap` (`take(1)` + `switchMap`, correct only for one-shot loader calls) and a second, unrelated-but-directly-applicable idiom already used repo-wide for **continuous** consumption: `isObservable(x) ? x : of(x)` (see `packages/core/src/observable/relay-selection.ts:57,69,83-84` and `packages/loaders/src/loaders/timeline-loader.ts:281,309,334`). This second idiom — not `unwrap` — is the correct base for concord's long-lived sites (live subscriptions, `connected$`/`authenticated$`); `unwrap`'s hard-coded `take(1)` would freeze those at the first emission (this is exactly what D-11 warns against). A small `extraRelays$` derivation belongs once at the top of each engine's constructor: `isObservable(options.extraRelays) ? options.extraRelays : of(options.extraRelays ?? [])`, piped through `startWith([])` (D-10) so nothing blocks pre-emission, then reused at every `switchMap` consumption site (D-08/D-09).

Verified by direct file read, every code reference in CONTEXT.md's `<canonical_refs>` (all `community.ts`/`client.ts`/`sync.ts`/`channel-sync.ts`/`private-channel.ts`/`invite-manager.ts`/`invite-watcher.ts`/`relay-auth.ts` line numbers) matches the current source exactly — the CONTEXT.md decisions are directly actionable without re-deriving the site enumeration. This RESEARCH.md's job is narrower: confirm the reactive-consumption idiom against actual repo precedent (found — see State of the Art below), enumerate every transport call site as one master table for the planner's task breakdown, and surface the one real implementation risk this phase carries: writing a **non-`take(1)` unwrap-equivalent** correctly for ~9 long-lived sites without copy-pasting `isObservable(...) ? ... : of(...)` nine times (Don't Hand-Roll below addresses this).

**Primary recommendation:** Add a tiny local helper — `toRelaysObservable(v: string[] | Observable<string[]> | undefined): Observable<string[]>` returning `(isObservable(v) ? v : of(v ?? [])).pipe(startWith([]), distinctUntilChanged(sameRelaySet))` — live in `packages/concord/src/helpers/relays.ts` (new file) or inlined in `keys.ts`/`community.ts`; call it once per engine constructor to produce `extraRelays$`, then thread `mergeRelaySets(this.relays(), extra)` through a private `transport()` method (D-04) at every pool call site, and `switchMap` over `extraRelays$` only at the ~4 continuous sites (D-11). No new npm dependency — both `mergeRelaySets` and the observable-or-value idiom are patterns already living in this monorepo.

### Pattern 1: Continuous (long-lived) extraRelays consumption

**What:** Derive `extraRelays$: Observable<string[]>` once per engine at construction, then `switchMap` over it at every site that holds a socket or emits over time.
**When to use:** Live subscriptions (`openLive()` in `community.ts`/`private-channel.ts`, `invite-watcher.ts`'s `openLive()`), and the `connected$`/`authenticated$` derivations.
**Example:**
```typescript
// Source: repo precedent, packages/core/src/observable/relay-selection.ts:57,69
// (isObservable ? value : of(value)) — NOT applesauce-loaders' unwrap, which take(1)s.
function toRelaysObservable(v: string[] | Observable<string[]> | undefined): Observable<string[]> {
  return (isObservable(v) ? v : of(v ?? [])).pipe(startWith([]), distinctUntilChanged(sameRelaySet));
}

// In ConcordCommunity constructor (mirrors D-08's connected$/authenticated$ example):
this.extraRelays$ = toRelaysObservable(options.extraRelays);
this.connected$ = this.extraRelays$.pipe(
  switchMap((extra) => this.relayAuth.connected$(mergeRelaySets(this.relays(), extra))),
);
```

### Pattern 2: Per-operation (one-shot) extraRelays consumption

**What:** Snapshot the current extras value once at the moment of the call, merge, and use it for a single `pool.request`/`pool.publish`.
**When to use:** Publishes, `pool.request`, `syncAuthors`/`syncEpoch` backfill, invite-bundle fetch — anything that does not hold a socket open (D-11's "per-operation snapshot" list).
**Example:**
```typescript
// applesauce-loaders' unwrap IS correct here — the call is one-shot.
// Source: packages/loaders/src/helpers/loaders.ts:18-20
unwrap(this.extraRelaysOption, (extra) => {
  const relays = mergeRelaySets(this.relays(), extra);
  return this.pool.request(relays, filters);
});
// OR equivalently, snapshot via firstValueFrom(extraRelays$) if extraRelays$ is already derived.
```

### Pattern 3: The per-class `transport()` helper (D-04)

**What:** A private method on each engine that resolves the CURRENT extras snapshot and returns the merged set, so ~20 call sites stay one-liners instead of repeating `mergeRelaySets(this.relays(), <snapshot>)` inline.
**When to use:** Every pool call site inside `ConcordCommunity`/`ConcordPrivateChannel`/`ConcordInviteManager`/`InviteWatcher`.
**Example:**
```typescript
// community.ts — mirrors the existing private relays() accessor at :600-602
private transport(): string[] {
  return mergeRelaySets(this.relays(), this.currentExtraRelaysSnapshot());
}
// Call sites (community.ts:657,660,682,741,1106,1154,1190,1228,1267,1324,1391) become:
this.ensureAuth(this.transport());
this.pool.publish(this.transport(), signed);
```
For the *continuous* sites (openLive, connected$/authenticated$), `transport()`'s snapshot form is insufficient — those need the `switchMap`-over-`extraRelays$` form from Pattern 1 instead, since they must react to a later emission, not just read a snapshot at construction time.

### Anti-Patterns to Avoid

- **Reusing `unwrap` (with its hard-coded `take(1)`) at a continuous site:** `unwrap`'s `value.pipe(take(1), switchMap(next))` freezes at the first `extraRelays` emission. Applied to `openLive()` or `connected$`/`authenticated$`, a later `extraRelays$` update would silently stop taking effect — exactly the D-11 failure mode. Use the `isObservable(x) ? x : of(x)` idiom (Pattern 1) instead for these ~4 sites.
- **Merging extras into `this.relays()` itself (widening the accessor):** D-01 explicitly forbids this — `relays()` (`community.ts:600-602`) and `defaultRelays` (`client.ts`) must keep returning ONLY the protocol set. Every merge happens at the point of use via a *separate* `transport()`/`mergeRelaySets` call, never by mutating what `relays()` returns.
- **Counting extras in the refounding majority gate:** D-06 is explicit — `refound()`'s `majorityThreshold` denominator must stay `relays().length` (the configured protocol set), never `transport().length`. Only the publish TARGET widens; the response-counting arithmetic (`community.ts:1358-1364`) is untouched.
- **Asserting the canary/regression tests against publish TARGETS:** D-05 — a cache relay legitimately appearing as a `pool.publish`/`pool.request`/`pool.subscription` target is the feature working correctly. Tests must assert on event CONTENT and returned ARTIFACTS (invite URLs, `material.relays`, community-list entries) never containing the canary string — never on which relays were called.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Merging + deduping + normalizing relay URL lists | A custom `Set`-based union loop | `mergeRelaySets` (`applesauce-core/helpers/relays.ts:48`) | Already normalizes via `normalizeRelayUrl`, silently drops malformed URLs (try/catch per entry), and is the exact helper `applesauce-loaders` already uses for the identical `extraRelays` pattern — reusing it keeps concord's relay-URL semantics identical to the rest of the monorepo |
| Consuming a `T \| Observable<T>` option reactively | A bespoke class-level subject + manual subscribe/unsubscribe wiring per engine | The repo-wide `isObservable(x) ? x : of(x)` idiom (`packages/core/src/observable/relay-selection.ts`, `packages/loaders/src/loaders/timeline-loader.ts`) wrapped in one small shared helper | This exact idiom is already proven correct for continuous RxJS consumption elsewhere in the monorepo; writing a new one risks subtly different semantics (e.g. forgetting `startWith`, or accidentally reusing `unwrap`'s `take(1)`) |
| Detecting "did the merged relay set actually change" to avoid needless socket churn | A naive `JSON.stringify` comparison or re-deriving on every emission | `distinctUntilChanged` with a content comparator (mirrors `community.ts`'s existing `sameSet`/`sameChannelViews`/`sameStanding` helpers) combined with the EXISTING author-signature dedupe guard in `openLive()` (`community.ts:645-648`) | `openLive()` already has a churn-avoidance guard (`if (sig === this.liveAuthors && this.liveSub) return`) keyed on authors only; D-09 requires this intent survive the reactive form — don't replace it, extend its key to include the merged relay set |

**Key insight:** Every primitive this phase needs (merge, dedupe, normalize, continuous-observable consumption, content-equality dedup) already exists somewhere in this monorepo and is already used for a structurally identical problem. The entire phase is composition, not new algorithm design — the risk is in getting the WIRING right (which sites get the continuous form vs the snapshot form, D-11) not in the merge logic itself.

## Common Pitfalls

### Pitfall 1: Reusing `unwrap` at a long-lived site freezes reactivity

**What goes wrong:** A live subscription or `connected$`/`authenticated$` derivation built with `unwrap(extraRelaysOption, extra => ...)` only ever sees the FIRST value `extraRelaysOption` emits — subsequent app-side updates to the `Observable<string[]>` are silently ignored, because `unwrap` internally does `.pipe(take(1), switchMap(...))`.
**Why it happens:** `unwrap` (`packages/loaders/src/helpers/loaders.ts:18-20`) was designed for ONE-SHOT loader calls, where "resolve to the current value and move on" is exactly correct. Concord's long-lived engine, however, needs "resolve to the current value AND react to every future one."
**How to avoid:** Reserve `unwrap` strictly for the D-11 "per-operation snapshot" call sites (publishes, `pool.request`, sync backfill). For the ~4 continuous sites, derive `extraRelays$` once via the `isObservable(x) ? x : of(x)` idiom and `switchMap` over it at each continuous consumption point.
**Warning signs:** A test that emits a second value on a `Subject`/`BehaviorSubject` passed as `extraRelays` and asserts the live subscription's relay set updated — if that test is missing, this pitfall can ship silently.

### Pitfall 2: Merging extras into `relays()`/`defaultRelays` instead of at the point of use

**What goes wrong:** If a future edit (or a careless refactor) folds `extraRelays` into what `this.relays()` returns, EVERY downstream consumer of `relays()` — including `buildInviteLink`, `buildInviteBundle`, `material.relays` persistence, and the kind-13302 community list — silently starts leaking the app-local relay into protocol content. This is exactly the bug class the phase exists to prevent.
**Why it happens:** `relays()` is used at both transport AND protocol sites today (the two "dual-meaning locals," `community.ts:1154` and `client.ts:481`), so it's tempting to "fix it once" by widening the shared accessor rather than merging separately at each transport call site.
**How to avoid:** D-01's structural guarantee: `relays()`/`defaultRelays` NEVER changes shape or behavior. All merging happens through a clearly-named `transport()` (or equivalent) method that is a DIFFERENT function, called only at transport sites.
**Warning signs:** Any diff that touches the body of `relays()` (`community.ts:600-602`) or `defaultRelays`'s assignment (`client.ts:258`) is a signal to re-read D-01 before proceeding.

### Pitfall 3: The refounding majority gate silently changes its passing threshold

**What goes wrong:** If `refound()`'s `majorityThreshold` is computed off `transport().length` instead of `relays().length`, adding one extra relay changes a 2-of-3 majority bar to 3-of-4 (or worse, lets a user satisfy the whole gate by always-acking against their own localhost cache relay while zero real community relays confirm the wrap) — silently breaking ROTATE-09's Phase-8-settled arithmetic.
**Why it happens:** The wrap IS published to the merged set (transport widening is correct and intentional here per D-06) — it's easy to accidentally also widen the COUNTING side while touching this function, since both the publish target and the response filter are adjacent lines (`community.ts:1358-1364`).
**How to avoid:** Publish target = `transport()` (merged). Response filter (`from ∈ relays()`) and `majorityThreshold` denominator = `relays()` (protocol-only, UNCHANGED). Two different relay-set values in the same function — name them distinctly in code (e.g. `publishTargets` vs `relays`) to make the split visible at the call site, not just in a comment.
**Warning signs:** A refound() test with extras configured that still can't be made to pass with fewer real-relay acks than before this phase — if adding extras changes what "majority" requires, the gate moved.

### Pitfall 4: `openLive()`'s author-signature dedupe guard breaks under a reactive `extraRelays$`

**What goes wrong:** `openLive()` currently guards against redundant socket teardown/reopen with `if (sig === this.liveAuthors && this.liveSub) return`, keyed ONLY on the sorted author list. Once `openLive()` is invoked from inside a `switchMap` over `extraRelays$`, EVERY extras emission — even a no-op re-emission of the same set — re-enters `openLive()`. If the guard isn't extended to also key on the merged relay set, either (a) the guard still fires correctly (fine) or (b) if the guard is accidentally dropped during the refactor, every `extraRelays$` emission tears down and reopens the live socket even when nothing actually changed.
**Why it happens:** The existing guard predates any relay-set reactivity — it was written when `relays()` never changed after construction. Wiring in `extraRelays$` introduces a second reactive dimension the guard wasn't designed for.
**How to avoid:** Extend the guard's signature key to include the merged relay set (e.g. `` `${sig}|${transport().sort().join(",")}` ``), or apply `distinctUntilChanged` on `extraRelays$` itself (comparing merged-set content, not raw extras) BEFORE the `switchMap`, so a no-op emission never reaches `openLive()` in the first place. The `distinctUntilChanged` approach is cleaner: it keeps `openLive()`'s existing internal guard untouched and pushes the churn-avoidance responsibility to the observable pipeline, matching how `community.ts`'s other derived observables (`channels$`, `status$`) already use `distinctUntilChanged` with a content comparator.
**Warning signs:** A test with `extraRelays$` re-emitting the SAME array reference or content and asserting the live subscription was NOT torn down — absence of this test is the risk signal.

## Code Examples

### Deriving a continuous `extraRelays$` (the shared helper)

```typescript
// Source: repo precedent — packages/core/src/observable/relay-selection.ts:57,69,83-84
// New file: packages/concord/src/helpers/relays.ts
import { isObservable, Observable, of, startWith, distinctUntilChanged } from "rxjs";
import { mergeRelaySets } from "applesauce-core/helpers/relays";

function sameRelaySet(a: string[], b: string[]): boolean {
  if (a.length !== b.length) return false;
  const setB = new Set(b);
  return a.every((r) => setB.has(r));
}

/** Resolve an `extraRelays?: string[] | Observable<string[]>` option to a continuous
 *  Observable — NEVER take(1)s, so a long-lived engine reacts to every future update.
 *  For one-shot call sites, prefer applesauce-loaders' `unwrap` instead. */
export function toRelaysObservable(v: string[] | Observable<string[]> | undefined): Observable<string[]> {
  return (isObservable(v) ? v : of(v ?? [])).pipe(startWith([]), distinctUntilChanged(sameRelaySet));
}

export { mergeRelaySets };
```

```typescript

// extraRelays$ emission, gated by the existing author-signature check PLUS the
```

## Open Questions

1. **Should `ConcordInviteManager` and the non-live parts of `InviteWatcher` get their own `extraRelays$` field, or just snapshot the option per call?**
   - What we know: Neither engine holds a long-lived socket EXCEPT `InviteWatcher.openLive()` (`invite-watcher.ts:360-365`) — every other call site in both classes (`refresh()`, `revokeBundle()`, `save()`, `authenticateUser()`) is per-operation per D-11.
   - What's unclear: Whether `InviteWatcher` should hold one shared `extraRelays$` (continuous) used both by its live sub AND its per-operation sites (simpler, one field), or split into a continuous field for the live sub and `unwrap`+snapshot for the rest (matches D-11's letter more precisely but adds a second code path in one class).
   - Recommendation: Hold one `extraRelays$` continuous field per engine (simplest, matches D-08's community precedent) and use `firstValueFrom(this.extraRelays$)` (or an internal snapshot getter) for the per-operation sites rather than maintaining two separate consumption mechanisms in the same class. This keeps every engine's `transport()`-equivalent helper implemented identically.

2. **Exact merge-helper naming and whether it is `transport()` verbatim per-class, per Claude's Discretion.**
   - What we know: CONTEXT.md's `transport()` is explicitly a "working proposal," not locked.
   - What's unclear: Whether all four engines (`ConcordCommunity`, `ConcordPrivateChannel`, `ConcordInviteManager`, `InviteWatcher`) should literally share this name, or each pick something locally idiomatic (`ConcordInviteManager` currently has no analogous private accessor to mirror; `InviteWatcher` calls its field `relays$`, which would collide in spirit with a `relays()`-vs-`transport()` split).
   - Recommendation: Use `transport()` uniformly across all four engines for grep-ability and consistency with the CONTEXT.md's canonical language, adjusting only where a name collision with an existing public field forces a rename (e.g. `InviteWatcher` already has a public `relays$: BehaviorSubject<string[]>` for DISCOVERED inbox relays — a private `transport()` method coexists fine, no collision).
