## Transport/protocol split

### Q1 — What mechanism enforces that extraRelays never reach a protocol write?

| Option | Description | Selected |
|--------|-------------|----------|
| Two named accessors | `protocolRelays()` / `transportRelays()`; 13 transport sites switch, `buildInviteLink` keeps protocol | (initially chosen, then superseded) |
| Pool boundary union | `relays()` never widened; extras unioned inside a private routing layer over `pool.*` | |
| Union + explicit subtract | `relays()` returns the union; protocol sites filter extras back out | |
| **Other (user)** | Keep `.relays` as-is; add `extraRelays` alongside it, following applesauce-loaders | ✓ |

**User's choice:** *"we should keep the .relays, but extra relays should be an additional name of extraRelays in a similar way to how the applesauce-loaders are handling them"*

**Notes:** Verified the loaders convention before revising — `extraRelays?: string[] | Observable<string[]>` declared alongside the pointer's own relays, merged only at point of use via `mergeRelaySets`, resolved through `unwrap` (`tag-value-loader.ts:73-81`, `address-loader.ts:154`, `event-loader.ts:111`, `helpers/loaders.ts:18-20`). This yields the same leak-proof property as the routing-layer option without an indirection layer: `relays()` is never widened, so `buildInviteLink(..., inviteRelays)` at `community.ts:1169` is safe by construction. The `protocolRelays()`/`transportRelays()` rename was dropped.

## Quorum & status signals

### Q1 — How does the refounding majority gate treat extraRelays?

| Option | Description | Selected |
|--------|-------------|----------|
| Publish to merged, count protocol only | Extras receive the wraps; `okCount` filters by `from ∈ relays()`, denominator stays `relays().length` | ✓ |
| Skip extras for gated wraps | Quorum code untouched, but the cache relay misses rotation events | |
| Publish to merged, count all OKs | Named explicitly as the trap — a localhost OK would satisfy the discoverability quorum | |

**User's choice:** Publish to merged, count protocol only
**Notes:** Walked the ROTATE-09 comment at `community.ts:1350-1358` first — publish acks are the only evidence anyone else can discover the new epoch, and "a relay that never answers counts against the denominator." Both corruption directions were traced: inflated denominator (2-of-3 → 3-of-4, potentially unsatisfiable) and a fraudulent numerator (always-OK localhost satisfying quorum alone). `PublishResponse.from` (`packages/relay/src/types.ts:78`) makes attribution straightforward.

### Q2 — Do extraRelays feed connected$ / authenticated$?

| Option | Description | Selected |
|--------|-------------|----------|
| No — protocol set only | Signals keep answering "is the COMMUNITY reachable"; roadmap's auth clause honored by Q1's ensureAuth decision | |
| Protocol only + separate extras signal | Truthful and complete, but new public API beyond the roadmap goal | |
| Yes — merged set | Literal read of the roadmap goal | ✓ |

**User's choice:** Yes — merged set
**Notes:** Chosen after being shown both failure modes explicitly: `connected$` is `.some()` (`relay-auth.ts:88`) so an always-up local relay pins it true and can mask a full community outage; `authenticated$` is `.every()` (`relay-auth.ts:109-112`) so — given extras now participate in AUTH — a gating extra that rejects our stream keys pins it false indefinitely. User ruled with these in view; consequences to be documented in JSDoc, not mitigated.

### Q3 — How do connected$/authenticated$ stay current when extraRelays emits?

| Option | Description | Selected |
|--------|-------------|----------|
| switchMap at the community | `RelayAuth` signatures stay `string[]`; only `community.ts:360-361` changes | ✓ |
| Widen relay-auth signatures | One implementation serves community and private-channel, but changes `RelayAuth`'s public shape and all callers/tests | |
| You decide | Planner picks by call-site count | |

**User's choice:** switchMap at the community
**Notes:** Raised because `connected$`/`authenticated$` are constructed once from a snapshot and close over it — without this, the merged-set decision would silently work only for the plain-array form of `extraRelays`.

## Observable reactivity

### Q1 — What happens to live subscriptions when extraRelays emits a new set?

| Option | Description | Selected |
|--------|-------------|----------|
| Resubscribe via widened signature | Fold merged set into `openLive()`'s dedupe signature; smallest change to the existing imperative lifecycle | |
| Only future operations, no resubscribe | Literal loaders `take(1)` semantics; no reconnect churn, but a runtime-added cache relay feeds nothing live | |
| switchMap the subscription | Reactive pipeline, consistent with the connected$ decision; restructures `openLive()` | ✓ |

**User's choice:** switchMap the subscription
**Notes:** Flagged that `openLive()`'s guard (`community.ts:648`) keys on the author signature only, so its churn-avoidance intent must be reproduced (e.g. `distinctUntilChanged` over authors + merged set) rather than dropped.

### Q2 — How is the extraRelays Observable consumed?

| Option | Description | Selected |
|--------|-------------|----------|
| Client-owned BehaviorSubject | One upstream subscription at construction, never blocks, single dispose handle | |
| unwrap per operation | Exact loaders parity, but a not-yet-emitted Observable blocks Concord traffic | |
| unwrap + startWith([]) | Per-operation unwrap that never blocks; producer re-subscribed per operation/sub-engine | ✓ |

**User's choice:** unwrap + startWith([])
**Notes:** Consequence recorded — apps should pass a hot/shared source (`BehaviorSubject` or `shareReplay(1)`), since a cold producer is re-subscribed per operation and per sub-engine.

### Q3 — Which sites get the continuous (no take(1)) form?

| Option | Description | Selected |
|--------|-------------|----------|
| The three live sockets + status | Community/private-channel/invite-watcher live subs plus `connected$`/`authenticated$`; everything else snapshots | ✓ |
| Live sockets only, status snapshots | Simpler, but partially undoes the merged-status decision | |
| You decide | Planner enumerates while implementing | |

**User's choice:** The three live sockets + status
**Notes:** Asked because applying `take(1)` at a long-lived site would freeze it at the first value and silently undo the two switchMap decisions. Resulting rule: `startWith([])` everywhere, `take(1)` only for one-shot operations — anything holding a socket or emitting over time is continuous.

- Whether `ConcordPrivateChannel`'s `connected$`/`authenticated$` (`private-channel.ts:80-82`) receive the same merged treatment as the community's.

## Deferred Ideas

- Per-community / per-channel `extraRelays` override.
- A separate extras-only status observable (the natural mitigation if the merged-status caveats prove annoying).
- Extras satisfying the `STOCK_RELAYS` fallback for single-option local-only setups.
- Concord documentation under `docs/` — the package has no docs page at all yet.
