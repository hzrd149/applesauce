# Phase 12.3: Code Review Report (re-review after gap-closure plans 08–10)

**Reviewed:** 2026-07-22
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

This is a fresh adversarial review of the current state of the code, superseding the pre-08/09/10 REVIEW.md.

The three gap-closure plans do what they claim on their own terms. `ExtraRelays` is now fail-soft on an errored source and keeps its internal subject live (verified against `helpers/__tests__/relays.test.ts`); `sameRelaySet` no longer masks a genuine membership change behind a duplicate; the empty-extras identity fast path is genuinely byte-identical (pinned with `toBe`, not `toEqual`). `refound()` derives `protocolRelays` once via `mergeRelaySets` and computes both the threshold and the attribution set from that single local, and `normalizeAckOrigin` correctly swallows absent/malformed `from` values so a hostile ack can never surface as a `TypeError` in place of the majority-abort. `PublishResponse.from` is set by the pool from its own `relay.url` (`packages/relay/src/relay.ts:962`), not by relay-supplied content, so ack attribution is not spoofable. The per-URL auth-driver registry prunes correctly, `Map` deletion-during-`for..of` is safe, and all four `ensureAuth` call sites in `sync.ts`/`channel-sync.ts` do pass the complete transport set. The `stop()`-is-pause / `dispose()`-releases split is implemented consistently across all four engines.

However, the review found a **security gap that the T-12.3-09-04 fix did not close and in fact widened**, plus a **crash-to-poisoned-state path** reachable through the same attacker-controlled invite surface. Both are BLOCKERs. The remaining eight WARNINGs are correctness/robustness defects introduced or left standing by this phase.

WR-09 from the prior review is explicitly deferred and is not re-raised.

## Critical Issues

### CR-01: Invite-link fragment relays bypass the `isSafeRelayURL` filter and still land in `material.relays` — the gap-closure fix made this path *more* reachable

**File:** `packages/concord/src/client/client.ts:566`, `packages/concord/src/client/client.ts:637`, `packages/concord/src/helpers/invite-bundle.ts:78-110`

**Issue:** T-12.3-09-04 added `isSafeRelayURL` filtering to `validateInviteBundle` with this stated rationale:

> "This array flows into `JoinMaterial.relays` and from there into the refounding quorum's protocol set … so an unvalidated entry (non-string, junk string, or a non-websocket scheme like `http://`) is attacker-reachable input and must never survive."

That rationale applies verbatim to the **second** path into `JoinMaterial.relays`, which was left unfiltered. `decodeFragment` (`invite-bundle.ts:78-110`) decodes attacker-supplied bytes from the link fragment into relay strings with no validation at all — the `lead === 0xff` branch (`:101-104`) pushes an **arbitrary decoded string** as a relay URL, and the only filter applied is `.filter(Boolean)` (`:109`). Those strings become `ParsedInvite.bootstrapRelays`, and:

```ts
// client.ts:566
const relays = parsed.bootstrapRelays.length ? parsed.bootstrapRelays : this.defaultRelays;
...
// client.ts:637 (inside joinFromBundle)
relays: bundle.relays.length ? bundle.relays : fallbackRelays,
```

So whenever `bundle.relays` is empty, the unvalidated fragment relays become the joined community's `material.relays` — persisted to the local mirror, published inside the user's kind-13302 Community List, embedded in every invite bundle and invite link that user later mints, and used as the protocol/quorum set in `refound()`.

The gap-closure fix **increased** the reachability of this path: `validateInviteBundle` now *empties* `bundle.relays` when every entry is junk, and `invite-bundle.test.ts:120-130` explicitly pins that "a junk relay list is not grounds to reject the bundle". That is precisely the condition that forces the fallback onto the unfiltered fragment relays. An attacker can therefore reliably steer a joiner onto arbitrary attacker-chosen relay strings by shipping a bundle whose `relays` are all junk while the fragment carries the real payload.

Separately, `client.ts:572` dials `this.transport(relays)` from that same unvalidated array before any validation runs at all.

**Fix:** Filter at the decode boundary so both consumers are covered:

```ts
// helpers/invite-bundle.ts — decodeFragment
import { isSafeRelayURL } from "applesauce-core/helpers/relays";
...
return { token, relays: relays.filter((r) => !!r && isSafeRelayURL(r)) };
```

and, defensively, re-filter at the point the fallback becomes protocol state:

```ts
// client.ts — joinFromBundle
relays: bundle.relays.length ? bundle.relays : fallbackRelays.filter(isSafeRelayURL),
```

Add a regression test: a link whose bundle relays are entirely junk and whose fragment carries `http://evil.test` plus a non-URL blob must yield a `community.material.relays` containing neither.

### CR-02: A malformed-but-owner-proof-valid invite bundle throws out of `recordJoin` *after* the membership is already in `this.list`, leaving a permanently poisoned client

**File:** `packages/concord/src/helpers/invite-bundle.ts:216-250`, `packages/concord/src/client/client.ts:688-696`, `packages/concord/src/client/client.ts:886-900`

**Issue:** `validateInviteBundle` is documented as the function that "bounds and self-certifies an attacker-crafted bundle". It validates `owner`, `owner_salt`, the `community_id` proof, the `channels`/`relays` array shapes, the channel count cap, the relay cap+filter, and `refounder`'s type. It does **not** validate `community_root`, `root_epoch`, or the shape of individual `channels[]` entries.

The owner proof is no obstacle to an attacker — they generate `owner` and `owner_salt` themselves, so `community_id === sha256(owner || salt)` is trivially satisfiable for a bundle that is otherwise entirely hostile. A bundle with `community_root: "zz"` (or absent, or odd-length) therefore validates, and then:

- `client.joinFromBundle` → `recordJoin(material)` → `this.list = joinCommunity(...)(...)` **mutates the document first** (`client.ts:689-694`)
- → `addCommunity(material)` → `new ConcordCommunity({ material })` → `deriveConcordKeys` → `baseKeysFor` → `hexToBytes(material.community_root)` (`helpers/keys.ts:132`) — **throws** (verified: `hex string expected, got non-hex character "zz"`).

The throw propagates out of `joinByLink`, but `this.list` retains the phantom membership. Consequences:

1. Any subsequent `saveMirror()` / `saveCommunityList()` persists and **publishes** the attacker's entry into the user's kind-13302 list.
2. `reconcileCommunities()` (`client.ts:886-900`) runs on **every** Community List cast emission and unconditionally calls `this.addCommunity(community.current)` for every live entry — so it throws on every reconcile from then on, as an unhandled promise rejection from `void this.reconcileCommunities()` (`client.ts:869`), and the loop aborts before starting any community that iterates after the poisoned one.

`root_epoch` has the same problem via `material.root_epoch + 1` (`keys.ts:133`) → `NaN` epochs propagated into `epoch$`, plane store keys, and the guestbook plane key. Unvalidated `channels[]` entries reach `deriveConcordKeys` the same way.

**Fix:** Two independent changes; both are needed.

(a) Complete the validator:

```ts
// helpers/invite-bundle.ts — validateInviteBundle
if (!isHex(bundle.community_root, 64)) return undefined;
if (!Number.isSafeInteger(bundle.root_epoch) || bundle.root_epoch < 0) return undefined;
const channels = bundle.channels.filter(
  (c): c is ChannelKey =>
    !!c && typeof c.id === "string" && isHex(c.key) && Number.isSafeInteger(c.epoch),
);
if (channels.length > INVITE_BUNDLE_MAX_CHANNELS) return undefined;
```

(b) Make `recordJoin` atomic — construct the engine *before* mutating the document, so a rejected join leaves no residue:

```ts
private recordJoin(material: JoinMaterial): ConcordCommunity {
  const community = this.addCommunity(material); // may throw — nothing mutated yet
  this.list = joinCommunity({ ... })(this.list, this.tombstones).communities;
  return community;
}
```

and wrap the `addCommunity` call in `reconcileCommunities()` in a try/catch that logs and skips the bad entry rather than aborting the loop.

## Warnings

### WR-01: `isSafeRelayURL` admits `ws://`, so an attacker bundle can downgrade a joiner's transport to plaintext

**File:** `packages/concord/src/helpers/invite-bundle.ts:246`

**Issue:** The comment on the new filter says junk "or a non-websocket scheme like `http://`" is dropped, implying scheme safety. `isSafeRelayURL`'s regex is `^wss?:\/\/…` (`packages/core/src/helpers/relays.ts:38-44`) — it accepts `ws://` as well. A bundle carrying `ws://attacker.test` survives validation, becomes `material.relays`, and is then dialled unencrypted for community sync, publishes, and NIP-42 AUTH. Concord's payloads are E2E-encrypted so this is not a content disclosure, but it exposes stream pubkeys, subscription filters, connection metadata and the NIP-42 challenge/response flow to any network observer — and the code's own comment asserts a guarantee it does not provide.

**Fix:** Tighten the concord-side check rather than relying on the generic core helper:

```ts
.filter((entry): entry is string =>
  typeof entry === "string" && entry.startsWith("wss://") && isSafeRelayURL(entry))
```

and correct the comment to state what is actually enforced.

### WR-02: No `disposed` guard on `ConcordClient` / `ConcordInviteManager` / `InviteWatcher` — `start()` after `dispose()` silently produces a permanently frozen engine

**File:** `packages/concord/src/client/client.ts:470-475`, `packages/concord/src/client/invite-manager.ts:176-179`, `packages/concord/src/client/invite-watcher.ts:213-233`, `packages/concord/src/client/invite-watcher.ts:277-281`

**Issue:** All three classes document "NOT restartable after `dispose()`" but none enforces it — unlike `ConcordCommunity`, which has a real `disposed` flag checked in `start()` (`community.ts:523`). `dispose()` calls `stop()`, which sets `started = false`, so a subsequent `start()` passes its own idempotence guard and runs to completion against a completed `ExtraRelays` subject.

The `InviteWatcher` case is the sharpest: `start()` explicitly re-establishes extras reactivity via `if (this.extrasSub.closed) this.subscribeExtras()` (`:222`). After `dispose()`, `this.extras.relays$` is a *completed* `BehaviorSubject`, so the resubscription emits the last value and immediately completes — `extrasSub.closed` is `true` again and the watcher is frozen on a stale extras snapshot forever, with no error and no log. `ExtraRelays.current` still returns a value (a completed `BehaviorSubject` retains `.value`), so `transport()` keeps silently merging a dead snapshot rather than failing loudly.

**Fix:** Add a `private disposed = false;` to each class, set it in `dispose()`, and guard:

```ts
async start(): Promise<void> {
  if (this.disposed) throw new Error("InviteWatcher disposed — construct a new instance");
  if (this.started) return;
  ...
}
```

### WR-03: `ExtraRelays.merge`'s "dropped unparseable" diagnostic miscounts normalization collapses as parse failures

**File:** `packages/concord/src/helpers/relays.ts:144-153`

**Issue:** `rawUnionSize` is computed from **raw** strings (`new Set([...base, ...this.current])`) while `merged` comes from `mergeRelaySets`, which **normalizes** before deduplicating. Two entries that differ only by normalization (`wss://a.test` vs `wss://a.test/`, or differing case) collapse in `merged` but not in `rawUnionSize`, so a completely healthy configuration logs `merge dropped 1 unparseable relay URL(s)`. This is a permanently-misleading diagnostic on the exact configuration shape (`relays` and `extraRelays` overlapping in different written forms) that operators are most likely to hit — and `relays.test.ts:139-144` exercises that very case without asserting on the log.

**Fix:** Compare against a normalized baseline, or drop the count and log the identity of the dropped entries:

```ts
const dropped = [...base, ...this.current].filter((url) => {
  try { return !merged.includes(normalizeRelayUrl(url)); } catch { return true; }
});
if (dropped.length > 0) log("merge dropped %d unparseable relay URL(s): %o", dropped.length, dropped);
```

### WR-04: `ConcordClient.start()` leaves an uncancelled 8-second timer

**File:** `packages/concord/src/client/client.ts:419`

**Issue:**

```ts
await Promise.race([this.listHydration, new Promise((r) => setTimeout(r, 8000))]);
```

The timer is never cleared. When hydration wins the race (the common case) an 8s timer stays pending, keeping the Node event loop alive — so every test that constructs a client with a stored 13302 holds the process open for up to 8s past teardown — and it fires after any `stop()`/`dispose()` that happens in the interim. The `8000` is also a magic number duplicated from the `timeout(8000)` in `fetchList` (`:838`) with no shared constant, so the two can silently diverge.

**Fix:**

```ts
const HYDRATION_TIMEOUT_MS = 8000;
let timer: ReturnType<typeof setTimeout> | undefined;
try {
  await Promise.race([
    this.listHydration,
    new Promise((r) => { timer = setTimeout(r, HYDRATION_TIMEOUT_MS); }),
  ]);
} finally {
  if (timer) clearTimeout(timer);
}
```

### WR-05: `refound()`'s target trace is unreachable on the abort path, and the deduplicated denominator silently weakens the quorum on a malformed relay list

**File:** `packages/concord/src/client/community.ts:1455`, `packages/concord/src/client/community.ts:1505`, `packages/concord/src/client/community.ts:1530-1537`

**Issue:** Two related problems in the reworked quorum block.

1. The `publishLog("refounding publish targets=%d protocol=%d", …)` call sits **after** both `requireMajority` loops (`:1533-1537`). Every failure path throws before reaching it, so the one trace that would tell an operator *why* the quorum was missed — how many protocol relays the threshold was computed against, versus how many transport targets were actually dialled — is emitted only when it is least needed.

2. `protocolRelays = mergeRelaySets(this.relays())` shrinks the denominator both by normalization-dedup **and** by dropping unparseable entries. Dedup is defensible; silently dropping is a real behavioural weakening versus pre-phase. A community configured with 5 relays of which 3 are junk goes from `⌈6/2⌉ = 3` required acks to `⌈3/2⌉ = 2`. `community.test.ts:1992-2026` pins the weakened behaviour as correct, with no counterpart test asserting that a *majority-junk* list still fails safe. Given CR-01/CR-02, `material.relays` is attacker-influenceable, so the denominator is attacker-influenceable.

**Fix:** Move the trace above the `requireMajority` loops. Then either (a) throw when `mergeRelaySets` drops entries during a refounding — a security-critical operation should not silently operate on a partially-parsed protocol set — or (b) at minimum log a warning and add a test pinning that a list whose valid entries are a minority of the raw list still requires a majority of the raw count.

### WR-06: `ExtraRelays.merge`'s identity fast path hands out live internal arrays by reference

**File:** `packages/concord/src/helpers/relays.ts:141-142`, `packages/concord/src/client/community.ts:675-686`, `packages/concord/src/client/client.ts:369-371`, `packages/concord/src/helpers/invite-bundle.ts:45`

**Issue:** `merge()` returns `base` itself (documented, and pinned with `toBe` at `relays.test.ts:146-159`). But no caller supplies a defensive copy:

- `ConcordCommunity.transport()` → `this.relays()` → `this.material.relays` — the community's **protocol state array**.
- `ConcordClient.transport()` → `this.defaultRelays` → `STOCK_RELAYS`, a **module-level exported mutable array**.
- `ConcordPrivateChannel.transport()` → `this.opts.relays`.

So with no extras configured — the default, and precisely the configuration whose byte-identity D-14 exists to protect — the relay-target array handed to `pool.publish` / `pool.subscription` / `pool.request` and stored in `SyncContext.relays` *is* the protocol relay array. Any in-place mutation by a consumer (a `targets.push(...)`, or an in-place `sort()`) silently corrupts community material or the package-global `STOCK_RELAYS`. Both `openLive()` implementations already defensively copy before sorting (`[...targets].sort()`), which shows the authors know this array must not be mutated — the invariant is simply unenforced everywhere else.

**Fix:** Freeze the shared constant (`export const STOCK_RELAYS = Object.freeze([...])`) and declare `merge()`'s return type as `readonly string[]` so a mutating call site fails at compile time rather than at runtime.

### WR-07: `authDrivers` is keyed on un-normalized transport URLs, so toggling extras churns every driver and a duplicate-by-normalization relay gets two drivers

**File:** `packages/concord/src/client/community.ts:719-730`, `packages/concord/src/client/private-channel.ts:340-351`

**Issue:** `ensureAuth` keys the registry by the raw string in the transport set. But the transport set changes *form* depending on whether extras are configured: with no extras it is the base array verbatim (`wss://a.test`), and with extras it is `mergeRelaySets`-normalized (`wss://a.test/`). So the first non-empty extras emission causes `ensureAuth` to prune **every** existing driver and register a brand-new one for the same relays — exactly the churn the WR-04 gap-closure work and the D-09 guard were written to prevent. `openLive()`'s signature guard compares sorted joined transport strings and is subject to the same form flip, so the live socket is torn down and reopened too.

Independently, a protocol relay list containing both `wss://x.test` and `wss://x.test/` (which `mergeRelaySets` deduplicates, but which the identity fast path passes through verbatim) registers **two** `authenticateStreamKeys` drivers against the single `pool.relay(url)` instance.

**Fix:** Normalize the registry key and the churn-guard key independently of whether the merge fast path ran:

```ts
private ensureAuth(relays: string[]): void {
  const wanted = new Set<string>();
  for (const url of relays) {
    try { wanted.add(normalizeRelayUrl(url)); } catch { /* unparseable — cannot dial */ }
  }
  ...
}
```

Add a test asserting that turning extras on from empty does not tear down the drivers for the pre-existing protocol relays.

### WR-08: `decodeFragment` has no bounds checking — truncated fragments produce `NaN` indices and up to 255 garbage iterations

**File:** `packages/concord/src/helpers/invite-bundle.ts:78-110`

**Issue:** Every read is an unchecked `bytes[i++]`. On a truncated fragment: `lead` becomes `undefined`, falls into the final `else` branch, `len = bytes[i++]` is `undefined`, `bytes.slice(i, i + undefined)` evaluates to `slice(i, NaN)` → empty, and `i += undefined` sets `i = NaN` for the remainder of the loop, which then spins up to `count` (max 255) times pushing empty strings. `token = bytes.slice(i, i + 16)` with `i === NaN` silently returns an **empty token**, producing a confusing downstream NIP-44 decrypt failure instead of a clear "malformed invite link" error. This is attacker-controlled input (the same surface as CR-01), reached before any validation.

**Fix:** Validate up front and bound each read:

```ts
const need = (n: number) => { if (i + n > bytes.length) throw new Error("malformed invite fragment"); };
...
need(1); const count = bytes[i++];
...
need(len); const host = new TextDecoder().decode(bytes.slice(i, i + len));
...
need(16); const token = bytes.slice(i, i + 16);
```

### WR-10: `InviteWatcher.dispose()` unsubscribes `extrasSub` twice

**File:** `packages/concord/src/client/invite-watcher.ts:263-281`

**Issue:** `dispose()` calls `this.stop()` (which already does `this.extrasSub.unsubscribe()` at `:270`) and then unsubscribes the same subscription again at `:279`. Harmless today, but it signals uncertainty about which method owns the teardown, and it diverges from `ConcordClient.dispose()`, `ConcordInviteManager.dispose()` and `ConcordPrivateChannel.dispose()`, none of which double-unsubscribe. If `stop()` is later changed to *not* close `extrasSub` (the natural way to make the pause semantics cheaper), the redundant line here becomes the only thing keeping `dispose()` correct — and if this line is deleted instead, `dispose()` silently stops releasing.

_(Numbered WR-10 to avoid collision with the deferred WR-09 from the prior review.)_

**Fix:** Delete the duplicate line and comment that `stop()` owns the `extrasSub` teardown, or move `extrasSub.unsubscribe()` out of `stop()` and keep it only in `dispose()` — pick one owner and document it.

### IN-02: `ConcordClient.stop()` leaves per-session dedup state and one subscription handle stale

**File:** `packages/concord/src/client/client.ts:445-462`

**Issue:** `stop()` is documented as a restartable pause, but it does not clear `handledInvites` (`:290`) or `autoUnlocked` (`:293`), and it unsubscribes `directInviteSub` (`:446`) without setting it to `undefined` (unlike `autoSaveSub`, which is nulled two lines later). After a `stop()`/`start()` cycle the freshly-constructed `InviteWatcher` re-emits its full invite list and every previously-handled invite is skipped by the stale `handledInvites` guard — correct only by accident, since the new community engines happen to be rebuilt from persisted material.

**Fix:** Clear both sets in `stop()` and null `directInviteSub` for consistency with the other subscription fields.

### IN-03: `ensureAuth`'s "always call with the COMPLETE transport set" invariant is unenforced across a module boundary

**File:** `packages/concord/src/client/community.ts:704-718`, `packages/concord/src/client/sync.ts:137`, `packages/concord/src/client/sync.ts:191`, `packages/concord/src/client/channel-sync.ts:49`, `packages/concord/src/client/channel-sync.ts:96`

**Issue:** Now that WR-04's fix made `ensureAuth` prune, passing it a subset silently tears down auth drivers for relays that are still wanted. All four `sync.ts`/`channel-sync.ts` call sites currently pass `ctx.relays` (verified), but the invariant lives only in a doc comment in a different file from the callers — a future `ctx.ensureAuth(someSubset)` would be a silent NIP-42 regression with no test coverage.

**Fix:** Remove the parameter entirely and have `ensureAuth()` read `this.transport()` itself; the `SyncContext` callback becomes `ensureAuth: () => this.ensureAuth()` and the invariant becomes structurally unbreakable.
