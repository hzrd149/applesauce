## Summary

This is an adversarial re-review of the current tree after plan 12.3-11 closed the two prior BLOCKERs. Full suite verified locally: `pnpm --filter applesauce-concord test` → **50 files / 377 tests passing**.

### Prior BLOCKER disposition

**CR-01 (fragment relays bypass the relay filter) — CLOSED.** `isSafeInviteRelayURL` (`helpers/invite-bundle.ts:107-111`) is now applied at `decodeFragment`'s terminal filter (`:152`), at `validateInviteBundle`'s relay filter (`:366`), and at `joinFromBundle`'s fallback branch (`client.ts:644`). I traced every path a stranger's bytes can reach `JoinMaterial.relays`: the `lead === 0xff` arbitrary-string branch, the `lead === 0x00` host-reassembly branch, and the dictionary-miss `""` case are all gated by the one predicate. The e2e regression test (`client.test.ts:1122`) hand-builds the exact hostile fragment and asserts neither hostile value survives. The anchored `LOOPBACK_PLAINTEXT_WS` regex is tight — I probed `ws://127.0.0.1.evil.com`, `ws://localhost@evil.com`, and trailing-newline evasion (JS `$` is end-of-input only, unlike Python), all rejected. The prior **WR-01** (`ws://` downgrade) is also CLOSED by the same predicate, with the loopback carve-out documented and independently reviewable.

**CR-02 (malformed bundle poisons `this.list`) — CLOSED, with a residual leak (see WR-01 below).** `validateInviteBundle` now bounds `community_root`, `root_epoch`, each `channels[]` entry and its `held` sub-entries, and `held_roots` (`:328-352`). I verified the claim "no field that reaches a `hexToBytes` call or an arithmetic epoch expression can be malformed" holds for the *scalar* fields: `community_id` is implicitly hex-bounded by the owner proof's `try/catch`, `channel.id`/`channel.key`/`held[].key` all feed `deriveChannelKeys`'s `hexToBytes` and are all `isHexKey`-checked, and `createChannel`/`addChannelKey` genuinely mint 64-hex ids and keys so the new filter cannot false-negative real bundles. `recordJoin` (`client.ts:708-717`) constructs the engine before mutating the document, and `reconcileCommunities` (`:926-935`) skips-and-logs. The cap-ordering comment ("cap on the RAW array, before filtering") is correct and load-bearing.

### What this review found

The scalar half of CR-02 is closed; the **cardinality half is not**. `validateInviteBundle`'s own doc comment states the channel cap exists "to refuse an unbounded-allocation link", but the two attacker-controlled arrays that actually dominate join cost — `held_roots` and each channel's `held` — are shape-validated and left **uncapped**, and `name` is left entirely unvalidated on the stated (and incorrect) grounds that it is "display-only". Both are BLOCKERs: one hangs a joining client, the other permanently wedges the user's published Community List. A third defect — an `ExtraRelays` subscription leaked out of every failed `ConcordCommunity` construction — is *amplified* by 12.3-11's own skip-and-continue reconcile change.

The transport/protocol split this phase exists to enforce holds. I re-traced every protocol write: `material.relays` (never merged), `buildInviteLink(..., protocolRelays)` (`community.ts:1289`), `buildInviteBundle(this.material, ...)`, `saveCommunityList`'s serialized document, and `refound()`'s `protocolRelays = mergeRelaySets(this.relays())` denominator and ack-attribution set. No extras reach any of them.

Per `12.3-11-SUMMARY.md`'s scope restatement, WR-02…WR-08, WR-10 and IN-01…IN-03 from the prior review were explicitly declared out of scope. ## Critical Issues

### CR-01: `validateInviteBundle` caps `channels` but leaves `held_roots` and `channels[].held` unbounded — a single invite link hangs the joining client

**File:** `packages/concord/src/helpers/invite-bundle.ts:342-352`, `packages/concord/src/client/client.ts:651`, `packages/concord/src/client/sync.ts:324-345`, `packages/concord/src/helpers/keys.ts:667-672`

**Issue:** The validator's own doc comment states the channel count "is capped (on the RAW array) to refuse an unbounded-allocation link". That bound is defeated by the two arrays it does not cap:

```ts
// invite-bundle.ts:348-352 — shape-validated per entry, never counted
if (
  bundle.held_roots !== undefined &&
  (!Array.isArray(bundle.held_roots) || !bundle.held_roots.every(isValidHeldKeyEntry))
)
  return undefined;
```

`held_roots` flows straight into protocol state — `joinFromBundle` writes `held_roots: bundle.held_roots ?? []` (`client.ts:651`) despite the adjacent comment asserting "invite bundles omit held_roots" (true of *our* builder, not of an attacker's bundle). From there:

1. `syncEpochs` → `buildChain(material)` (`sync.ts:327`) builds one `JoinMaterial` per held root, each via an `O(N)` `.filter(...)` over the chain — **`O(N²)`** allocation for `N` attacker-chosen entries.
2. `syncEpochs` then **sequentially `await`s a full epoch sync per chain element** (`sync.ts:331-333`), each doing NIP-42 auth plus a full per-plane relay sync. Nothing bounds the loop.

The bundle plaintext is NIP-44-capped at 65535 bytes and each `{epoch,key}` entry costs ~80 bytes of JSON, so an attacker fits **~800 entries** in one link — 800 sequential networked epoch walks before `ConcordCommunity.start()` can ever reach `phase: "live"`. The community is added to the document and mirrored *before* the walk begins, so the hang persists across restarts.

The identical omission applies to `channels[].held`: `isValidChannelEntry` validates every entry but counts none, and `deriveChannelKeys` (`keys.ts:667-672`) performs one `channelGroupKey` (an X25519/ECDH derivation) per held entry — 256 channels × unbounded `held` is a CPU storm on the private-channel spawn path.

**Fix:** Cap both arrays at the same boundary that already caps `channels`, before any per-entry work:

```ts
// helpers/invite-bundle.ts
export const INVITE_BUNDLE_MAX_HELD_ROOTS = 64;
export const INVITE_BUNDLE_MAX_HELD_CHANNEL_KEYS = 64;

// in isValidChannelEntry, after the Array.isArray(entry.held) check:
if (entry.held !== undefined && (entry.held as unknown[]).length > INVITE_BUNDLE_MAX_HELD_CHANNEL_KEYS) return false;

// in validateInviteBundle, alongside the existing held_roots guard:
if (bundle.held_roots !== undefined && bundle.held_roots.length > INVITE_BUNDLE_MAX_HELD_ROOTS) return undefined;
```

Add a regression test asserting a bundle with `held_roots.length > cap` is rejected, and a `channels[0].held.length > cap` entry is dropped. Consider also bounding `syncEpochs`' chain length defensively, since `buildChain` input can also arrive from a peer device's Community List.

### CR-02: `bundle.name` is unvalidated and is written into published protocol state, permanently wedging the user's Community List

**File:** `packages/concord/src/helpers/invite-bundle.ts:270-280`, `packages/concord/src/client/client.ts:645`, `packages/concord/src/client/client.ts:986-990`, `packages/concord/src/helpers/community-list.ts:86`

**Issue:** `isValidChannelEntry`'s doc comment establishes the project's stated rationale for leaving strings unvalidated:

> "`name` is intentionally left unvalidated: it is display-only, never enters a key derivation…"

That rationale is applied to the *bundle's* top-level `name` too (it has no check anywhere in `validateInviteBundle`), and it is **factually wrong for that field**. `joinFromBundle` writes it into protocol state:

```ts
// client.ts:645
name: bundle.name,
```

`JoinMaterial.name` is then serialized twice per entry (`seed` **and** `current`) into the local mirror and into the published kind-13302 document. `saveCommunityList` gates on a hard byte cap:

```ts
// client.ts:986-990
if (!communityListWithinByteCap(list, tombstones)) {
  this.publishLog("community list exceeds the NIP-44 byte cap; not publishing");
  console.warn(...);
  return;                       // ← never publishes again; dirty flag not even cleared
}
```

`LIST_MAX_BYTES` is 65535 (`community-list.ts:86`). A bundle whose `name` is a ~40KB string — comfortably inside the NIP-44 plaintext limit and any relay's event-size limit — is duplicated into `seed` + `current` and pushes the document past the cap on the very first save after the join.

The state is **unrecoverable through the public API**: `leave(cid)` only appends a tombstone (`client.ts:682`) and never prunes `this.list`, so the oversized entry stays in the serialized document forever. From that point the user's Community List can never be published again — joins, leaves and refounding removals made on this device silently stop propagating to their other devices, and a leave performed elsewhere can no longer be echoed. One clicked invite link permanently breaks the account's membership sync.

`bundle.name` is also the only attacker-controlled string that is neither type-checked nor length-checked: a non-string (`{}`, an array) reaches `JSON.stringify` and the persisted mirror unchanged.

**Fix:** Validate and bound it at the boundary, and prune on leave so the state is recoverable:

```ts
// helpers/invite-bundle.ts
export const INVITE_BUNDLE_MAX_NAME_BYTES = 256;
...
const name =
  typeof bundle.name === "string" && bundle.name.length <= INVITE_BUNDLE_MAX_NAME_BYTES ? bundle.name : undefined;
return { ...bundle, channels, relays, refounder, name };
```

```ts
// client.ts — leave(): drop the entry as well as tombstoning it, so a poisoned
// entry can be evicted and the document can publish again.
this.list = this.list.filter((e) => e.community_id !== cid);
```

Apply the same length bound to `label` (also attacker-controlled, also echoed into the joiner's Guestbook Join via `bundle.label` at `client.ts:665`).

## Warnings

### WR-01: every failed `ConcordCommunity` construction leaks an `ExtraRelays` subscription to the app-supplied source — and 12.3-11's skip-and-continue reconcile makes the leak unbounded

**File:** `packages/concord/src/client/community.ts:369-374`, `packages/concord/src/client/client.ts:926-935`, `packages/concord/src/helpers/relays.ts:100-107`

**Issue:** The `ConcordCommunity` constructor subscribes to the app's `extraRelays` source *before* the statement that can throw:

```ts
// community.ts:372-374
this.extras = new ExtraRelays(options.extraRelays);   // ← subscribes to the app's Observable
this.keys = deriveConcordKeys(options.material, []);  // ← throws (hexToBytes) on malformed material
```

`ExtraRelays`'s constructor calls `.subscribe(...)` (`relays.ts:100`). When line 374 throws, the half-built instance is discarded and **nothing ever calls `extras.dispose()`** — the subscription stays attached to the app's source forever.

Before 12.3-11 this leaked once per reconcile because the throw aborted the loop. The new fault-tolerant loop `continue`s past the bad entry, and `reconcileCommunities` runs on **every** Community List cast emission (`client.ts:890`, and the cast re-emits on outbox/replaceable churn). So an unconstructable entry — precisely the scenario the new try/catch exists to survive — now accumulates one permanent subscriber per emission on the app's `BehaviorSubject`, which is the sharing shape the option docs explicitly recommend (`client.ts:153-156`).

Note this is invisible with a static `string[]` (`of(...)` completes immediately) and only bites the reactive-Observable configuration this phase was built for.

**Fix:** Make construction self-cleaning, and stop retrying a known-bad entry:

```ts
// community.ts — constructor
this.extras = new ExtraRelays(options.extraRelays);
try {
  this.keys = deriveConcordKeys(options.material, []);
} catch (err) {
  this.extras.dispose();
  throw err;
}
```

and have `reconcileCommunities` record the failed `cid` in a `Set` so it logs and skips once rather than reconstructing on every emission.

### WR-02: the CR-01 defence-in-depth filter guards the *trusted* branch and leaves `joinByBundle`'s untrusted branch unfiltered

**File:** `packages/concord/src/client/client.ts:638-644`, `packages/concord/src/client/client.ts:617-619`

**Issue:** The filter added by 12.3-11 sits on the fallback branch only:

```ts
relays: bundle.relays.length ? bundle.relays : fallbackRelays.filter(isSafeInviteRelayURL),
```

with the comment claiming `fallbackRelays` is "load-bearing for the hostile one [joinByLink]". Trace both callers:

- `joinByLink` (`:567`) passes `parsed.bootstrapRelays` — **already filtered** by `decodeFragment` — or `this.defaultRelays`. So on this path the second gate is pure redundancy.
- `joinByBundle` (`:617-619`) passes `this.defaultRelays` — the app's own trusted config — and hands through `bundle` **with no validation at all**. `joinByBundle` is a public API documented as taking an "already-validated §1 bundle", but nothing enforces that: when `bundle.relays` is non-empty it goes into `material.relays` unfiltered, and every other field (`community_root`, `channels`, `held_roots`, `name`) bypasses `validateInviteBundle` entirely.

So the added filter fires only on values that were already safe, while the one branch that can carry unvalidated attacker input (`bundle.relays` via `joinByBundle`) is ungated. In-repo consumers happen to be safe (`ConcordDirectInvite.bundle` routes through `getDirectInviteBundle` → `validateInviteBundle`), which is why no test catches this — but the guarantee rests on an undocumented invariant across a public API boundary.

**Fix:** Make `joinByBundle` a validation boundary in its own right — the validator is idempotent, so revalidating an already-validated bundle is free:

```ts
async joinByBundle(bundle: InviteBundle): Promise<ConcordCommunity> {
  const validated = validateInviteBundle(bundle);
  if (!validated) throw new Error("invite failed validation");
  return this.joinFromBundle(validated, this.defaultRelays);
}
```

and correct the `joinFromBundle` comment, which currently misdescribes which branch is load-bearing.

### WR-03: `joinFromBundle` silently discards the app's own configured fallback relays when they don't match the invite-hardening predicate

**File:** `packages/concord/src/client/client.ts:644`, `packages/concord/src/helpers/invite-bundle.ts:107-111`

**Issue:** `isSafeInviteRelayURL` is designed for *stranger-supplied* relay strings, but it is now applied to `this.defaultRelays` — i.e. the `relays` option the application configured itself. Anything that isn't `wss://` or a loopback `ws://` is dropped with no log and no error:

- `ws://192.168.1.10:4869` (a LAN cache relay — the same class of endpoint as the `ws://localhost:4869` the carve-out was written to preserve) → dropped.
- `ws://relay.internal:7777` (a relay reached over a private/VPN network) → dropped.
- Any `wss://` URL that `isSafeRelayURL`'s hostname regex rejects → dropped.

When `bundle.relays` is empty and every configured fallback is filtered out, the joined community gets `material.relays: []`. That is not immediately fatal (`relays()` re-falls back to `defaultRelays` at runtime), but it *is* published: the empty array goes into the kind-13302 document, so the user's **other** devices — which may have a different `relays` option — resolve the community to their own defaults instead of the intended relay set, and `refound()`'s quorum denominator (`mergeRelaySets(this.relays())`) is computed against whatever each device happens to default to.

**Fix:** Do not filter trusted local config with the untrusted-input predicate. Either drop the fallback filter (CR-01 is closed at `decodeFragment` and `validateInviteBundle`), or, if it is kept as defence-in-depth, log when it drops anything and fall back to the unfiltered set rather than to `[]`:

```ts
const safeFallback = fallbackRelays.filter(isSafeInviteRelayURL);
if (safeFallback.length !== fallbackRelays.length)
  this.log("join dropped %d configured relay(s) failing the invite relay gate", fallbackRelays.length - safeFallback.length);
relays: bundle.relays.length ? bundle.relays : (safeFallback.length ? safeFallback : fallbackRelays),
```

### IN-01: `recordJoin`'s reorder trades document residue for engine residue

**File:** `packages/concord/src/client/client.ts:708-717`

**Issue:** The new order eliminates the phantom-membership defect, but it introduces the mirror image: `addCommunity` has already done `communities.set(...)`, `stateSubs.set(...)`, `void community.start()` and `emitCommunities()` by the time `joinCommunity(...)` runs. If that operation throws (or a future refactor makes it fallible), a started, network-active engine is left registered with no document entry — it would run until the next `reconcileCommunities` reaped it as "not live". The 12.3-11 summary reasons carefully about why construction cannot observe `this.list`, but not about the reverse direction.

**Fix:** Wrap the document mutation and roll back the engine on failure, or compute the new list first (pure) and commit both in one non-throwing step:

```ts
const community = this.addCommunity(material);
try {
  this.list = joinCommunity({ ... })(this.list, this.tombstones).communities;
} catch (err) {
  this.removeCommunity(material.community_id);
  throw err;
}
```

### IN-02: `expires_at`, `icon` and `label` survive `validateInviteBundle` untyped

**File:** `packages/concord/src/helpers/invite-bundle.ts:311-369`, `packages/concord/src/client/client.ts:629`

**Issue:** The validator's contract is "bound and self-certify an attacker-crafted bundle", but three fields pass through the `{ ...bundle }` spread unchecked. `expires_at` reaches a bare relational comparison — `if (bundle.expires_at && unixNow() > bundle.expires_at)` (`client.ts:629`) — so a non-number (`{}`, `"soon"`) yields `false` and the expiry check is silently bypassed. `icon` (a `BlobPointer` with a `url`) and `label` are handed to consuming apps for preview rendering with no shape or scheme validation, from an untrusted source.

**Fix:** Coerce-or-drop each: `expires_at` only when `Number.isSafeInteger(...) && >= 0`; `icon` only when it is an object with a `wss`/`http(s)` `url` string; `label` only when a bounded string (see CR-02).

### IN-03: `isHexKey` is case-insensitive, so uppercase-hex protocol state validates and de-canonicalizes the key memo

**File:** `packages/concord/src/helpers/invite-bundle.ts:284-288`, `packages/concord/src/helpers/keys.ts:657-672`

**Issue:** `isHexKey` lowercases before matching (`packages/core/src/helpers/string.ts:12`), so an attacker's `community_root`/`channels[].key` in uppercase hex passes the new CR-02 checks. `hexToBytes` decodes both cases identically so the derived keys are correct, but `deriveChannelKeys`'s memo signature is built from the raw string (`keys.ts:658-664` via `channelKeyMemo`'s `` `p|${id}|${held.key}|${held.epoch}` ``), so a case-flipped duplicate allocates a second memo entry for the same key. It also means `material.channels[].key` is not guaranteed canonical when compared as a string anywhere downstream.

**Fix:** Either normalize to lowercase in the validator (`key: entry.key.toLowerCase()`) or tighten to a strict lowercase check (`/^[0-9a-f]{64}$/.test(entry.key)`), and prefer the latter to keep the validator a shape checker rather than a rewriter, consistent with its stated D-01 posture.
