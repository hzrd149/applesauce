# Phase 12.3: Code Review Report

**Reviewed:** 2026-07-22T16:20:00Z
**Depth:** standard
**Files Reviewed:** 12
**Status:** issues_found

## Summary

The core protocol/transport separation this phase targets **holds**. I traced every write surface
that could leak the merged set into persisted or published artifacts and found none:

- `ConcordCommunity.createInvite` (`community.ts:1237-1253`) correctly splits `protocolRelays`
  (URL) from `transportRelays` (publish).
- `buildInviteBundle(this.material, …)` always sources `bundle.relays` from `material.relays`;
  `transport()` never mutates or is assigned to material.
- `ConcordClient.joinByLink` (`client.ts:534-573`) keeps `relays` unmerged into `joinFromBundle`,
  with `requestRelays` a separate local.
- `InviteWatcher.resolveRelays` never consults `extras`; `relays$` is never fed a merged value.
- `saveCommunityList` / `save()` serialize `this.list` / `this.invites`, never a transport set.

The defects are elsewhere: in `refound()`'s quorum arithmetic (which now mixes a raw,
un-normalized denominator with a normalized numerator and introduces an unguarded
`normalizeRelayUrl` on attacker-influenceable data), in `ExtraRelays`' total absence of error
handling on an app-supplied Observable, in lifecycle disposal gaps (`ConcordClient` and
`ConcordInviteManager` never dispose their holder), and in a set comparator that is not a correct
multiset equality. Separately, the "byte-identical to pre-phase behavior (D-14)" claim repeated in
five doc comments is **factually false** and is contradicted by the phase's own tests.

Tests pass (34/34 for the extras-related suites) and `tsc --noEmit` is clean.

## Critical Issues

### CR-01: `refound()` throws `TypeError: Invalid URL` on attacker-influenceable relay entries

**File:** `packages/concord/src/client/community.ts:1460`

```ts
const protocolRelaySet = new Set(relays.map(normalizeRelayUrl));
```

**Issue:** `normalizeRelayUrl` throws on any string that `new URL()` rejects (verified:
`normalizeRelayUrl("not a url")` → `Invalid URL`; `normalizeRelayUrl("")` → `Invalid URL`;
non-string input → `Invalid URL`). `relays` here is `this.relays()` → `material.relays`, and
`material.relays` for any community joined via an invite comes straight from
`bundle.relays` — which `validateInviteBundle` only checks with `Array.isArray(bundle.relays)`
plus a length cap (`helpers/invite-bundle.ts:230-233`); individual entries are **never validated
as strings or URLs**.

Consequence: a malicious or merely malformed invite link permanently disables **Refounding** for
that community. Refounding is the CORD-06 exclusion/ban mechanism, so this is a denial of a
security-critical operation, reachable from untrusted content. This line is new in this diff —
pre-phase, `refound()` never normalized `relays` at all.

The same unguarded call appears on the ack side (`community.ts:1463`, `normalizeRelayUrl(r.from)`)
— see WR-08.

**Fix:** Derive the protocol set through the same fail-soft path everything else already uses, so
invalid entries are dropped rather than thrown on, and reuse it for both numerator and denominator
(see CR-02):

```ts
import { mergeRelaySets } from "applesauce-core/helpers/relays";

const protocolRelays = mergeRelaySets(relays); // normalized, deduped, invalid dropped
const protocolRelaySet = new Set(protocolRelays);
const majorityThreshold = Math.ceil((protocolRelays.length + 1) / 2);
```

Additionally, harden `validateInviteBundle` to `bundle.relays.filter((r) => typeof r === "string" && isSafeRelayURL(r))`.

### CR-02: Refounding quorum denominator and numerator are computed over different sets

**File:** `packages/concord/src/client/community.ts:1459-1465`

**Issue:** The threshold is derived from the **raw** protocol array:

```ts
const majorityThreshold = Math.ceil((relays.length + 1) / 2);   // raw, un-normalized, un-deduped
const protocolRelaySet = new Set(relays.map(normalizeRelayUrl)); // normalized + deduped
const okCount = responses.filter((r) => r.ok === true && protocolRelaySet.has(normalizeRelayUrl(r.from))).length;
```

but the numerator can only ever count acks from relays that were actually dialled — i.e. from
`transportRelays = mergeRelaySets(relays, extras)`, which **normalizes, dedupes, and silently
drops invalid entries**. Any of the following make `okCount` structurally unable to reach
`majorityThreshold`, so `refound()` aborts unconditionally and forever:

- `material.relays = ["wss://a.test", "wss://a.test/"]` — `relays.length === 2`, threshold 2, but
  the merged set has one entry → max `okCount` 1.
- any entry that fails URL parsing — dropped from `transportRelays`, still counted in the denominator.

This is not hypothetical for `["wss://x", "wss://x/"]`: relay lists are hand-entered user input at
`createNewCommunity(name, description, relays)` and are attacker-supplied via invite bundles.
Pre-phase, `okCount` was `responses.filter(r => r.ok === true).length` with no set-membership
filter, so the two halves at least came from the same publish call.

**Fix:** Compute both from one normalized protocol set (see CR-01's snippet). One derived value,
one source of truth — the same "computed ONCE so the guard can never disagree" rule this phase
already applies in `openLive()`.

### CR-03: `ExtraRelays` silently swallows / globally rethrows errors from the app's Observable

**File:** `packages/concord/src/helpers/relays.ts:60-62`

```ts
constructor(option?: ExtraRelaysOption) {
  this.subscription = toRelaysObservable(option).subscribe((relays) => this.subject.next(relays));
}
```

**Issue:** `extraRelays` is a documented public API input accepting an arbitrary
`Observable<string[]>` supplied by the consuming app (`client.ts:160`). The subscription installs
a **next-only** handler with no `error` callback and `toRelaysObservable` applies no `catchError`.
If the app's source errors (a stream derived from a failing config fetch, a relay-list loader,
a `switchMap` that throws), RxJS routes it to `reportUnhandledError`, which rethrows on a
macrotask — an uncaught exception in the host application, raised once **per engine** (the client,
the invite manager, the watcher, every community, and every private channel each construct their
own holder, so a single bad source produces N simultaneous uncaught errors).

Independently of the crash: `this.subject` is never errored or completed, so `current` silently
freezes at the last good value and every downstream consumer (`connected$`, `authenticated$`, the
`extrasSub` live re-open) goes permanently inert with no signal at all. A library must not let an
app-supplied stream fail this way.

**Fix:** Make the resolver fail-soft and terminal-safe:

```ts
export function toRelaysObservable(option?: ExtraRelaysOption): Observable<string[]> {
  const source = isObservable(option) ? option : of(option ?? []);
  return source.pipe(
    catchError(() => of([] as string[])), // an app-side failure degrades to "no extras", never a crash
    startWith([] as string[]),
    distinctUntilChanged(sameRelaySet),
  );
}
```

and/or pass an explicit error handler to `subscribe` that logs via the package `logger` before
falling back to `[]`.

## Warnings

### WR-01: `sameRelaySet` is not a correct multiset equality — a real change can be suppressed

**File:** `packages/concord/src/helpers/relays.ts:18-24`

**Issue:** The comparator checks `a.length === b.length` and then set-membership of `b`'s elements
in `Set(a)`. With duplicates it returns `true` for genuinely different arrays and is asymmetric:

- `sameRelaySet(["x","y"], ["x","x"])` → `true` (wrong)
- `sameRelaySet(["x","x"], ["x","y"])` → `false`

Because `distinctUntilChanged(sameRelaySet)` is called as `(previous, current)`, an app pushing
`["x","y"]` then `["x","x"]` has the second emission **silently dropped**: every engine keeps
dialling `y` and never dials the intended set. The unit tests
(`helpers/__tests__/relays.test.ts:14-26`) only cover duplicate-free inputs.

**Fix:** Compare deduplicated sets, not lengths:

```ts
export function sameRelaySet(a: string[], b: string[]): boolean {
  if (a === b) return true;
  const setA = new Set(a);
  const setB = new Set(b);
  if (setA.size !== setB.size) return false;
  for (const value of setB) if (!setA.has(value)) return false;
  return true;
}
```

### WR-02: The "byte-identical to pre-phase behavior (D-14)" claim is false and contradicted by this phase's own tests

**Files:** `packages/concord/src/client/community.ts:110-115`, `client.ts:142-146`,
`invite-manager.ts:68-71`, `invite-watcher.ts:61-67`, `private-channel.ts:55-60`

**Issue:** All five doc comments assert that with no `extraRelays` configured the relay sets are
identical to pre-phase. They are not: `transport()` routes through `mergeRelaySets`, which
**normalizes** (`wss://x.test` → `wss://x.test/`), **dedupes**, and **drops invalid URLs** even
when the extras snapshot is empty. The phase's own tests encode the change:

- `private-channel.test.ts:674` — `expect(subscriptionTargets.at(-1)).toEqual(CHANNEL_RELAYS.map((u) => `${u}/`))`
- `invite-watcher.test.ts` diff — the existing status-map key had to be changed from
  `"wss://relay.example"` to `"wss://relay.example/"` for a **pre-existing** test to keep passing.

That second change is the tell: a pre-existing behavior assertion was edited to match new output.
The invariant these comments state is the load-bearing safety claim of the phase, and a future
contributor will trust it.

**Fix:** Correct the comments to state what is true — "with no extras configured the transport set
is the *normalized, deduplicated* form of the base set" — and note the normalization explicitly in
the `extraRelays` option docs, since it changes what consumers see in `pool.status$` keys.

### WR-03: `mergeRelaySets` silently drops malformed protocol relays from every transport target, with no logging

**File:** `packages/concord/src/helpers/relays.ts:82-84` (and every `transport()` call site)

**Issue:** `mergeRelaySets` swallows parse failures per-entry (`core/src/helpers/relays.ts:56-66`).
A malformed entry in `material.relays` — which, per CR-01, is attacker-supplied — now silently
vanishes from *every* request/subscription/publish/auth target. Pre-phase it was passed through to
`pool.publish` / `pool.subscription` verbatim. The community then syncs against a smaller relay set
than its protocol state claims, with zero diagnostics.

**Fix:** In `ExtraRelays.merge`, compare the merged output length against the base and emit a
debug-log line when entries were dropped, so the silent narrowing is at least observable:

```ts
merge(base: string[]): string[] {
  const merged = mergeRelaySets(base, this.current);
  // log when merged.length < new Set([...base, ...this.current]).size
  return merged;
}
```

### WR-04: Auth drivers for removed extras are never torn down — the engine keeps NIP-42-authenticating to de-configured relays

**Files:** `packages/concord/src/client/community.ts:688-694`,
`packages/concord/src/client/private-channel.ts:326-332`

**Issue:** `ensureAuth` is monotonic: every URL ever seen is added to `this.seenRelays` and gets an
`authenticateStreamKeys` driver appended to `this.authDrivers`, which is only unsubscribed in
`dispose()`. When an app pushes a new extras value that *removes* a relay (the exact reactivity
this phase built — `community.test.ts` proves the live subscription drops `EXTRA_ONE`), the
subscription target correctly narrows but the auth driver for the removed relay stays live for the
engine's whole lifetime. The engine keeps holding the relay handle and keeps answering that
relay's NIP-42 challenges with the community's stream keys, on a relay the app explicitly
de-configured. `seenRelays` also means re-adding the relay later never re-registers a driver.

**Fix:** Key the drivers per URL and prune on transport-set change:

```ts
private authDriverByRelay = new Map<string, Subscription>();

private syncAuth(relays: string[]): void {
  const want = new Set(relays);
  for (const [url, sub] of this.authDriverByRelay)
    if (!want.has(url)) { sub.unsubscribe(); this.authDriverByRelay.delete(url); }
  for (const url of want)
    if (!this.authDriverByRelay.has(url))
      this.authDriverByRelay.set(url, this.relayAuth.authenticateStreamKeys(this.pool.relay(url)));
}
```

### WR-05: `ConcordClient` and `ConcordInviteManager` never dispose their `ExtraRelays`

**Files:** `packages/concord/src/client/client.ts:299` + `426-443` (`stop()`),
`packages/concord/src/client/invite-manager.ts:123` + `154-164` (`stop()`)

**Issue:** `ConcordCommunity`, `ConcordPrivateChannel`, and `InviteWatcher` all call
`this.extras.dispose()` in their teardown. `ConcordClient` and `ConcordInviteManager` do not — and
`ConcordClient` has no `dispose()` at all, so its holder's subscription to the app-supplied
Observable survives `stop()` forever. An app that constructs a client per login (logout → new
client) accumulates two permanently-live subscriptions plus their retained `BehaviorSubject`s per
cycle, each pinning the app's extras source hot.

**Fix:** Add `this.extras.dispose()` to `ConcordInviteManager.stop()` and to
`ConcordClient.stop()`. Because `ConcordClient.start()` is restartable after `stop()`, either
re-construct the holder in `start()` from the retained `extraRelaysOption`, or split disposal into
a separate `dispose()` and document `stop()` as pause-only.

### WR-06: `InviteWatcher.stop()` disposes extras but leaves the instance restartable with frozen, non-reactive extras

**File:** `packages/concord/src/client/invite-watcher.ts:234-244`

```ts
stop(): void {
  this.started = false;      // <- restartable
  ...
  this.extrasSub.unsubscribe();
  this.extras.dispose();     // <- one-way
}
```

**Issue:** `stop()` resets `started = false`, so `start()` is a supported follow-up (and `stop()`
is public API). But `extras` is disposed one-way: the source subscription is gone and the internal
subject is completed. After a `stop()`/`start()` cycle the watcher merges a **stale frozen
snapshot** of the extras (`BehaviorSubject.value` survives `complete()`), and `extrasSub` is gone
so later extras emissions never re-open the live subscription — the reactivity the phase's own
D-08/D-09 tests assert is silently lost, with no error.

**Fix:** Rebuild the holder and its subscription on restart, e.g. retain `options.extraRelays` and
in `start()` do `if (this.extras.disposed) { this.extras = new ExtraRelays(this.extraRelaysOption); this.extrasSub = … }`; or make `stop()` non-destructive and move disposal to a distinct `dispose()`.

### WR-07: `private-channel.openLive()` recomputes `transport()` four times, violating the guard-consistency invariant it documents

**File:** `packages/concord/src/client/private-channel.ts:334-353`

**Issue:** `community.ts:719` explicitly computes `const targets = this.transport()` once with the
comment *"Computed ONCE and reused for the guard, the auth registration, and the subscription
target, so the guard can never disagree with what was dialled (D-09/Pitfall 4)."* The mirrored
method in `private-channel.ts` calls `this.transport()` four separate times — for the churn
signature (`:341`), `ensureAuth` (`:345`), the subscription target (`:348`), and a log line
(`:352`). It happens to be safe today only because `extras.current` cannot change synchronously
mid-function; the invariant is asserted by construction in one file and by accident in the other.
The fourth call also allocates a merged array purely to compute `.length` for a debug log.

**Fix:** Mirror the community exactly:

```ts
const targets = this.transport();
const sig = `${[...authors].sort().join(",")}|${[...targets].sort().join(",")}`;
...
this.ensureAuth(targets);
this.liveSub = this.opts.pool.subscription(targets, …);
this.log("live subscription open targets=%d", targets.length);
```

### WR-08: Unguarded `normalizeRelayUrl(r.from)` on publish-response data inside the quorum filter

**File:** `packages/concord/src/client/community.ts:1463`

**Issue:** `r.from` is data returned by the relay transport layer. If any response carries an
empty, absent, or non-URL `from`, `normalizeRelayUrl` throws (verified) from inside the `.filter`
callback, propagating out of `requireMajority` and rejecting `refound()` with an opaque
`TypeError: Invalid URL` rather than the intended `"refounding aborted: … not confirmed by a
majority of relays"`. The failure mode is indistinguishable from a genuine quorum abort to a
caller that matches on the message (the phase's own test does: `rejects.toThrow(/majority/)`).

**Fix:**

```ts
const normalized = (url: string | undefined) => {
  try { return url ? normalizeRelayUrl(url) : undefined; } catch { return undefined; }
};
const okCount = responses.filter((r) => {
  const from = normalized(r.from);
  return r.ok === true && !!from && protocolRelaySet.has(from);
}).length;
```

### WR-09: `ConcordClient` retains no way to release the extras holder and provides no `dispose()`, while documenting per-engine subscription fan-out

**File:** `packages/concord/src/client/client.ts:145-152, 225-232`

**Issue:** The option docs correctly warn that "each engine (this client, every community, the
invite manager, and the invite watcher) subscribes its own source" and tell apps to pass a hot
source. But the class exposes no lifecycle hook that releases those subscriptions (see WR-05), so
the documented consequence is unbounded: for a cold source such as
`defer(() => fetchRelays())`, every community engine re-executes the source on construction, and
nothing ever unsubscribes on the client side. The guidance ("pass a `BehaviorSubject` or
`shareReplay(1)`") is a comment, not an enforced contract, and a cold source is the natural thing
an app reaches for.

**Fix:** Have `ConcordClient` share **one** resolved stream with its sub-engines instead of
handing each the raw option — e.g. pass `this.extras.relays$` (already hot and de-duped) down as
the `extraRelays` value for communities/watcher/manager. This preserves full reactivity (D-11/D-13)
while collapsing N subscriptions on the app's source to exactly one, and makes the "pass a hot
source" caveat unnecessary. Pair with the `dispose()` from WR-05.
