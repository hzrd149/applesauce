## Decisions Made
- The invite-link fragment is a custom byte-packed, base64url-encoded format — a raw substring check against `invite.url` cannot see a leak packed inside it. Every invite-link artifact assertion decodes the link with `parseInviteLink` first and inspects `bootstrapRelays`, never the raw URL string. This was discovered empirically during the mandated non-vacuity revert (see Issues Encountered).
- Canary/protocol relay literals are substring-matched on a bare hostname (`"canary.invalid"`) rather than the exact post-merge normalized string, since `mergeRelaySets` adds a trailing slash.
- `PROTOCOL_RELAYS` is a two-entry literal distinct from both the canary and `STOCK_RELAYS`, so the targeted equality assertions can never pass by coincidence.
- The fake pool serves `request()` from its own growing publish log rather than a static event list, letting a second client instance genuinely fetch the first client's invite bundle through the same shared pool — no separate mock-server machinery needed.
- Task 1 and Task 2 committed as separate `test(...)` commits per the plan's own task split, even though both edit the same file — Task 1's commit stands alone as a complete, buildable, green lifecycle canary before Task 2's targeted block is added.

## Deviations from Plan

None — plan executed exactly as written. The `parseInviteLink`-vs-raw-substring correction below was a fix made DURING Task 1's own mandated non-vacuity verification step (the plan's own acceptance criteria required the revert-and-observe check), not a new scope item.

## Issues Encountered
- The first draft of the lifecycle canary's invite-URL artifact assertion checked `invite.url` for the raw canary substring. The mandated non-vacuity revert (temporarily passing `transportRelays` instead of `protocolRelays` to `buildInviteLink` in `community.ts`) unexpectedly left the lifecycle canary test GREEN — only the targeted invite-link test failed. Root cause: the invite-link fragment is `encodeFragment`'d into a custom byte-packed format and then base64url-encoded, so a literal ASCII substring like `"canary.invalid"` does not survive into the encoded fragment as a matchable substring. Fixed by decoding `invite.url` with `parseInviteLink` first and asserting absence over the resulting `bootstrapRelays` array. Re-ran the revert: both the lifecycle canary and the targeted invite-link test then failed with clear messages (`AssertionError: expected true to be false` for the canary presence check, and a full array diff — `wss://protocol-one.test/`, `wss://protocol-two.test/`, `wss://canary.invalid/` vs the expected two-entry literal — for the targeted test), confirming the fix is non-vacuous. Restored `protocolRelays` and reconfirmed all 5 tests green.
