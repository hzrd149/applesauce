# Phase 12.3: Transport-Only Extra Relays in applesauce-concord - Context

**Gathered:** 2026-07-22
**Status:** Ready for planning

<domain>
## Phase Boundary

`ConcordClient` accepts `extraRelays?: string[] | Observable<string[]>` so apps can
supply app-local relay endpoints (e.g. a `ws://localhost:4869` cache relay) that
Concord uses **purely for network I/O** — as additional transport targets for all
Concord-managed traffic — but that are **never** written into community material or
metadata, invite bundles, invite links, or user-published lists, and never surfaced
as part of the community's protocol state.

**In scope:** the `extraRelays` option on `ConcordClientOptions`, threaded as a
pass-through field into every sub-engine option bag; merging it into every
`pool.request` / `pool.subscription` / `pool.publish` / `ensureAuth` target across
community sync, channel sync, live subscriptions, publishes, community-list and
invite-list reads/writes, direct-invite watching, invite-bundle fetch/publish/revoke,
and relay auth/connection status; reacting to Observable updates; and the regression
guards proving nothing leaks into protocol state.

**Out of scope:** any change to what Concord *stores or publishes* — the fallback
chain (`material.relays` → `options.relays` → `STOCK_RELAYS`) is untouched, and the
refounding quorum arithmetic is unchanged. No changeset (concord unreleased). No
docs/ page (concord is not yet documented under `docs/`); JSDoc only.

</domain>

<decisions>
## Implementation Decisions

### Transport / protocol split — the never-persisted guarantee

- **D-01:** `.relays` keeps its **exact current meaning** everywhere in concord
  (protocol: `material.relays` → `options.relays` → `STOCK_RELAYS`). It is never
  widened. `extraRelays` is carried as an **additional, separate field** alongside
  it, mirroring the applesauce-loaders convention
  (`loaders/tag-value-loader.ts:36,73-81`, `address-loader.ts:154`,
  `event-loader.ts:111`): the option is declared as
  `extraRelays?: string[] | Observable<string[]>`, rides alongside the pointer's own
  relays, and is merged **only at the point of use** via `mergeRelaySets`.
  *This is the structural guarantee:* because `relays()` never returns an extra, an
  app-local endpoint can only appear where someone explicitly wrote
  `mergeRelaySets(...)`. Every protocol write is safe by default.
  *(Superseded an earlier proposal to rename to `protocolRelays()` /
  `transportRelays()` — that rename is dropped. Field names stay `relays`.)*
- **D-02:** Merge helper is `mergeRelaySets` from `applesauce-core`
  (`packages/core/src/helpers/relays.ts:48`) — already a concord dependency, and it
  normalizes URLs and dedupes for free.
- **D-03:** `extraRelays` **do** participate in NIP-42 auth. `ensureAuth` runs over
  the merged set at `community.ts:657,682`, `sync.ts:137,191`,
  `channel-sync.ts:49,96`, `private-channel.ts:289`, so a gating cache relay still
  receives stream-key AUTH and can serve private-channel traffic.
- **D-04:** The `mergeRelaySets` call lives in a **private per-class helper** (e.g.
  `ConcordCommunity.transport()`) rather than being spelled out inline at each of
  ~20 pool call sites. One place resolves the extras snapshot and dedupes; call
  sites stay one-liners. This is a deliberate concord-scale adaptation of the
  loaders' inline style, justified by the site count.
- **D-05:** Regression guard is **canary + targeted, both**.
  - *Canary:* construct a client with `extraRelays: ["wss://canary.invalid"]`, drive
    create-community → create-invite-link → `joinByLink` → metadata edit → refound,
    and assert the canary string appears in **zero published event payloads and zero
    returned artifacts**. Catches leak paths nobody enumerated, including ones added
    in later phases.
  - *Targeted:* a spec-derived assertion at each enumerated protocol write (below).
  - **Critical:** these assert on **event content and returned artifacts, never on
    publish targets** — extras *should* appear as publish destinations; that is the
    feature.
  - TEST-01 (standing) applies: expectations derived independently, non-vacuity proven.

### The two dual-meaning locals (the actual leak surface)

Both are sites where **one local array serves transport and protocol at once**. D-01
covers both, but they must be handled explicitly and are the canary's primary targets:

- **`community.ts:1154-1169`** (`createInviteLink`):
  ```ts
  const inviteRelays = this.relays();
  this.pool.publish(inviteRelays, signed)                              // transport → merge
  url: buildInviteLink(options.base, linkPub, token, inviteRelays)     // PROTOCOL → must not merge
  ```
- **`client.ts:481-514`** (`joinByLink`):
  ```ts
  const relays = parsed.bootstrapRelays.length ? parsed.bootstrapRelays : this.defaultRelays;
  this.pool.request(relays, [...INVITE_BUNDLE_KIND...])   // :492 transport → merge
  return this.joinFromBundle(bundle, relays);             // :514 → material.relays at :546 PROTOCOL
  ```

Other protocol writes to keep clean: `buildInviteBundle(this.material, ...)`
(`community.ts:1142`), `createNewCommunity` → `material.relays` (`client.ts:465`,
explicit caller argument — already safe), the metadata edit path (`admin.ts:157`),
and anything surfaced on `state$.metadata`. Concord publishes no NIP-65/10050 itself,
so the roadmap's "user-published relay lists" clause is reached only via
`material.relays` inside the kind-13302 community list (`client.ts:878`) — covered by
the canary.

### Refounding quorum (ROTATE-09 must not move)

- **D-06:** **Publish to the merged set, count protocol responses only.**
  `refound()` at `community.ts:1358-1364` publishes each gated wrap to the merged set
  (so a cache relay does receive rekey/compaction/snapshot wraps), but the gate
  filters responses by `from ∈ relays()` (URL-normalized —
  `PublishResponse = { ok, message?, from }`, `packages/relay/src/types.ts:78`) and
  keeps `majorityThreshold = Math.ceil((relays().length + 1) / 2)`.
  **Rationale (Phase 8 D-09/D-11/ROTATE-09):** publish acks are the only evidence
  anyone else can discover the new epoch. Counting extras would let a user satisfy
  the discoverability quorum entirely against their own always-OK localhost relay
  while no community relay accepted the wrap; inflating the denominator would raise
  a 2-of-3 bar to 3-of-4 and could make refounding permanently impossible. The
  arithmetic must stay bit-for-bit what Phase 8 settled on.

### Connection / auth status observables

- **D-07:** `connected$` and `authenticated$` **derive from the merged set** (user
  ruling, made with the consequences below in view — document them in JSDoc rather
  than mitigating):
  - `connected$` is `.some()` (`relay-auth.ts:88`) — an always-up local relay pins it
    **true**, so every community relay can be down while the UI still reads "live".
  - `authenticated$` is `.every()` over connected relays (`relay-auth.ts:109-112`).
    An extra that gates nothing short-circuits `true` at line 110 (harmless), but
    since D-03 has extras participating in AUTH, an extra that *does* gate and
    rejects our stream keys pins it **false** indefinitely.
- **D-08:** Those two are built **once at construction** from a snapshot
  (`community.ts:360-361`; `RelayAuth.connected$/authenticated$` take a plain
  `string[]` and close over it), so they must be re-derived reactively:
  **switchMap at the community** —
  `extraRelays$.pipe(switchMap(extra => relayAuth.connected$(mergeRelaySets(this.relays(), extra))))`.
  `RelayAuth`'s public signatures stay plain `string[]`; only the construction sites
  change.

### Observable reactivity

- **D-09:** Live subscriptions are rebuilt via **switchMap** over the extras
  observable: `extraRelays$.pipe(switchMap(extra => pool.subscription(merged, ...)))`.
  Consistent with D-08. **Constraint:** `openLive()`'s existing dedupe guard
  (`community.ts:645-648`, `if (sig === this.liveAuthors && this.liveSub) return`)
  keys on the **author signature only** — its churn-avoidance intent must be
  preserved in the reactive form (e.g. `distinctUntilChanged` over authors + merged
  set), or a no-op emission will needlessly tear down and reopen sockets.
- **D-10:** Consumption shape is **`unwrap` + `startWith([])`**, following
  `packages/loaders/src/helpers/loaders.ts:18-20`. `startWith([])` means a
  not-yet-emitted Observable resolves to the empty set and **never blocks** Concord
  traffic (a `ConcordClient` is long-lived, unlike a one-shot loader).
  *Known consequence to document:* a cold Observable's producer is re-subscribed per
  operation and per sub-engine — apps should pass a hot/shared source
  (`BehaviorSubject`, or `shareReplay(1)`).
- **D-11:** `startWith([])` applies **everywhere**; `take(1)` applies **only to
  one-shot operations**. Applying `take(1)` at a long-lived site would freeze it at
  the first value and silently undo D-08/D-09.
  - **Continuous (no `take(1)`):** community live sub (`community.ts:659`),
    private-channel live sub (`private-channel.ts:292`), invite-watcher live sub
    (`invite-watcher.ts:364`), and `connected$`/`authenticated$`
    (`community.ts:360-361`, `private-channel.ts:80-82`).
  - **Per-operation snapshot:** publishes, `pool.request`, `syncAuthors`/`syncEpoch`
    backfill, invite-bundle fetch.
  - **Rule:** anything holding a socket or emitting over time is continuous.

### Application surface

- **D-12:** Extras union onto **every pool call, no exceptions** — including relay
  sets Concord did not derive from community material: invite-link bootstrap fetch
  (`client.ts:492`), invite-manager revoke to the link's own bootstrap relays
  (`invite-manager.ts:206-207`), user-list reads/writes on `defaultRelays`
  (`client.ts:743,878`), and invite-watcher's discovered NIP-17/NIP-65 inboxes
  (`invite-watcher.ts:213,364`). Matches the roadmap's "all Concord-managed traffic";
  a cache relay then also caches the join path, where it helps most.
- **D-13:** **Client-level configuration only.** A single `extraRelays` on
  `ConcordClientOptions`, threaded down as a pass-through field into every community,
  private-channel, invite-manager and invite-watcher option bag. One source of truth,
  one leak surface to reason about. Per-community override is deferred.
- **D-14:** extras are **purely additive, always** — they never suppress or
  substitute for anything, and the fallback chain
  (`material.relays` → `options.relays` → `STOCK_RELAYS`) is untouched. An app
  wanting local-only traffic passes `relays: ["ws://localhost:4869"]` as the default
  set, which is the existing supported mechanism.

### Milestone conventions carried forward

- **D-15:** concord is unreleased — **no changeset**, and breaking-change caution
  does not apply. See [[concord-unreleased-no-changesets]]. Carried from Phase 12.2
  D-10.
- **D-16:** The namespaced `debug` logging landed in Phase 12.2
  (`applesauce:concord:*`) is available; new relay-merge plumbing should log under it
  where useful. Derive/store the `Debugger` once — never `.extend()` inline at a log
  call site. See [[debug-logger-hoist-extend]].

### Claude's Discretion

- Exact name of the per-class merge helper from D-04 (`transport()` is a working
  proposal) and whether each sub-engine gets its own or shares one.
- Whether `ConcordPrivateChannel`'s `connected$`/`authenticated$`
  (`private-channel.ts:80-82`) receive the same merged treatment as the community's
  (consistency argues yes).
- Exact JSDoc wording for the D-07 caveats (localhost pinning `connected$` true, a
  gating extra pinning `authenticated$` false) and the D-10 hot-source
  recommendation.
- Whether rapid `extraRelays` emissions need debounce before socket teardown, and
  whether duplicate events during a socket swap are already absorbed by the
  event store (they most likely are).
- Whether `storage.ts` needs an explicit assertion that extras are never persisted
  across reload (extras are app-supplied per construction, so this should be true by
  construction).
- Whether the canary test lives as one suite or is split per entry point.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `mergeRelaySets` (`applesauce-core`) — normalizes and dedupes in one call; no new
  helper needed, and no new dependency.
- `unwrap` (`applesauce-loaders`) — the Observable-or-array resolver the loaders use;
  concord already depends on applesauce-loaders. Usable directly for one-shot sites;
  long-lived sites need the un-`take(1)`-ed form (D-11).
- `RelayAuth.lookupStatus` (`relay-auth.ts:76-78`) — already tolerates un/normalized
  relay URLs; the same tolerance is what D-06's `from`-attribution needs.
- The Phase 12.2 `applesauce:concord:*` debug namespaces are in place for tracing the
  merged relay set at each call site.

### Established Patterns
- Concord consistently derives its relay set through a fallback chain
  (`material.relays` → `options.relays` → `STOCK_RELAYS`) and passes a plain
  `string[]` snapshot down into sub-engine option bags. `extraRelays` rides alongside
  as a second field rather than being folded into that chain (D-01, D-14).
- Sub-engines (`ConcordPrivateChannel`, `SyncContext`, `ChannelSyncContext`,
  `InviteManager`, `InviteWatcher`) are all pure transport consumers — none writes
  protocol state — so merging is unconditionally safe inside them.
- Live subscriptions are managed imperatively (`unsubscribe()` then reassign) with a
  signature guard to avoid churn; D-09's switchMap must reproduce that guard rather
  than drop it.
- Publish paths are best-effort `.catch()` except the refounding gate, which is the
  one place a response count carries a security meaning.

### Integration Points
- `ConcordClientOptions` (`client.ts:114-243`) is the single injection point (D-13),
  sitting next to the existing `relays?: string[]` fallback option.
- Every sub-engine option bag gains an `extraRelays` pass-through field.
- `community.ts:360-361` is where the status observables become reactive (D-08).
- `openLive()` / `private-channel.ts:289-293` / `invite-watcher.ts:361-364` are where
  live sockets become reactive (D-09).

<specifics>
## Specific Ideas

The user's framing: *"we should keep the `.relays`, but extra relays should be an
additional name of `extraRelays` in a similar way to how the applesauce-loaders are
handling them."* That is the anchor — concord's `extraRelays` should read and behave
like the loaders' `extraRelays` to anyone already familiar with the loaders API, with
concord-scale adaptations (a per-class merge helper instead of ~20 inline merges, and
a continuous form at the long-lived sites) made explicit rather than incidental.

The motivating setup: an app running a local cache relay at `ws://localhost:4869`
wants every byte Concord touches to also land in that cache — including the join
path — while a shared invite link generated by that app must be openable by someone
who has never heard of `localhost:4869`.

</specifics>

<deferred>
## Deferred Ideas

- **Per-community / per-channel `extraRelays` override** (`ConcordCommunityOptions.extraRelays`),
  for an app routing one community through a dedicated cache. Deferred by D-13 —
  it multiplies the override semantics (replace vs add) and every leak-surface
  argument would have to hold for two sources.
- **A separate extras-only status observable** exposing an app-local relay's own
  connection/auth health without conflating it with community health. Considered
  under D-07 and set aside as new public API beyond the roadmap goal; it is the
  natural mitigation if the merged-status caveats prove annoying in practice.
- **Extras satisfying the STOCK_RELAYS fallback** (local-only setup via a single
  option). Rejected by D-14 as blurring the transport-only line; the existing
  `relays:` option already covers the use case.
- **Concord documentation under `docs/`** — the package has no docs page at all yet.
  Out of scope here (unreleased); JSDoc only.
