# Phase 12.3 Plan 10: Auth-driver prune-and-refresh + extras-holder lifecycle Summary

**`community.ts`/`private-channel.ts` gained a per-URL, prune-on-narrow NIP-42 auth-driver registry; `client.ts`/`invite-manager.ts`/`invite-watcher.ts` gained a uniform stop()-is-pause / dispose()-releases lifecycle with a restart-safe extras subscription; and all five engine `extraRelays` option JSDoc sites were corrected to the true D-14 guarantee — closing every remaining WARNING-severity anti-pattern from 12.3-VERIFICATION.md with 12 new regression tests.**

## Performance

- **Duration:** ~30 min
- **Tasks:** 3
- **Files modified:** 9 (5 source, 4 test)

## Accomplishments

- **WR-04/WR-07 (Task 1):** `ConcordCommunity` and `ConcordPrivateChannel`'s `authDrivers` field changed from a single aggregate `rxjs.Subscription` plus a monotonic `seenRelays` Set to a `Map<string, Subscription>`. `ensureAuth(relays)` now synchronizes the registry against its argument: any driver whose URL left the set is unsubscribed and deleted (so a de-configured relay stops receiving NIP-42 responses driven by the community's/channel's stream keys), and a fresh driver registers for any URL not yet present — including a URL re-added after removal, which the old monotonic set silently suppressed. An entry already present and still wanted is left completely untouched, so a no-op re-emission of an unchanged transport set churns nothing (D-09). `dispose()` on both classes now unsubscribes every registry entry. Separately, `private-channel.ts`'s `openLive()` now computes its merged transport set exactly ONCE into a `targets` local (mirroring `community.ts`'s existing pattern), reused for the churn-guard signature, the auth registration, the subscription target, and the debug log's length — replacing four separate `transport()` calls that previously only agreed with each other by accident.
- **WR-05/WR-06 (Task 2):** All five Concord engines now follow one lifecycle rule — `stop()` is pause-only (restartable, leaves the extras holder alive and subscribed) and `dispose()` is what releases it. `ConcordCommunity`/`ConcordPrivateChannel` already followed this; `InviteWatcher`, `ConcordInviteManager`, and `ConcordClient` now do too. `InviteWatcher`'s extras-reactivity subscription creation is extracted into a private `subscribeExtras()` method, called from both the constructor and `start()` (re-subscribing only when `start()` finds the prior subscription closed by a preceding `stop()`) — the `ExtraRelays` holder object itself is never rebuilt across a restart, since `needsAuth$`'s `combineLatest` is closed over its stream and rebuilding the holder would silently detach that derivation. `InviteWatcher.stop()` no longer disposes the holder; a new public `dispose()` stops, unsubscribes the extras subscription, and disposes the holder. `ConcordInviteManager` gained an analogous `dispose()` (its `stop()` was already non-destructive of the holder). `ConcordClient.stop()` now disposes (not merely stops) the discarded `InviteWatcher` instance — a discarded watcher is never restarted, so its extras subscription must be released at that point — and a new public `ConcordClient.dispose()` stops, disposes the client's own holder, the invite manager, and the invite watcher (if still running).
- **WR-02 residue (Task 3):** The `extraRelays` option JSDoc on all five engines (`community.ts`, `client.ts`, `invite-manager.ts`, `invite-watcher.ts`, `private-channel.ts`) now names `ExtraRelays.merge`'s empty-extras identity fast path as the D-14 mechanism, and adds the previously-missing note that a *configured* extras set routes through `mergeRelaySets`, changing the shape of relay-target strings and `pool.status$` lookup keys for that configuration. `community.ts`'s site additionally asserted unqualified "every arithmetic result in this file is byte-identical to pre-phase" — rewritten to state both halves precisely: `refound()`'s majority threshold and ack-attribution set are derived from `relays()` alone (never extras), bit-for-bit identical to pre-phase arithmetic for a well-formed, duplicate-free protocol relay list, and fail-soft (never a raw parse crash) for a degenerate one — matching plan 09's fix and its adjacent in-code comment, which was already accurate. `client.ts`'s option JSDoc also now names `dispose()` as the release path for the app's extras source. No executable statement changed in this task (verified via a comment-only diff check).
- 12 new regression tests added across the four client-layer test files: 3 in `community.test.ts` (auth-driver prune-on-narrow, re-add-registers-fresh, no-churn-on-no-op, dispose-tears-down-all), 3 in `private-channel.test.ts` (identical coverage for the channel engine), 3 in `client.test.ts` (client `dispose()` releases every subscriber, client `stop()` is pause-only across a restart, `ConcordInviteManager.dispose()` in isolation), and 3 in `invite-watcher.test.ts` (restart-safe reactivity, `dispose()` releases the source, `stop()` alone does not).

## WR-09 deferral rationale (restated per plan's `<verification>` requirement)

WR-09 proposes threading `ConcordClient.extras.relays$` down to sub-engines instead of each constructing its own holder over the raw `extraRelays` option, collapsing N independent cold-source subscriptions into one hot one. This remains **DEFERRED**, not a verification gap: it is a D-13 threading change (an architectural shift in how the option is consumed, not a correctness defect), and its stated failure mode — a cold source re-executing per engine — is a documented consequence of D-10 (the option's own JSDoc tells an app to pass a hot/shared source if several engines should react as one), already fully mitigated by plan 08's `catchError`/fail-soft handling. Task 2 in this plan further bounds its consequence by making every holder genuinely releasable via `dispose()`, so even in the worst case (many cold subscriptions) an app that disposes its client leaves nothing running. Recorded here again, per this plan's `<verification>` block, so the re-verification run can see it was decided, not silently dropped.

## Deviations from Plan

### Noted discrepancy (not a deviation — no code change made)

**1. Task 1's acceptance-criteria grep for `private-channel.ts`'s `this.transport()` count**
- **What the plan expected:** `grep -c 'this.transport()' packages/concord/src/client/private-channel.ts` is at most 2 (syncContext + openLive's single local).
- **What's actually in the file:** 4 matches. Two of those are the pre-existing (plan-02-era) `connected$`/`authenticated$` constructor bindings (`this.extras.relays$.pipe(switchMap(() => ...this.transport()...))`), which are outside this task's `<action>` scope (only `openLive()`'s four calls were named for consolidation) and structurally cannot be hoisted into a single shared local — they are two independent reactive pipelines that must each read the current snapshot fresh inside their own `switchMap` callback whenever `extras.relays$` emits.
- **Why not fixed:** Community.ts, which the plan does NOT hold to this grep bound, has the identical constructor-level `connected$`/`authenticated$` `this.transport()` calls (2 more, plus ~10 other one-shot call sites) — confirming this pattern is expected, pre-existing engine architecture, not something this task's diff introduced. The substantive `<done>` criterion — "the private-channel live-open path derives its transport target exactly once" — is fully met (`openLive()` now has exactly one `this.transport()` call, down from four).
- **Files affected:** None (no code change; documented here for the re-verification pass).

**Total deviations:** 0 code deviations (Rules 1–4 not invoked); 1 documented acceptance-criteria discrepancy above.
**Impact on plan:** None on functionality or scope — the discrepancy is a plan-authored grep bound that didn't account for two pre-existing, out-of-scope call sites.

## Issues Encountered

Two test-authoring pitfalls, both resolved before landing (not deviations — normal TDD iteration):
- The new auth-driver tests initially looked up `driverSubs` by the raw (un-normalized) test constant; `ensureAuth()` receives the already-`mergeRelaySets`-normalized transport set (a trailing slash appended), so the recorded registry key is the normalized form. Fixed by normalizing the lookup constants with `normalizeURL` once per describe block.
- The shared `extrasPool()`/`servingPool()` test harnesses used across this file return ONE shared relay object for every URL (fine for target-array assertions, but would collapse `ConcordRelayAuth`'s internal per-relay driver map, keyed by `relay.url`, into a single shared driver). The new auth-driver-lifecycle describe blocks use a dedicated harness returning a distinct relay object per URL instead, so `authenticateStreamKeys`'s spy correctly tracks one Subscription per relay.
