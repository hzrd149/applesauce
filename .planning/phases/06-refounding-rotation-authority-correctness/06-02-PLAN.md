<objective>
Fix H02 / ROTATE-04: a Refounding must actually drop excluded members from the Complete Memberlist. Today `planeStoreKey` (`sync.ts:254-256`) flattens every epoch's guestbook into one `"guestbook"` store and `rewireState`/`models/community.ts` merge `observed` across every plane and epoch, so `foldMembers`' forward-observation `!c` branch (`guestbook.ts:109-111`) re-admits a removed member's prior-epoch Join or authorship. Per CORD-02 §5 — "the Guestbook rides the epoch" — the fix is structural epoch separation (D-01/D-02), not a timestamp heuristic: key the guestbook store by epoch so the fold reads only the current epoch, and scope the live `observed` set to current-epoch activity. `foldMembers` itself is spec-correct and stays untouched (Pitfall 3).

Purpose: close the milestone's "prior-epoch activity resurrects an excluded member" defect (EoP / roster tampering) while staying inside Phase 6's boundary — no channel epoch-keying (Phase 7), no control-fold/compaction change (Phase 8/9).
Output: epoch-scoped guestbook stores, a scoped live observed set, the D-03 retention trim, the ROTATE-04 regression + D-04 keep-list tests, and a patch changeset.

## Design decisions resolved here (Claude's Discretion + RESEARCH Open Questions)
- **Observed composition (discretion "which planes feed observed"):** live `observed` = **current-epoch guestbook + channel stores only**. Control, dissolved, and rekey are EXCLUDED from the live observed set. Rationale: closes the guestbook resurrection vector (primary H02) and the control-authorship vector conservatively, WITHOUT epoch-partitioning the control store (which would change the roster fold's input, stale the admin's captured control-store reference, and create a compaction-reliability dependency = Phase 8/9). A genuinely-present admin appears via their guestbook Join; the spec places no plane restriction on observation, so narrowing observation is fail-safe (it can only shrink the memberlist, never resurrect a removed member).
- **Open Question 2 (should rekey traffic feed observed?):** No — rekey is protocol rotation traffic, not conversational "publishing" (CORD-02 §5). It is excluded from observed. The rekey store stays a single un-epoch-keyed store; `readRekeyScoped`'s `set.newEpoch === heldEpoch + 1n` filter already discards stale-epoch rekey wraps, so it needs no epoch key.
- **Open Question 1 (public-channel observed residual):** a member with an OLD PUBLIC-channel message could still be `observed` because channel stores are deliberately un-epoch-keyed (continuous chat history) — DEFERRED to Phase 7 (channel keying) with an explicit code comment. Task 2 adds a regression test documenting the current behavior for this scenario so the residual is proven, not assumed. The sync walk already computes `observed` per-epoch (`sync.ts:163-165`), so the exposure is limited to the live path.
- **Dissolved plane:** its address does NOT rotate with epoch (`dissolvedGroupKey(cid)` takes no root/epoch), so it stays a single un-epoch-keyed store and is excluded from observed (a tombstone plane, not activity).
</objective>
