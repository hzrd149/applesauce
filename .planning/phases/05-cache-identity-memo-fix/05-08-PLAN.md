**`encrypted-content.ts` — `setEncryptedContentCache` (CR-02 mirror).**
The current comment concludes "this write site is itself an identity memo" while simultaneously justifying the enumerable write by "preserves that carry-forward half's requirement to survive a spread". Both cannot hold: by D-05's write-site rule, a write that must survive a spread is a carry-forward payload. Reclassify the site as **carry-forward payload**, matching 05-06's corrected worked example. State the real reason it hand-rolls an enumerable write: an unlocked signed event re-entering the factory pipe hits `modifyPublicTags`'s `{ ...draft, tags }`, which copies only enumerable properties, so a non-enumerable write would be dropped there and force a repeat signer round-trip to re-decrypt. Its purpose is memoization, but purpose does not decide the category — the spread-survival requirement at the write site does. Cite `operations/tags.ts` by function name (`modifyPublicTags`), not line number.

**`cast.ts` — `performCast`.**
The current comment claims a copy must not inherit a stale cast bound to a different instance. The `Reflect.set(event, CASTS_SYMBOL, new Map(...))` beneath it is enumerable, so a spread copy inherits the SAME `Map` by reference — `performCast(copy, cls)` returns the cast constructed against the ORIGINAL event, and the subsequent `casts.set(cls, cast)` mutates a `Map` shared by both objects. Reword to state: the value is derived from and bound to this specific event instance (identity-memo shaped per `cache.ts`'s taxonomy), but this write is enumerable, so a spread copy today inherits the same `Map` by reference — aliased mutable state, not merely a stale value. Say plainly that this is a known, deliberately-deferred gap: migrating it onto the memo helper is out of this phase's scope, and only `pipeFromAsyncArray`'s delete loop masks it, on the one call path that runs it.

**`filter.ts` — `getIndexableTags`.**
Same shape. The memo rides onto `{ ...event, tags: newTags }`, so filter matching on the copy evaluates the ORIGINAL's tags and the `if (!indexable)` guard never recomputes because the copy has an inherited `Set`. Reword with that concrete consequence and the same known-gap framing.

For all three: keep the category name and the `cache.ts` pointer (D-06). Delete every assertion that the value does not survive a spread — replace it with what the code really does. Do not write a claim you have not confirmed against the code you just read.

Comment-only. Not one non-comment character may change.
