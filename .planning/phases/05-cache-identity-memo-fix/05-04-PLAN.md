## The D-18 prohibition (READ THIS BEFORE WRITING EITHER TEST)

**This milestone exists because all 189 concord tests passed while 9 HIGH bugs were live ‚Äî every one of those tests compared the implementation to itself.** These two tests are the first that do not. Getting them right matters more than getting them written.

**Expected values are computed from the spec formula, never by calling the implementation under test.**

The implementation under test in this phase is the **caching + spread mechanism**: `cache.ts`, `baseKeysFor`, `deriveConcordKeys`, `rollForward`, `channelKeyMemo`, `deriveChannelKeys`, `rollForwardChannel`. Expected values must not pass through any of them.

**Permitted source of expected values:** `controlGroupKey` and `channelGroupKey` from `packages/concord/src/helpers/crypto.ts:118-125`, called directly with the spec's `(label, secret, id, epoch)` tuple supplied explicitly. This is settled and user-confirmed ‚Äî do not re-open it. It counts as hand-derivation because `crypto.ts` is a distinct module: a byte-exact transcription of CORD-02 Appendix A that `PROJECT.md` registers as separately audited and out of scope. Calling it bypasses `baseKeysFor`/`rollForward` entirely, so the expected value does not depend on whether the memo bug is present. This is literally what the audit's own H01 proof did (`.planning/concord-audit.md:42-52`).

**PROHIBITED ‚Äî any self-referential form.** Do NOT write assertions shaped like:
- `rollForward(...).control.pk === deriveConcordKeys(oldMaterialWithNewRoot).control.pk`
- `deriveChannelKeys(material, rolled).current.pk === deriveChannelKeys(material, handBuiltChannel).current.pk`
- any expected value obtained by calling `deriveConcordKeys`, `deriveChannelKeys`, `baseKeysFor`, `rollForward`, or `rollForwardChannel`.

These reproduce the exact failure mode this milestone closes.

## The non-vacuity requirement (the subtle part ‚Äî do not skip)

A test that never arms the memo **passes against the broken code** and proves nothing.

The bug is that a memo written on the SOURCE object rides the spread onto the copy. If the source object never had a memo written on it, there is nothing to ride along, and the assertion passes with or without the 05-01 fix.

**Therefore each test MUST derive keys from the SOURCE object BEFORE rolling it forward** ‚Äî that write is what arms the bug. For H01(a) this happens naturally (`rollForward` takes a `ConcordKeys`, which came from `deriveConcordKeys(material, [])`). For H01(c) it does NOT happen naturally and must be done deliberately: call `deriveChannelKeys(material, channel)` on the ORIGINAL channel before `rollForwardChannel`, or the memo is never written and the test is vacuous.

Both tests carry a comment stating this. 05-05 systematically proves both are non-vacuous by temporarily reverting the fix.
