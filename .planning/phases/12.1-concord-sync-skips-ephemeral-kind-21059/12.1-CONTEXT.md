# Phase 12.1: Concord Sync Skips Ephemeral Kind 21059 - Context

**Gathered:** 2026-07-21
**Status:** Ready for planning

<domain>
## Phase Boundary

The concord community sync **backfill/historical-fetch path stops requesting kind
21059** (`EPHEMERAL_GIFT_WRAP_KIND`), an ephemeral gift-wrap kind that relays never
retain under NIP-01. The sole backfill entry point is `syncAuthors`
(`packages/concord/src/client/sync.ts:100`), which today issues
`{ kinds: [GIFT_WRAP_KIND, EPHEMERAL_GIFT_WRAP_KIND], authors }`. `syncAuthors` is
shared by the core-plane walk (`sync.ts`) and the channel walk (`channel-sync.ts`),
so a single fix covers every backfill path.

**Live subscriptions are explicitly OUT of scope and must keep requesting 21059** —
ephemeral wraps only ever arrive over a live subscription. The two live sites stay
untouched: `community.ts:619` and `private-channel.ts:256`.

This is a self-contained bugfix. No behavioral change to how retained (kind-1059)
wraps sync; only the dead ephemeral half of the backfill filter is removed.

</domain>

<decisions>
## Implementation Decisions

### Fix shape & regression guard
- **D-01:** Introduce an exported constant `BACKFILL_KINDS = [GIFT_WRAP_KIND]` in
  `packages/concord/src/helpers/gift-wrap.ts`, alongside the existing exported kind
  constants (`GIFT_WRAP_KIND`, `EPHEMERAL_GIFT_WRAP_KIND`, etc.). Use it as the
  `kinds` in `syncAuthors` (`sync.ts:100`). The constant name — plus a short
  rationale comment at its definition (ephemeral 21059 is never retained by relays,
  so it is intentionally excluded from backfill) — is the structural guard against
  someone silently re-adding 21059 to the walk.
- **D-02:** The two live-subscription sites (`community.ts:619`,
  `private-channel.ts:256`) keep their inline `[GIFT_WRAP_KIND, EPHEMERAL_GIFT_WRAP_KIND]`
  literal. Do NOT route them through a shared constant — the asymmetry (live keeps
  both kinds, backfill keeps one) is the point, and only the backfill site gets a
  named constant.

### Test strategy
- **D-03:** Two-directional filter-spy, reusing the existing `servingPool` test
  harness (`sync.test.ts:30`, mirrored in `private-channel.test.ts`) whose
  `request`/`subscription` methods receive the emitted filters:
  - Backfill direction: spy the filter `syncAuthors` passes to the sync loader and
    assert `kinds` is `[GIFT_WRAP_KIND]` with 21059 absent.
  - Live direction: assert a live-subscription filter still contains BOTH
    `GIFT_WRAP_KIND` and `EPHEMERAL_GIFT_WRAP_KIND`.
- **D-04:** Non-vacuity per standing **TEST-01** — the backfill assertion MUST fail
  against the current pre-fix two-kind filter (record the non-vacuity probe), so a
  pass genuinely attributes to the fix and not to a filter-matching quirk of the
  serving pool.

### Scope of the "misleading synced signal"
- **D-05:** **Filter-only.** The goal's "misleading synced signal" is the *reason*
  this defect matters, not a separate work item — removing the dead 21059 backfill
  filter already stops the false signal (the sync barrier no longer waits on / reports
  a kind that can never arrive historically). No synced-status / barrier-semantics
  rework in this phase; any such concern is deferred (see Deferred Ideas). No extra
  doc beyond D-01's constant + comment.

### Milestone conventions carried forward
- **D-06:** concord is unreleased — **no changeset** for this fix, and
  breaking-change caution does not apply. (See `concord-unreleased-no-changesets`.)

### Claude's Discretion
- Exact constant name (`BACKFILL_KINDS` is the working name; a clearer synonym is
  acceptable if it reads better next to the existing kind constants) and precise
  wording of the rationale comment.
- Test file placement (extend `sync.test.ts` vs a new focused spec) and the exact
  spy mechanism, provided both directions and the non-vacuity probe are covered.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `servingPool(events)` (`sync.test.ts:30`): dependency-injected `RelayPool`
  stand-in — no sockets, serves matching events and completes. Its `request` and
  `subscription` methods are the natural filter-spy points for the two-directional
  test.
- Exported kind constants in `helpers/gift-wrap.ts` establish the `export const KIND
  = ...` convention `BACKFILL_KINDS` follows.

### Established Patterns
- `syncAuthors` centralizes ALL backfill filter construction — one filter definition
  serves core-plane and channel walks. Fixing the filter there is complete by
  construction; no per-caller changes needed.
- Backfill (historical, via `createSyncLoader` — NIP-77 negentropy or paginated REQ)
  vs live subscription is an existing architectural split; this phase makes the
  kind-set difference between them explicit rather than incidental.

### Integration Points
- `createSyncLoader` (`applesauce-loaders/loaders`) consumes the `filter` object from
  `syncAuthors`; the constant flows straight into it with no signature changes.

</code_context>

<specifics>
## Specific Ideas

The asymmetry is deliberate and is the crux of the phase: **backfill = retained kinds
only (`[GIFT_WRAP_KIND]`); live = retained + ephemeral (`[GIFT_WRAP_KIND,
EPHEMERAL_GIFT_WRAP_KIND]`).** A reviewer seeing both sites should be able to read
that distinction directly from the code.

</specifics>
