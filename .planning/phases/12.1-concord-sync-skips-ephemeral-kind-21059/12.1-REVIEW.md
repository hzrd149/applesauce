# Phase 12.1: Code Review Report

**Reviewed:** 2026-07-22T08:27:23Z
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

The phase change is small and well-scoped: it introduces `BACKFILL_KINDS = [GIFT_WRAP_KIND]`
in `helpers/gift-wrap.ts` and routes the historical backfill filter in `syncAuthors`
(`client/sync.ts`) through it, so backfill REQs no longer request the ephemeral kind 21059
(which relays never retain). Two directional filter-spy tests were added.

I traced the correctness of the change adversarially:

- **All backfill paths funnel through `syncAuthors`.** `community.ts` (initial walk via
  `syncEpochs`, freshly-revealed-channel catch-up at `community.ts:641`) and `private-channel.ts`
  (`syncChannelEpochs`, forward-epoch catch-up at `private-channel.ts:304`) all delegate to
  `syncAuthors`, so the single-point fix is genuinely centralized — no second backfill site
  was left requesting 21059.
- **Both live-subscription sites still request both kinds.** Verified `community.ts:619` and
  `private-channel.ts:256` retain `[GIFT_WRAP_KIND, EPHEMERAL_GIFT_WRAP_KIND]`, matching the
  `BACKFILL_KINDS` doc comment's claim. The comment is accurate.
- **Negentropy local-store interaction is sound.** `createSyncLoader` reconciles against
  `ctx.eventStore`, which also holds 21059 wraps delivered live; because both sides of the
  reconciliation are constrained by the `{ kinds: [1059], ... }` filter, the retained 21059
  events do not pollute the reconciliation set.

I could not prove any incorrect runtime behavior introduced by this change — hence no BLOCKER.
The findings below are robustness, maintainability, and test-reliability concerns.

## Warnings

### WR-01: `BACKFILL_KINDS` is an exported mutable array (no freeze / `as const`)

**File:** `packages/concord/src/helpers/gift-wrap.ts:40`
**Issue:** `export const BACKFILL_KINDS = [GIFT_WRAP_KIND];` exports a mutable array reference.
`const` only prevents reassignment, not mutation. Any consumer that does
`BACKFILL_KINDS.push(...)` / `.sort()` / `[0] = ...` silently corrupts the filter for every
backfill in the process — and because this array is the single source of truth for "what a
backfill is allowed to request," a stray mutation could reintroduce the exact ephemeral-kind
bug this phase fixes, with no compile-time signal.
**Fix:**
```ts
export const BACKFILL_KINDS = [GIFT_WRAP_KIND] as const;
// or, if a mutable-typed number[] is required by callers:
export const BACKFILL_KINDS: readonly number[] = Object.freeze([GIFT_WRAP_KIND]);
```

### WR-02: Time-based `settle()` makes `private-channel.test.ts` flakiness-prone

**File:** `packages/concord/src/client/__tests__/private-channel.test.ts:26`
**Issue:** `const settle = () => new Promise((r) => setTimeout(r, 200));` gates the assertions
(`store.getTimeline(...)`, `sub.epoch$.value`, `persisted?.epoch`, `phase$.value`) behind a
fixed 200ms wall-clock wait after `sub.start()`. If the served-wrap sync/decrypt work exceeds
200ms under CI load (NIP-44 decrypt + verify runs per wrap), the assertions race the async
sync and fail intermittently — a false negative unrelated to the code under test. Fixed
`setTimeout` waits are the classic flaky-test pattern.
**Fix:** Await a deterministic condition instead of the clock — e.g. poll `sub.phase$` until it
reaches `"live"`, or await the first `status$` emission with `phase: "live"`, before asserting:
```ts
await firstValueFrom(sub.status$.pipe(filter((s) => s.phase === "live")));
```

### WR-03: Live-subscription kinds duplicated as inline literals across two sites

**File:** `packages/concord/src/client/private-channel.ts:256` (also `client/community.ts:619`)
**Issue:** This phase gave the backfill direction a named constant (`BACKFILL_KINDS`) but left
the symmetric "live" direction as the inline literal `[GIFT_WRAP_KIND, EPHEMERAL_GIFT_WRAP_KIND]`
duplicated in two live-subscription sites. The asymmetry is a maintenance trap: a future change
that adds another retained wrap kind updates `BACKFILL_KINDS` in one place but must separately
locate and update both inline live literals, and nothing enforces they stay in sync. (Noted here
because `private-channel.ts` is the in-scope file carrying one copy; `community.ts:619` is the
other and is referenced, not in the review scope.)
**Fix:** Mirror `BACKFILL_KINDS` with a `LIVE_KINDS = [GIFT_WRAP_KIND, EPHEMERAL_GIFT_WRAP_KIND]`
export in `helpers/gift-wrap.ts` and reference it from both live-subscription sites.

## Info

### IN-01: `decodeWrapCached` memoises independent of `convKey` — latent aliasing footgun

**File:** `packages/concord/src/helpers/gift-wrap.ts:126`
**Issue:** The cache symbol keys only on the wrap instance, not on `convKey`. If the same wrap
instance is ever passed to `decodeWrapCached` under two different keys, the second call returns
the first key's result (including a cached `null`). Current callers in `sync.ts` derive `convKey`
deterministically from `keys.planes.get(ev.pubkey)`, so each wrap is only ever decoded under one
key and the code is safe today — but the invariant is enforced only by a doc comment, not by the
signature. A future caller that probes a wrap under multiple keys (which the sibling pure
functions `getWrapSeal`/`decodeWrap` explicitly advertise as supported) would get silently wrong
results.
**Fix:** No change required now; consider asserting/labelling the bound `convKey` on the cache
entry (or documenting the invariant at the call sites) so the aliasing hazard is not a
comment-only guarantee.

### IN-02: `syncAuthors` inbound `authors` are not deduplicated/validated

**File:** `packages/concord/src/client/sync.ts:95`
**Issue:** `syncEpoch` builds `coreAuthors` from four derived plane pubkeys and passes them
straight into the REQ filter. These are trusted, derived values so this is not a security issue,
but duplicate or malformed author entries would flow into the relay filter unfiltered. Minor
robustness note only — not introduced by this phase.
**Fix:** Optional: `Array.from(new Set(authors))` at the top of `syncAuthors` if callers might
ever pass overlapping author lists.
