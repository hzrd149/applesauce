# Phase 05.1 Plan 03: Core Input-Safety Fixes (CR-05, Site-1) Summary

**stamp() now copies-then-deletes id/sig instead of mutating the caller's draft, and modifyHiddenTags's out-of-pipe unlock temp object uses a descriptor-preserving copy instead of a symbol-dropping spread — both regression-tested ahead of their sites' non-enumerable migrations.**

## Accomplishments
- `stamp()` (`operations/event.ts`) builds its copy first (`{ ...draft, pubkey }`), then deletes `id`/`sig` from the copy — never from the caller's `draft` — mirroring the existing `stripSignature`/`stripStamp` pattern. The `EncryptedContentSymbol` explicit re-copy blocks in `stamp`/`sign` (D-12 KEEP-AS-IS) are byte-for-byte unchanged.
- `modifyHiddenTags` (`operations/tags.ts`) replaces its `{ ...draft, pubkey }` temp object — passed into `unlockHiddenTags` — with `copyDraftWithPubkey`, a private helper that copies every own property descriptor (via `Object.getOwnPropertyDescriptors`) including non-enumerable symbols, then overrides `pubkey`. This future-proofs the call against `hidden-tags.ts`'s planned non-enumerable symbol migration (Plan 07).
- `tags.ts:90`'s separate build-path return (`{ ...draft, content, [EncryptedContentSymbol]: plaintext }`) is untouched, as scoped — its migration is Plan 12's job.

## Decisions Made
- **`copyDraftWithPubkey` stays module-private (not exported):** exporting it would flow through `operations/tags.ts`'s `export *` into `operations/index.ts`'s public barrel, breaking `operations/__tests__/exports.test.ts`'s inline snapshot of the public API surface. The plan's action asked for a local fix, not a new public export. The regression test mirrors the helper's exact implementation in the test file itself (documented inline as intentionally not imported, with a pointer to the export-snapshot guard) rather than expanding the public API to make it importable.
- **Regression test targets `unlockHiddenTags` directly rather than routing through `modifyHiddenTags`'s full control flow:** empirical verification (via a throwaway probe test, later deleted) proved that `modifyHiddenTags`'s own `getHiddenTags(draft)` call — which happens *before* the temp object at the target line is ever built — already checks `HiddenTagsSymbol in event` and `EncryptedContentSymbol`-backed `getHiddenContent(event)` via the enumerability-blind `in` operator. This means if `draft` already carries either symbol (enumerable or not), `hidden` resolves before line 65/85 is ever reached, and the temp object is *only* built when `draft` genuinely has neither symbol cached — i.e., nothing to lose on that specific call. The fix itself is still correct and necessary as a defensive/future-proofing measure (protects `unlockHiddenTags` callers generally, and any future control-flow change), so it was implemented exactly as specified. The regression test proves the fix's actual mechanism (a descriptor-preserving copy avoids a redundant decrypt) by calling `unlockHiddenTags` — the function that receives the temp object — directly with objects built both the old (naive spread) and new (descriptor-preserving) ways, confirmed RED against the naive spread and GREEN against the fix.
- Added a third sanity-check test (`modifyHiddenTags still decrypts exactly once via the real unlock path`) to confirm the fix doesn't regress the actually-reachable single-decrypt path, since no existing test in the suite previously exercised `modifyHiddenTags` with real (non-empty) ciphertext content.
- A TypeScript inference limitation required constraining `copyDraftWithPubkey<T extends EventTemplate | UnsignedEvent | NostrEvent>` and adding a local cast at the call site (`as EventTemplate & { pubkey: string }`) — passing a generic-typed value through an intermediate generic helper loses the flattening TypeScript performs on a direct object-literal spread of a constrained generic parameter. The cast only affirms what the descriptor-preserving copy already guarantees at runtime.

## Issues Encountered

- **Reachability analysis of the plan's literal test scenario:** the plan's `<behavior>` for Task 2 describes constructing a draft with a pre-cached non-enumerable `HiddenTagsSymbol` and "re-entering the modifyHiddenTags branch that builds the temp object." Empirical verification (a throwaway probe test using `vi.spyOn` on the signer's decrypt method, run then deleted) showed this exact precondition is unreachable via `modifyHiddenTags`'s public API: `getHiddenTags(draft)` resolves `hidden` (via an enumerability-blind `HiddenTagsSymbol in event` / `EncryptedContentSymbol`-backed content check) *before* the temp-object line is ever reached, for both symbols independently. This was resolved by testing the fix's actual mechanism against `unlockHiddenTags` directly (the function that receives the temp object) rather than insisting on an unreachable end-to-end path through `modifyHiddenTags` — see Decisions Made above for the full trace. Documented here per the project's "spec-derived, not self-referential" test discipline (05.1-RESEARCH.md's non-vacuity requirement) so a future reader doesn't assume the literal plan scenario is directly testable through the public API.
