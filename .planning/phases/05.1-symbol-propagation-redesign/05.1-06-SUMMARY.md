## Decisions Made

- Wrote both `EncryptedContentSymbol` (preserved) and a throwaway test-only symbol (not preserved) onto the draft directly via `setCachedValue`, before entering `eventPipe`, rather than as a pipe operation's own return — a symbol attached inside a pipe step is subject to the delete loop's own pass immediately after that step (deleting any non-preserved symbol right then), which would attribute the drop to the delete loop rather than to the later spread step. Attaching both symbols on the pre-pipe draft means the very first pipe operation's spread is what drops them, isolating the assertion to the carry-forward loop specifically.
- Verified `sign()`'s own `Reflect.has/get/set` re-copy (in `operations/event.ts`) cannot mask a disabled carry-forward loop: `sign()` reads from its own `draft` parameter, which is the value the PRIOR pipe step produced — if that step's spread dropped the symbol and carry-forward didn't restore it, `sign()`'s own check (`Reflect.has(draft, EncryptedContentSymbol)`) is false and it does not copy anything. This was confirmed by the RED probe: the test failed even with `sign()` still in the pipe.
- Left `cache.ts`'s descriptor-flags explanation (D-02: `configurable`/`writable`/throw-on-frozen) in place, appended directly after the new one-rule block, since it documents the (unchanged) function implementation, not the superseded write-site taxonomy the plan targeted for removal.
