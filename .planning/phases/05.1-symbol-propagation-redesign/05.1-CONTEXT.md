# Phase 5.1: Symbol Propagation Redesign - Context

**Gathered:** 2026-07-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Symbol propagation gets **one teachable rule** instead of three overlapping mechanisms: every symbol write is non-enumerable (via `setCachedValue`), nothing survives an object copy implicitly, and the factory pipeline explicitly carries the `PRESERVE_EVENT_SYMBOLS` whitelist forward between operations — with `stamp`/`sign` keeping their explicit copies so standalone `sign(signer)(draft)` still preserves plaintext.

**In scope (from ROADMAP.md):**
1. Move `GiftWrapSymbol`/`SealSymbol`/`RumorSymbol` to core (re-exported from common), making `PRESERVE_EVENT_SYMBOLS` a static set with no import-time mutation (kills the `common/operations/gift-wrap.ts:24-26` `.add()` side-effect).
2. Add the whitelist carry-forward half to `pipeFromAsyncArray` and `EventFactory.chain` (copy preserved symbols `prev`→`result`, not just delete non-preserved).
3. Migrate the ~41 enumerable symbol write sites across core, common, wallet, wallet-connect, and concord onto `setCachedValue`, fixing the `filter.ts` shared-Set defect and the `groups.ts` undefined-memoization defect en route.
4. Delete the two strip loops (`pipeline.ts:63`, `factories/event.ts:80-84`) **last**, once no enumerable write remains.
5. Rewrite `cache.test.ts`'s carry-forward suite against the pipeline-carry mechanism and replace `cache.ts`'s superseded taxonomy prose.

**Folded-in behavioral bug fixes (roadmap decision 2026-07-16, Phase 5 code review):** 5 confirmed blocker defects (CR-01..05) in write-sites this redesign already touches — fixed here, **each in its own commit with its own spec-derived regression test, landed BEFORE the site's enumerable→non-enumerable migration** so a failing test attributes to the bug fix, not the refactor. Full detail: `.planning/phases/05-cache-identity-memo-fix/05-REVIEW.md`.

**Out of scope:** the rotation/channel/authority logic fixes (Phases 6–9); anything not sitting on a symbol write-site this migration touches. The 11 WARNING-severity review findings are milestone-review work except where a trivial+isolated fold applies (see D-11).

</domain>

<decisions>
## Implementation Decisions

### En-route defect discipline (filter.ts + groups.ts)

- **D-01: Split by asymmetry — the two "en route" defects are NOT symmetric.** `filter.ts`'s shared-Set defect is fixed *by* the migration itself (a non-enumerable Set is dropped by `{ ...event, tags: newTags }`, so the copy recomputes from its own tags — there is no bug independent of the migration). `groups.ts`'s undefined-memo defect is *not* fixed by the migration (the poisoning lives in `getOrComputeCachedValue` caching `undefined`, and `GroupsHiddenSymbol in bookmark` is a presence check that sees non-enumerable properties too — enumerability changes nothing). Treat them differently.
- **D-02: `groups.ts` joins the CR-01..05 discipline.** It gets its **own commit + own spec-derived regression test, landed BEFORE its write-site migration**. It is effectively a member of the CR-01 undefined-as-array family (`unlockHiddenGroups` returns poisoned `undefined` against its `Promise<GroupPointer[]>` signature, bypassing its own `if (!groups) throw`). Regression test: after a lock→unlock cycle (or an already-poisoned bookmark whose tags later unlock), `unlockHiddenGroups` returns real groups (or throws), never poisoned `undefined`.
- **D-03: `groups.ts` fix shape = "don't memoize undefined".** At the `getHiddenGroups` write site, only cache when the value is defined (skip the `setCachedValue`/compute-write when `undefined`). Narrowest fix, local to `groups.ts`; leaves `getOrComputeCachedValue`'s general behavior untouched for other callers. The presence check then only ever sees a real value.
- **D-04: `filter.ts` is validated BY its migration commit — no separate pre-fix commit.** Because the migration *is* the fix, a behavioral test (proving a copy built with different `tags` recomputes its own indexable set rather than inheriting the original's) lands with the migration commit, not before it. A pre-migration commit here would be artificial (there is no independent bug to isolate).

### cache.ts taxonomy fate (CACHE-02)

- **D-05: Replace the superseded three-category prose with a short one-rule doc block, kept in `cache.ts`.** The block states the single rule: every symbol write is non-enumerable via `setCachedValue`; nothing survives a copy implicitly; carry-forward is performed explicitly by the pipeline against `PRESERVE_EVENT_SYMBOLS`. Co-located with the helper it describes (not deleted to nothing, not relocated to `pipeline.ts`). This is what makes "one rule instead of three" a teachable artifact rather than tribal knowledge.
- **D-06: CACHE-02 recorded as "superseded + reinterpreted" in REQUIREMENTS.md / traceability.** Its original intent (document the memo-vs-carry-forward distinction so a future cleanup cannot collapse them) is *deliberately inverted* by this phase, which collapses them. Note that the memo-vs-carry-forward SAFETY goal is now met **structurally** (non-enumerable writes + explicit pipeline carry) rather than by a hand-maintained documented distinction. New evidence: the one-rule doc block (D-05) + the rewritten `cache.test.ts` carry-forward suite. Preserves traceability without pretending the original wording still holds.

### Changeset & release strategy

- **D-07: Mechanism change bumps `patch`.** The mass enumerable→non-enumerable migration + pipeline carry-forward + strip-loop deletion follows Phase 5's D-03 precedent: no documented behavior ever promised symbols survive a spread; the at-risk pattern (caching onto a mutable-then-spread object) has no in-repo callers outside concord. A defect-class fix, not a feature.
- **D-08: Gift-wrap-symbols-to-core move bumps `patch` on both core and common.** `Symbol.for()` means the symbol identity is unchanged regardless of module, so no consumer's behavior shifts; the move + removal of the import-time `PRESERVE_EVENT_SYMBOLS` mutation is an internal refactor. Consistent with the mechanism bump.
- **D-09: One-change-per-file, single-sentence changesets (CLAUDE.md).** The 5 behavioral fixes (CR-01..05) are each their own changeset file; `groups.ts` (D-02) gets its own; the mechanism and symbol-move each get theirs. Granularity is mandated by CLAUDE.md, not a judgment call — pick the smallest applicable bump per package per file.

### WARNING findings fold-in

- **D-10: Fold a WARNING only when its fix is trivial AND isolated in its own commit.** Never bundle a warning fix into a migration commit (preserves attribution). Anything non-trivial or entangled with the migration stays deferred to milestone review. Keeps 5.1 focused on the redesign + the 5+1 blockers without scope drift.
- **D-11: A folded warning still meets the milestone verification bar** — its own commit carries its own regression test. If it can't be tested trivially, it isn't "trivial" and is deferred.

### Claude's Discretion

- Phase-wide execution ordering within the roadmap's fixed constraints (carry-forward half must exist before any site migrates; strip loops deleted last; each behavioral fix before its site's migration) — the exact plan sequencing is left to plan-phase.
- What the rewritten `cache.test.ts` carry-forward suite asserts beyond exercising the real pipeline's explicit whitelist copy (per Phase 5 D-15's promoted probe, now retargeted onto the pipeline-carry mechanism).
- Whether `setEncryptedContentCache`'s read-path write also migrates to non-enumerable — implied yes by the redesign premise ("every symbol write is non-enumerable"; carry-forward now handled by the pipeline copy), but confirm during planning against the read-path re-entering the pipe.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`setCachedValue`** (`cache.ts`) — already writes non-enumerable with the load-bearing `{enumerable:false, writable:true, configurable:true}` descriptor (Phase 5). The migration target; no change to the helper's write mechanism is needed, only to its callers.
- **`PRESERVE_EVENT_SYMBOLS`** (`pipeline.ts:5`) — the existing carry-forward allowlist. Becomes static (gift-wrap symbols added at definition, not by import-time mutation).
- **`pipeFromAsyncArray`'s existing delete loop** (`pipeline.ts:57-66`) and **`EventFactory.chain`'s copy** (`factories/event.ts:80-84`) — the two spots that gain the carry-forward half and are later deleted.
- **`stamp`/`sign`** (`operations/event.ts`) — already do explicit `Reflect.has`/`get`/`set` re-copies (enumerability-blind), so standalone `sign(signer)(draft)` keeps working after the migration. Do NOT remove these.
- **The corrected unlock-guard pattern** at `bookmark.ts:82-86` / `groups.ts:146-148` — the template for the CR-01 family and the `groups.ts` read side.

### Established Patterns
- **The import-time-mutation anti-pattern is exactly what scope point 1 kills:** `common/operations/gift-wrap.ts:24-26` mutates a core-owned Set on import, so the whitelist's contents depend on whether common was loaded. Moving the symbols to core makes the Set statically complete.
- **The two strip loops currently only DELETE non-preserved symbols; neither COPIES preserved symbols forward.** Once writes are non-enumerable, an operation's own internal spread (e.g. `modifyPublicTags`'s `{ ...draft, tags }`) drops the symbol, so the pipe must actively copy `PRESERVE_EVENT_SYMBOLS` from `prev`→`result` — that is the "carry-forward half" scope point 2 adds.
- **`Symbol.for()` registry symbols** — Seal/Rumor/GiftWrap are `Symbol.for(...)`, so the module they live in does not change their identity; the move is safe and non-breaking (D-08).

### Integration Points
- `getExpirationTimestamp` → `getOrComputeCachedValue` runs on every `EventStore.add`/`AsyncEventStore.add` before kind branching (except `EventDeletion` early-return) — the throw-on-frozen path from Phase 5 is already live; the migration does not widen it.
- The read/unlock path (`setEncryptedContentCache`, gift-wrap unlock) re-enters the factory pipe via `modifyPublicTags` — the reason carry-forward must be pipeline-driven once writes are non-enumerable (see Discretion note on `setEncryptedContentCache`).

<specifics>
## Specific Ideas

**The filter.ts/groups.ts asymmetry is the load-bearing insight of this discussion** (drives D-01..D-04). Both are described in the roadmap as fixed "en route," but only `groups.ts` is a bug independent of the migration. Surfaced by reading both sites in full:
- `filter.ts:23-29` comment already admits the shared-Set gap is "only masked by `pipeFromAsyncArray`'s delete loop, on the one call path that runs it" — i.e. the non-enumerable migration removes the need for that mask and fixes the gap directly.
- `groups.ts`'s ~20-line comment documents the poisoned-undefined mechanism precisely; it is enumerability-independent, so it needs its own fix regardless of the migration.

**CACHE-02 is being deliberately inverted, not accidentally dropped** (D-06). Phase 5 wrote a careful three-category taxonomy *to preserve* a distinction; 5.1's whole thesis is that the distinction is a maintenance hazard and should be collapsed into one structural rule. The requirement record must reflect that this was a decision, not a regression — a future reader finding the taxonomy gone should find the supersession note, not conclude the safety property was lost.

</specifics>
