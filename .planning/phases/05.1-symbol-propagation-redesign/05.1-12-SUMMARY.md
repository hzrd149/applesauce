# Phase 05.1 Plan 12: Build-path write migrations (tags.ts, encrypted-content.ts, gift-wrap.ts) Summary

**Migrated the two core object-literal build-path writes (`operations/tags.ts:90`, `operations/encrypted-content.ts:29`) to construct-then-`setCachedValue`, and all five `common/operations/gift-wrap.ts` writes (four accumulated-state + one build-path) to `setCachedValue`, each with a non-enumerability assertion and a full-pipe survival test proving the carry-forward/preserve mechanism actually delivers the plaintext to the final event.**

## Accomplishments

- `operations/tags.ts`'s `modifyHiddenTags` build-path return was `{ ...draft, content, [EncryptedContentSymbol]: plaintext }` (an enumerable object-literal computed-key write). Replaced with `const result = { ...draft, content }; setCachedValue(result, EncryptedContentSymbol, plaintext); return result;` (PATTERNS Group B). Added a test asserting the write is non-enumerable, and a full-pipe survival test: `eventPipe(modifyHiddenTags(...), includeAltTag(...), sign(user))` run against a fresh template, asserting the signed event's `getEncryptedContent`/`getHiddenTags` read back the plaintext — proving `pipeFromAsyncArray`'s same-kind carry-forward loop restored the symbol across `includeAltTag`'s intervening `modifyPublicTags` spread.
- `operations/encrypted-content.ts`'s `setEncryptedContent` had the identical shape at `:29`; applied the same Group B transformation and the same full-pipe survival test pattern (kind 4 / EncryptedDirectMessage, nip04 override), asserting `getEncryptedContent(signed)` matches the plaintext independently supplied to the operation.
- `common/operations/gift-wrap.ts`'s five `Reflect.set` writes were migrated to `setCachedValue`: `sealRumor`'s `RumorSymbol` (downstream ref on the seal) and `SealSymbol` Set-initialization (upstream ref on the rumor), and `wrapSeal`'s `GiftWrapSymbol` (upstream ref on the seal), `SealSymbol` (downstream ref on the gift wrap), and the build-path `EncryptedContentSymbol` write. Each site got a dedicated non-enumerability test (`Object.getOwnPropertyDescriptor(...).enumerable === false` plus a spread-drops-it control assertion for the two reference-link sites). The build-path `EncryptedContentSymbol` site got both a `wrapSeal`-scoped test and a `giftWrap()`-scoped full-pipe survival test, each asserting non-enumerability and a value match against an independently recipient-side-decrypted plaintext (not the operation's own internal `plaintext` variable read back).
- Confirmed via `packages/core/src/helpers/pipeline.ts`'s current source (the `sameEvent` guard added in Plan 01, refined per this plan's `critical_mechanism_note`) that the gift-wrap build-path site does **not** rely on cross-kind carry-forward — `wrapSeal` produces a new kind-1059 event from a kind-13 input, so the `sameEvent` check is false between those steps, and carry-forward never fires there. What actually keeps `EncryptedContentSymbol` alive on the final `gift` event is `PRESERVE_EVENT_SYMBOLS` membership surviving the delete loop that runs immediately after `wrapSeal` returns its own value — the symbol was never "carried forward from" a prior step in the first place. The full-pipe survival test was designed around this fact rather than asserting the (unsupported) cross-kind carry-forward behavior.
- Rewrote the `:141` comment: removed the WR-06 "ONLY carry-forward site in applesauce-common" framing (no longer accurate or checkable post-migration) and replaced it with a narrower, verifiable statement — the site survives because it's written directly onto the operation's own returned event and `PRESERVE_EVENT_SYMBOLS` protects it through that step's own delete-loop pass.
- Confirmed the import-time `PRESERVE_EVENT_SYMBOLS.add()` calls (Plan 01's deletion target) remain absent from `common/operations/gift-wrap.ts` — only the explanatory comment noting the static-Set membership remains at the top of the file.

## Decisions Made

- The gift-wrap build-path site's full-pipe survival test derives its expected plaintext by decrypting the final event's own ciphertext content with the recipient's `nip44` implementation (`other.nip44.decrypt(gift.pubkey, gift.content)`), rather than reading back the operation's own `plaintext` local variable — this keeps the assertion independent of the implementation under test, matching the plan's "derived from the fixture, not from operation output" convention used for the core-package sites.
- Had to run `pnpm --filter applesauce-core build` before `applesauce-common`'s test suite could resolve `applesauce-core/helpers/cache` (and other subpath exports) — the package's `exports` map in `package.json` points at `dist/`, and while `turbo.json`'s `test` task depends on `build`, invoking `vitest` directly via `pnpm --filter <pkg> exec vitest run` (the plan's own verify command) bypasses that dependency graph. `dist/` is gitignored, so this build step left no artifact in any commit — documented here so a future re-run of this plan's verify command in a fresh clone knows to build `applesauce-core` first.
