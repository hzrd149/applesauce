# Phase 05.1 Plan 09: Common identity-memo/flag write migration Summary

Migrated all eight common-package identity-memo/flag writes named in the plan onto `setCachedValue`, replacing hand-rolled enumerable `Reflect.set` writes, and lifted the D-12 hot-path deferral on `lists.ts`'s `getOrComputeListCache` by making its `setCachedValue` call fire only once per cache-object creation instead of every getter call.

## What Was Built

**Task 1 — CR-02-family memo writes (mute, emoji-pack ×2, trusted-assertions):**
- `mute.ts:93` (`getHiddenMutedThings`'s `MuteHiddenSymbol` write)
- `emoji-pack.ts:109` (`getHiddenFavoriteEmojis`'s `FavoriteEmojiPacksHiddenSymbol` write) and `:132` (`getHiddenFavoriteEmojiPackPointers`'s `FavoriteEmojiPacksHiddenPointersSymbol` write)
- `trusted-assertions.ts:95` (`getHiddenProviders`'s `TrustedProvidersHiddenSymbol` write)

All four migrated from `Reflect.set` to `setCachedValue` (imported from `applesauce-core/helpers/cache`); the stale write-site comments describing the "deliberately-deferred gap" were replaced with comments noting the 05.1-09 migration. The Plan 04 CR-02 guard fixes in these same files were left untouched.

**Task 2 — app-data, bookmark, lists (D-12 hot-path lift):**
- `app-data.ts:71` (`getAppDataContent`'s `AppDataContentSymbol` write) — migrated to `setCachedValue`; the Plan 05 CR-03 `lockAppData` fix (`Reflect.deleteProperty`) needed no changes since it already operates on the symbol regardless of enumerability, and its regression test still passes.
- `bookmark.ts:107` (`getHiddenBookmarks`'s `BookmarkHiddenSymbol` write) — migrated to `setCachedValue`.
- `lists.ts:53` (`getOrComputeListCache`'s generic two-level cache write) — migrated to `setCachedValue`, but restructured: the function previously called `Reflect.set` unconditionally on every invocation (even when the cache object already existed), which is why Phase 5 deferred it as a hot-path concern (D-08). The migration only calls `setCachedValue` once, the first time the cache container object is created; subsequent calls read the existing non-enumerable memo via `Reflect.get` and mutate the returned container in place (`cache[cacheType] = value`), so no repeated `Object.defineProperty` cost is added on the hot path. A code comment at the write site documents the D-12 hot-path lift and the once-only invariant.

**Task 3 — encrypted-content-cache flag write:**
- `encrypted-content-cache.ts:49` (`markEncryptedContentFromCache`'s `EncryptedContentFromCacheSymbol` write) — migrated to `setCachedValue`. `isEncryptedContentFromCache`'s `Reflect.has`-based reader was left unchanged (enumerability-blind, so it still sees the flag); WR-11's separate `=== true` tightening remains deferred per D-10, confirmed out of scope for this plan.

## Tests Added

Each migrated write got a non-enumerability test asserting:
1. The symbol is absent from `Object.keys(event)` (non-enumerable, hidden from `Object.keys`/`JSON.stringify`).
2. The symbol is present via `Object.getOwnPropertySymbols(event)`.
3. `Object.getOwnPropertyDescriptor(event, symbol)?.enumerable === false`.
4. A plain object spread (`{ ...event }`) drops the symbol (`Reflect.has(spread, symbol) === false`).

Additional targeted regression coverage:
- `app-data.test.ts`: a new test confirms `lockAppData` still clears the now-non-enumerable `AppDataContentSymbol` memo (Plan 05 CR-03 regression, re-verified under the new write mechanism).
- `lists.test.ts`: confirms the migrated cache still functions as a working memo (repeat calls to `getEventPointersFromList` return the same cached reference).
- `encrypted-content-cache.test.ts`: confirms `isEncryptedContentFromCache` still returns `true` after `markEncryptedContentFromCache` runs, proving the `Reflect.has`-based reader is unaffected by the enumerability change.

All Plan 04 CR-02 guard regression tests (`isHiddenMutesUnlocked`, `isHiddenFavoriteEmojiPacksUnlocked`, `isHiddenProvidersUnlocked`) and the Plan 05 CR-03 `lockAppData` regression test remain green — confirmed by running the full `applesauce-common` suite after each task.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - correctness] Restructured `getOrComputeListCache`'s write to call `setCachedValue` once instead of per-call**

- **Found during:** Task 2
- **Issue:** The plan's `<action>` says to migrate `lists.ts:53` to `setCachedValue`, but a literal one-line swap (calling `setCachedValue` unconditionally on every `getOrComputeListCache` invocation, mirroring the original unconditional `Reflect.set`) would call `Object.defineProperty` on every single getter call on this hot path — reintroducing the exact performance concern (D-08/D-12) that caused Phase 5 to defer this site in the first place, just with a more expensive write mechanism than the `Reflect.set` it replaces.
- **Fix:** Restructured the function to only call `setCachedValue` the first time the cache container object is created (`if (!cache) { cache = {}; setCachedValue(list, symbol, cache); }`), then read the existing memo via `Reflect.get` and mutate the returned container in place on every other call — mirroring the once-only write pattern `getOrComputeCachedValue` itself already uses elsewhere in the core cache module.
- **Files modified:** `packages/common/src/helpers/lists.ts`
- **Commit:** `be8d1c8a`
