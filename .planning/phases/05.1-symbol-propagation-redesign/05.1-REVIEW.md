# Phase 05.1: Code Review Report

**Reviewed:** 2026-07-16
**Depth:** standard
**Files Reviewed:** 35
**Status:** issues_found

## Summary

The non-enumerable migration itself is largely sound. Every `setCachedValue` site I traced
writes non-enumerable and matches the "one rule" doc block in `cache.ts`. The five CR fixes are
substantively correct:

- **CR-01** (`contacts.ts` `isHiddenContactsUnlocked`) — correct; the `Symbol in event || getHiddenContacts(...) !== undefined` guard no longer lies to a caller.
- **CR-02** (mute / emoji-pack / trusted-assertions guards) — correct and consistent with the bookmark/groups pattern.
- **CR-03** (`app-data.ts` two-delete `lockAppData`) — correct; parsed memo dropped before `lockHiddenContent`.
- **CR-04** (`event-store.ts` `copySymbolsToDuplicateEvent` `&&`→`||`) — the guard fix is correct (it now throws when EITHER pubkey OR identifier differs). See WR-01 for a residual gap the fix does not close.
- **CR-05** (`operations/event.ts` `stamp` copy-then-delete) — correct; `stamp` no longer mutates the caller's draft.

The carry-forward kind-guard (`result.kind === input.kind`) correctly skips the gift-wrap
rumor→seal→wrap boundaries (each step changes kind, so accumulated per-envelope symbols are not
carried across). However the guard interacts badly with the carry-forward's use of `Reflect.set`
(WR-02), and several latent correctness/security issues remain in the reviewed files (below).

## Critical Issues

### CR-01: Gift-wrap seal signature verification result is discarded

**File:** `packages/common/src/helpers/gift-wrap.ts:209`
**Issue:** In `getGiftWrapSeal`, `verifyWrappedEvent(seal)` is called but its boolean return is
ignored, so a seal that fails signature verification is still added to `internalGiftWrapEvents`
and cached via `setCachedValue(seal, GiftWrapSymbol, gift)` / `setCachedValue(gift, SealSymbol, seal)`.
`getSealRumor` then binds the rumor to `seal.pubkey` (`rumor.pubkey !== seal.pubkey` check), but
`seal.pubkey` was never proven by a valid signature. An attacker can craft a gift wrap whose inner
seal claims any `pubkey` (with a matching rumor `pubkey`) and it will be accepted as an authentic
message from that pubkey — an author-spoofing bypass. The Concord counterpart does this correctly
(`concord/src/helpers/gift-wrap.ts:46`: `if (!verifyWrappedEvent(seal)) return null;`).
Note: this line is pre-existing (unchanged by this phase) but sits in a reviewed file and the phase
explicitly asked for verified-flag/plaintext leak review.
**Fix:**
```ts
// Verify the seal event, reject if invalid
if (!verifyWrappedEvent(seal)) return undefined;
// Add to the internal event set
internalGiftWrapEvents.add(seal);
```

## Warnings

### WR-01: `copySymbolsToDuplicateEvent` copies content-derived symbols across replaceable versions

**File:** `packages/core/src/event-store/event-store.ts:197-228` (and the async mirror `async-event-store.ts:237,252,272`)
**Issue:** For replaceable events the sameness guard only checks `kind`, `pubkey`, and
`getReplaceableIdentifier` — NOT `id`, `content`, or `created_at`. Two *different versions* of a
replaceable event share an address but have different content/sig. The merge loop then copies
`verifiedSymbol` and `EncryptedContentSymbol` from `source` onto `dest` whenever `dest` lacks them.
At the `add` call site `copySymbolsToDuplicateEvent(event, winner)` (line 269), `event` can be an
older losing version; if it already carries a decrypted `EncryptedContentSymbol` (pre-decrypted
before re-add) or a `verifiedSymbol`, that content-specific symbol is grafted onto a
different-content `winner`. Result: `getEncryptedContent(winner)` returns the *wrong* plaintext, or
a version gets a `verifiedSymbol` it never earned. `verifiedSymbol` and `EncryptedContentSymbol` are
tied to exact event bytes and must not propagate between versions.
**Fix:** Gate the `verifiedSymbol` / `EncryptedContentSymbol` copy on `source.id === dest.id` (only
`SeenRelaysSymbol` / `FromCacheSymbol` are safe to merge across versions):
```ts
const idMatched = source.id === dest.id;
const symbols = idMatched
  ? [FromCacheSymbol, verifiedSymbol, EncryptedContentSymbol]
  : [FromCacheSymbol];
```

### WR-02: Carry-forward (and `stamp`/`sign`) use `Reflect.set`, re-enumerating preserved symbols and defeating the same-kind guard

**File:** `packages/core/src/helpers/pipeline.ts:88`, `packages/core/src/factories/event.ts:87`, `packages/core/src/operations/event.ts:143,180`
**Issue:** The whole phase rests on the "one rule": every symbol write is non-enumerable so a spread
drops it and carry-forward is the *only* propagation path. But the carry-forward loop itself writes
with `Reflect.set(result, symbol, ...)`, which creates a **default enumerable** data property. So
after a same-kind pipe/chain step, `EncryptedContentSymbol` (and any other `PRESERVE_EVENT_SYMBOLS`
member) becomes enumerable on `result`. The next step's `{ ...draft }` spread then copies it
implicitly — and if that next step *changes kind*, the enumerable plaintext rides across the kind
boundary, exactly the leak the `sameEvent` guard exists to prevent. `stamp`/`sign` have the same
defect (they `Reflect.set(newDraft/signed, EncryptedContentSymbol, ...)`), leaving decrypted
plaintext as an enumerable own-property on the signed event. Use `setCachedValue` for these writes
to keep them non-enumerable and preserve the invariant.
**Fix:**
```ts
// pipeline.ts / factories/event.ts carry-forward:
if (Reflect.has(prevValue, symbol) && !Reflect.has(result, symbol))
  setCachedValue(result, symbol, Reflect.get(prevValue, symbol));
// operations/event.ts stamp/sign:
setCachedValue(newDraft, EncryptedContentSymbol, Reflect.get(draft, EncryptedContentSymbol)!);
```

### WR-03: `getSealRumor` undefined-sentinel makes `isSealUnlocked` report a parse-failed seal as unlocked

**File:** `packages/common/src/helpers/gift-wrap.ts:160`, `240`
**Issue:** When rumor JSON fails to parse, `getSealRumor` writes `setCachedValue(seal, RumorSymbol, undefined)`.
`isSealUnlocked` only checks presence (`RumorSymbol in seal`), which is now `true`, so the seal is
reported unlocked. `unlockSeal` then short-circuits `if (isSealUnlocked(seal)) return seal[RumorSymbol]`
and returns `undefined` typed as `Rumor`, bypassing its own `if (!rumor) throw` guard. Callers get an
`undefined` rumor with no error. The code documents this as a KNOWN GAP deferred out of scope, but it
is a live correctness defect (type lies to callers).
**Fix:** Have `isSealUnlocked` check the value, not mere presence, or use a distinguishable sentinel:
```ts
export function isSealUnlocked(seal: NostrEvent): seal is UnlockedSeal {
  return (RumorSymbol in seal && seal[RumorSymbol] !== undefined) ||
    (isEncryptedContentUnlocked(seal) === true && getSealRumor(seal) !== undefined);
}
```

### WR-04: `EventFactory.kind()` never resolves its promise

**File:** `packages/core/src/factories/event.ts:151-153`
**Issue:** `return new EventFactory((e) => ({ ...e, kind }))` — the Promise executor's first parameter
`e` is the `resolve` function, not the draft. The arrow spreads the resolve function (`{ ...e, kind }`)
and returns it (discarded by the Promise constructor); `resolve` is never called, so the factory
promise never settles and any `await` on it hangs. Compare `strip()` (line 111) which correctly does
`res(this.then(...))`. Currently unused (no chained `.kind(` call sites found), so latent, but it is a
real deadlock waiting for the first caller.
**Fix:**
```ts
kind<Kind extends number>(kind: Kind): EventFactory<Kind, KnownEventTemplate<Kind>> {
  return new EventFactory((res) => res(this.then((e) => ({ ...e, kind })) as Promise<KnownEventTemplate<Kind>>));
}
```

## Info

### IN-01: `stamp()` comment claims `modifyHiddenTags` writes `EncryptedContentSymbol` enumerably — it no longer does

**File:** `packages/core/src/operations/event.ts:140-142`
**Issue:** The comment states "Today's `{ ...draft, pubkey }` spread already copies it too, since
modifyHiddenTags writes it enumerably — so this copy is belt-and-braces on that path." As of this
phase, `operations/tags.ts` `modifyHiddenTags` writes `EncryptedContentSymbol` **non-enumerably** via
`setCachedValue` (line 122), so the spread does NOT copy it and the explicit `Reflect` copy is now
load-bearing, not belt-and-braces. Misleading comment.
**Fix:** Update the comment to reflect that the spread drops the (now non-enumerable) symbol and the explicit copy is required.

### IN-02: `lockWallet` does not clear `WalletRelaysSymbol`

**File:** `packages/wallet/src/helpers/wallet.ts:142-146`
**Issue:** `lockWallet` deletes `WalletPrivateKeySymbol` and `WalletMintsSymbol` and calls
`lockHiddenTags`, but leaves `WalletRelaysSymbol` cached. After "locking", `getWalletRelays` still
returns the cached relays (its `WalletRelaysSymbol in wallet` early-return fires). Non-sensitive data,
so low impact, but the lock is incomplete/inconsistent with the mint+key clears.
**Fix:** `Reflect.deleteProperty(wallet, WalletRelaysSymbol);` in `lockWallet`.

### IN-03: `getGiftWrapSeal` uses throwing `JSON.parse` while `getSealRumor` uses `safeParse`

**File:** `packages/common/src/helpers/gift-wrap.ts:200`
**Issue:** `const seal = JSON.parse(content)` will throw on malformed decrypted content, whereas the
sibling `getSealRumor` uses `safeParse` and returns `undefined`. Inconsistent error handling; a
corrupt/attacker-supplied gift wrap surfaces as an unhandled throw from a getter.
**Fix:** Use `safeParse<NostrEvent>(content)` and `return undefined` on failure.

### IN-04: `getAppDataContent` truthy checks mishandle falsy parsed values

**File:** `packages/common/src/helpers/app-data.ts:53-64`
**Issue:** `if (cached) return cached;` re-parses whenever the cached app-data value is falsy
(`0`, `""`, `false`), and `if (!data)` similarly treats a legitimately-falsy parsed JSON value as
"not parsed" and falls through to the hidden-content branch. Only matters for non-object app data,
but it is a latent correctness edge.
**Fix:** Use presence checks (`if (AppDataContentSymbol in event) return Reflect.get(...)`) and
`data === undefined` comparisons instead of truthiness.
