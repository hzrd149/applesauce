# Phase 05.1: Symbol Propagation Redesign - Research

**Researched:** 2026-07-16
**Domain:** Internal SDK plumbing — symbol-keyed property descriptors, factory-pipe carry-forward semantics, TypeScript monorepo (applesauce). No external libraries; this is pure `Object.defineProperty`/`Reflect`/pipe-composition work over existing code.
**Confidence:** HIGH — every claim below is grounded in direct source reads of this session (file:line cited throughout), not training-data assumptions. The only `[ASSUMED]`-tagged items are two open design calls the roadmap explicitly left to planning.

## Summary

This phase has no external dependency risk — it is a pure internal refactor of `applesauce-core`'s cache/pipeline mechanism and every downstream site that writes a symbol-keyed property. The mechanism itself (`setCachedValue`'s `Object.defineProperty(..., {enumerable:false, writable:true, configurable:true})`) already exists and is correct (Phase 5, verified again by the Phase 5 code review). What 05.1 adds is: (1) a **carry-forward half** in the two existing strip loops (`pipeline.ts`'s `pipeFromAsyncArray` and `factories/event.ts`'s `EventFactory.chain`) that copies `PRESERVE_EVENT_SYMBOLS` from `prev`→`result` after each pipe step, and (2) migrating every remaining enumerable symbol write onto `setCachedValue` now that the pipe — not write-site enumerability — is what guarantees survival.

I enumerated every symbol write site in the monorepo by grep (`Reflect.set`, `Object.defineProperty`, bracket assignment, and object-literal symbol keys) and read all of them in full. The actual count is **43 write sites that must migrate**, not the roadmap's approximate "~41" — the discrepancy is explained below (2 extra candidates the roadmap's ~41 estimate didn't count, because they predate this phase's grep and weren't `Reflect.set`-shaped). Two sites (`stamp`/`sign`'s explicit re-copies) **must not** migrate — they are the mechanism that keeps standalone `sign(signer)(draft)` working outside any pipe. One site (`groups.ts:139`) is not a migration target at all — it is already non-enumerable at its final descriptor and should be **deleted**, not converted, once the CR-adjacent `groups.ts` defect fix lands. `cast.ts:63`'s `CASTS_SYMBOL` write was **explicitly deferred as out of scope in Phase 5** (D-08) for reasons partially orthogonal to enumerability (aliased mutable `Map`); planning must make an explicit call on it rather than silently migrating or silently skipping it.

**Primary recommendation:** Treat this as a large, mechanical, low-risk migration wave (batch by package) preceded by five independent one-commit-one-test behavioral fixes (CR-01..05) and the `groups.ts` fix, with the two strip-loop carry-forward additions landing before any site migrates and their deletion landing only after the last site migrates. The two "out-of-pipe spread" sites CONTEXT.md flags need small, targeted code changes (not just an audit) — see Common Pitfalls.

## The Enabling Mechanism (Already Correct — Do Not Change)

`packages/core/src/helpers/cache.ts:120-133`:
```ts
export function setCachedValue<T extends unknown>(event: any, symbol: symbol, value: T) {
  Object.defineProperty(event, symbol, { value, enumerable: false, writable: true, configurable: true });
}

export function getOrComputeCachedValue<T extends unknown>(event: any, symbol: symbol, compute: () => T): T {
  if (Reflect.has(event, symbol)) return Reflect.get(event, symbol);
  const value = compute();
  Object.defineProperty(event, symbol, { value, enumerable: false, writable: true, configurable: true });
  return value;
}
```
This is unchanged by 05.1 — every migration target simply becomes a caller of one of these two functions instead of a raw `Reflect.set`/object-literal/bracket-assignment. `cache.ts`'s **taxonomy prose** (lines 1-115, the three-category doc comment) is what gets replaced (scope point 5 / D-05), not the functions.

## Architecture Patterns

### The two strip loops — exact current shape

**`packages/core/src/helpers/pipeline.ts:51-70`** (`pipeFromAsyncArray`):
```ts
export function pipeFromAsyncArray<T, R>(fns: Array<Operation<T, R>>, preserve?: Set<symbol>): Operation<T, R> {
  if (fns.length === 0) return identity as Operation<any, any>;
  return async function piped(input: T): Promise<R> {
    return fns.reduce(async (prev: any, fn: Operation<T, R>) => {
      const result = await fn(await prev);
      if (preserve && typeof result === "object" && result !== null && typeof prev === "object" && prev !== null) {
        const keys = Reflect.ownKeys(result).filter((key) => typeof key === "symbol");
        for (const symbol of keys) {
          if (!preserve.has(symbol)) Reflect.deleteProperty(result, symbol);
        }
        // <-- CARRY-FORWARD HALF GOES HERE (scope point 2)
      }
      return result;
    }, input as any);
  };
}
```
`prev` here is the **awaited previous step's output** (`await prev`, resolved inside the reduce closure before `fn` runs) — it is exactly the right reference to copy *from*. The carry-forward addition:
```ts
// Copy every preserved symbol prev had that result is missing (an operation's own
// internal spread — e.g. modifyPublicTags's `{ ...draft, tags }` — drops non-enumerable
// writes; this restores them explicitly instead of relying on write-site enumerability).
for (const symbol of preserve) {
  if (Reflect.has(prev, symbol) && !Reflect.has(result, symbol)) {
    Reflect.set(result, symbol, Reflect.get(prev, symbol));
  }
}
```
Note `prev` at this point is the *awaited* value used above (the reduce callback already did `await prev` to get the input for `fn`; that resolved value, not the `Promise`, is what must be referenced — the existing code already has this resolved value in scope as the result of `await prev` used to call `fn(await prev)`, so the addition needs to capture that into a local rather than re-await `prev` a second time, since re-awaiting a settled promise is safe but the *value* must be the one `fn` actually received, not a fresh call to `prev`. Recommend hoisting `const prevValue = await prev;` once and reusing it for both `fn(prevValue)` and the carry-forward loop.

**`packages/core/src/factories/event.ts:69-95`** (`EventFactory.chain`):
```ts
protected chain(operation: EventOperation<T>): this {
  const Constructor = this.constructor as typeof EventFactory;
  const next = new Constructor((res) =>
    res(
      this.then(async (draft) => {
        const result = await operation(draft);
        for (const key of Reflect.ownKeys(result)) {
          if (typeof key === "symbol" && !PRESERVE_EVENT_SYMBOLS.has(key)) {
            Reflect.deleteProperty(result, key);
          }
        }
        // <-- CARRY-FORWARD HALF GOES HERE (scope point 2)
        return result;
      }),
    ),
  );
  ...
}
```
Same shape: `draft` is `prev`, `result` is `fn`'s output. Add:
```ts
for (const symbol of PRESERVE_EVENT_SYMBOLS) {
  if (Reflect.has(draft, symbol) && !Reflect.has(result, symbol)) {
    Reflect.set(result, symbol, Reflect.get(draft, symbol));
  }
}
```

**Deletion order (scope point 4):** both delete-loops stay until every migration site lands (they are what currently masks `filter.ts`'s and other identity-memo sites' spread-survival gaps — see Common Pitfalls). Only after zero enumerable symbol writes remain should the two loops (not the carry-forward halves — the carry-forward halves are permanent) be deleted... actually re-read scope point 4: *"Delete the two strip loops last, once no enumerable write remains."* This means the **delete-loop half** (the `Reflect.deleteProperty` for non-preserved symbols) is removed entirely once nothing enumerable is left to strip — but the **carry-forward half** must stay permanently, since it is the mechanism scope point 2 adds and the phase's Goal statement explicitly keeps ("the factory pipeline explicitly carries the `PRESERVE_EVENT_SYMBOLS` whitelist forward between operations"). Confirm this reading during planning: it is `[ASSUMED]` based on the Goal statement's wording, not stated as a literal instruction anywhere in scope point 4 itself. The safer literal reading of "delete the two strip loops" removes the whole per-step preserve/delete block; but that would remove carry-forward too, contradicting the Goal. Recommend the plan state explicitly: *delete-loop half removed, carry-forward half kept, once step 3's migration is complete.*

### Out-of-pipe spread audit (CONTEXT.md explicit ask)

**Site 1 — `unlockHiddenTags`'s `{ ...draft, pubkey }`** (`packages/core/src/operations/tags.ts:65`):
```ts
if (hidden === undefined) {
  if (hasHiddenTags(draft)) {
    pubkey = await signer.getPublicKey();
    hidden = await unlockHiddenTags({ ...draft, pubkey }, signer);   // <-- spread here
  }
  else hidden = [];
}
```
This spread builds a **throwaway temp object** passed into `unlockHiddenTags`, not the pipe operation's return value — it is genuinely out-of-pipe (the outer pipe's carry-forward mechanism never sees this intermediate object). `unlockHiddenTags` (`helpers/hidden-tags.ts:130`) checks `isHiddenTagsUnlocked(event)` → `HiddenTagsSymbol in event`. **Today**, `HiddenTagsSymbol` is written enumerably (`hidden-tags.ts:109,157`), so the spread copies it and the short-circuit works. **Once migrated to non-enumerable**, this spread silently drops `HiddenTagsSymbol` (and any other already-cached symbol on `draft`), so `unlockHiddenTags` always takes the "not yet unlocked" branch on this temp object — forcing a redundant `unlockHiddenContent` round-trip (extra signer call) even when hidden tags were already unlocked on `draft`. This is a **functional regression risk**, not just cosmetic: this code path only runs when hidden tags are NOT already parseable from cache (`hidden === undefined` at line 61) but the event DOES have hidden tags (`hasHiddenTags(draft)` true) — so in practice `HiddenTagsSymbol` usually isn't set yet either. But if a caller passes a `draft` that already has `EncryptedContentSymbol` (carry-forward payload, not `HiddenTagsSymbol`) — e.g. re-entering `modifyHiddenTags` on an event that was already unlocked earlier in the same pipe — the spread drops it too, and `unlockHiddenTags` → `unlockHiddenContent` re-decrypts unnecessarily. **Fix:** replace `{ ...draft, pubkey }` with an approach that doesn't spread — e.g. `Object.assign(Object.create(Object.getPrototypeOf(draft)), draft, { pubkey })` is not idiomatic here; simplest fix is a helper that copies own enumerable AND symbol properties (`Reflect.ownKeys` instead of spread), or just mutate a shallow clone via `Object.defineProperties(structuredCloneShallow, Object.getOwnPropertyDescriptors(draft))` plus `pubkey` override. Recommend: build the temp object with `Object.create(null)` + `Object.defineProperties(tmp, { ...Object.getOwnPropertyDescriptors(draft), pubkey: {value: pubkey, enumerable: true} })` — preserves descriptors (including non-enumerable symbols) exactly. Simpler alternative: don't spread at all — call `unlockHiddenTags(draft, signer)` first (unlocking mutates state via `setHiddenTagsCache`, not the passed object's identity) then separately track `pubkey`. Planning should pick one; this is a real code change, not just documentation.

**Site 2 — gift-wrap's `{ ...draft }` rumor copy** (`packages/common/src/operations/gift-wrap.ts:42`, inside `toRumor`):
```ts
export function toRumor(signer?): EventOperation<..., Rumor> {
  return async (draft) => {
    const rumor: Rumor = { ...draft };   // <-- spread here
    ...
    return rumor;
  };
}
```
`toRumor` **is** a pipe operation (first step of `giftWrap`'s `eventPipe(toRumor(signer), sealRumor(...), wrapSeal(...))`, `operations/gift-wrap.ts:158-163`). Its return value (`rumor`) **is** what the outer pipe's carry-forward mechanism inspects as `result`. So once the carry-forward half (scope point 2) is added to `pipeFromAsyncArray`, any preserved symbol present on the pipe's input (`prev`, i.e. the original `draft` passed into `giftWrap(...)`) that this spread drops **will be restored automatically** by the pipe after this step runs — this is a case the new mechanism handles correctly by design, unlike Site 1. **No source change needed at this site** beyond the general carry-forward addition; CONTEXT.md's audit request is satisfied by confirming this (it is functionally different from Site 1 despite the superficially identical `{ ...draft }` shape — the distinguishing factor is whether the spread's result is itself the pipe operation's return value). Document this conclusion in the plan/commit so a future reader doesn't re-litigate it.

### Confirmed: `setEncryptedContentCache`'s read-path write should migrate (CONTEXT.md Discretion question, resolved)

`packages/core/src/helpers/encrypted-content.ts:116-129` (`setEncryptedContentCache`) writes `EncryptedContentSymbol` **enumerably today specifically because** an unlocked event re-entering the factory pipe hits `modifyPublicTags`'s `{ ...draft, tags }` (`operations/tags.ts:29`), and a non-enumerable write would be dropped there (comment at `encrypted-content.ts:117-124` states this explicitly, and the Phase 5 code review verified the claim: *"`setEncryptedContentCache`'s enumerable write is required because `modifyPublicTags`'s `{ ...draft, tags }` would drop a non-enumerable one. Claim holds."*, `05-REVIEW.md` "Verified" section). **Once the carry-forward half is added to `pipeFromAsyncArray`**, this reasoning is obsolete: `modifyPublicTags`'s spread dropping the symbol is exactly the case the pipe-level carry-forward now compensates for on every subsequent step, the same way it compensates for `modifyHiddenTags`'s own write. `EncryptedContentSymbol` is already a permanent member of `PRESERVE_EVENT_SYMBOLS` (`pipeline.ts:5`), so no allowlist change is needed — only the write-site conversion. **Recommendation: migrate `encrypted-content.ts:125` to `setCachedValue` as part of this phase's core-package wave**, and update `setEncryptedContentCache`'s comment (currently asserts the opposite) accordingly. Confidence: HIGH — grounded in direct trace of `modifyPublicTags` plus the Phase 5 reviewer's independent verification of the same code path.

## Common Pitfalls

### Pitfall 1: Migrating a hot-path site without confirming Phase 5's D-08 deferral is actually being reversed

**What goes wrong:** `filter.ts:29` (`getIndexableTags`), `event.ts:133` (`getEventUID`), and `lists.ts:53` (`getOrComputeListCache`) were **explicitly and deliberately** left enumerable in Phase 5, with a named rationale: *"they are hot paths, none sits on a spread object today, and widening the blast radius of the phase everything else depends on is the wrong trade"* (05-CONTEXT.md D-08). `getIndexableTags` runs on every `matchFilter` call (i.e., every subscription match against every incoming event) — this is one of the hottest paths in the whole SDK. `getOrComputeCachedValue`'s `Object.defineProperty` write is measurably slower than a plain `Reflect.set` in V8 (property shape transitions). 05.1's roadmap scope broadly says "migrate the ~41 sites," and CONTEXT.md's D-01/D-04 specifically re-litigates `filter.ts` (concluding the migration IS the fix, so it must move) — but `event.ts:133` and `lists.ts:53` are not named anywhere in 05.1's CONTEXT.md discussion. This phase is silently reversing a documented Phase 5 performance decision for at least 2 of the 43 sites without having re-litigated the performance tradeoff.
**Why it happens:** the roadmap's "migrate everything" framing doesn't distinguish "sites with a known defect the migration fixes" from "sites Phase 5 explicitly carved out for performance."
**How to avoid:** planning should either (a) explicitly confirm with the user that the hot-path exemption is being lifted phase-wide (not just for `filter.ts`), or (b) carry the exemption forward for `event.ts:133`/`lists.ts:53` specifically and document why they're treated differently from the other 41 sites. Either way this needs to be a **stated decision** in the plan, not a silent default.
**Warning signs:** if the plan's task list migrates `filter.ts` but skips `event.ts:133`/`lists.ts:53` without comment, or migrates all three without addressing the perf tradeoff, that's this pitfall.

### Pitfall 2: Treating the carry-forward loop's `prev` reference incorrectly

**What goes wrong:** inside `pipeFromAsyncArray`'s `reduce`, `prev` is a `Promise` at the top of the callback (`async (prev, fn) => {...}`) and is `await`ed once, inline, as an argument (`fn(await prev)`). A carry-forward addition that does `Reflect.has(prev, symbol)` **without** awaiting first will check properties on a `Promise` object, not the resolved event — always `false`.
**Why it happens:** the existing code pattern awaits `prev` inline as a function-call argument rather than assigning it to a local first, so a naive addition copy-pastes the wrong reference.
**How to avoid:** hoist `const prevValue = await prev;` once at the top of the reduce callback, use it both for `fn(prevValue)` and the carry-forward loop.
**Warning signs:** a new test that models a symbol carried forward across a spread-heavy pipe stage passes even when the carry-forward code is commented out (vacuous test — see Validation Architecture's non-vacuity requirement).

### Pitfall 3: `groups.ts:139` gets converted to `setCachedValue` instead of deleted

**What goes wrong:** a mechanical "convert every `Reflect.set(event, Symbol, ...)` to `setCachedValue`" sweep will hit `groups.ts:139` and convert it, when the correct action (per WR-08, verified) is **deletion** — the write is already provably a no-op today (the enclosing `getOrComputeCachedValue` immediately overwrites it non-enumerable) and becomes fully redundant once D-03's "don't memoize undefined" fix gates the callback.
**Why it happens:** it's the one site in the whole inventory whose current descriptor already ends up non-enumerable, making it look like a normal migration candidate on a first grep pass.
**How to avoid:** treat `groups.ts:139` as its own line item, sequenced strictly after the `groups.ts` D-02/D-03 fix commit, action = delete the line + the 8-line comment explaining why it's harmless (`groups.ts:107-115`), not convert.

### Pitfall 4: The `{ ...draft, pubkey }` out-of-pipe spread in `modifyHiddenTags` is treated as "just needs a comment update"

**What goes wrong:** CONTEXT.md's phrasing — "audit the out-of-pipe spreads... before migrating their sources" — reads like a documentation task. It isn't, for Site 1 (`operations/tags.ts:65`). See the Architecture Patterns section above: this spread genuinely breaks `unlockHiddenTags`'s short-circuit once `HiddenTagsSymbol` becomes non-enumerable, forcing redundant decrypt round-trips. This needs an actual code fix (descriptor-preserving copy, or restructuring to avoid the spread) before `hidden-tags.ts:109`/`:157` migrate, or as part of the same commit.
**Why it happens:** the word "audit" undersells the finding.
**How to avoid:** plan a dedicated task/commit for this fix, sequenced before or alongside the `hidden-tags.ts` migration, with its own regression test (assert no redundant decrypt call occurs when hidden tags are already unlocked and `modifyHiddenTags` re-enters this branch).

### Pitfall 5: Forgetting `cast.ts:63` needs an explicit scope decision, not silent inclusion/exclusion

See the `applesauce-core` table's note above. Either outcome (migrate descriptor only / leave fully deferred) is defensible; silently doing one without recording the decision is the pitfall.

   - Recommendation: sequence these 2 sites' migration commits immediately after the `cache.test.ts` rewrite lands (not before), since they are the sites most likely to surface a carry-forward mechanism bug.
