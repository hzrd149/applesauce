## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Build on `applesauce-core`'s exported `logger` (`debug("applesauce")`, already a concord dependency). Create a concord base logger `logger.extend("concord")` and derive per-area / per-instance child loggers from it (see D-08). No new `debug` dependency is added — the `Debugger` type rides in transitively (import the type from `debug` as `import type`, matching `applesauce-wallet`).
- **D-02:** Add an optional injectable `logger?: Debugger` to the relevant options (client-level `ConcordClientOptions`; thread down so community/channel instances can be given / derive one). When injected, the client uses it as the base (extending it per-instance); when omitted, it falls back to the module `logger.extend("concord")` base. Mirrors `applesauce-wallet`'s nut-wallet `logger?: Debugger` pattern (`this.log = options.logger ?? baseLogger.extend(...)`).
- **D-03:** Full sweep. Add namespaced debug logging at every notable concord operation, not just sync: relay auth (NIP-42 lifecycle), publishes, invite lifecycle (bundle build/refresh/revoke, direct-invite watching), control/member folds, and the epoch/channel walk progression. Sync/decode is the deepest and most detailed (D-05..D-07); the rest gets lighter operational tracing.
- **D-04:** Convert the existing scattered diagnostic `console.*` calls to the namespaced logger as part of the sweep — BUT see D-09 for how genuine failures are treated (they are not silently downgraded).
- **D-05:** At each sync decode batch (per plane group, per epoch — i.e. the loops in `syncEpoch` step 2 & 3, `syncMessagePlanes`, `syncRekeyAndAdvance`), emit an **aggregate summary**: number of wraps fetched, number decoded successfully, and number dropped as undecryptable. This aggregate is what directly answers "no events vs arrived-but-failed-decrypt."
- **D-06:** ALSO emit a **per-failed-wrap line** for each wrap that `decodeWrap` drops, carrying identifying detail: wrap id, plane type + plane/stream address (the outer `ev.pubkey` / `info`), and epoch. Note: the *real* author is inside the seal we couldn't open, so it is unavailable for undecryptable wraps and is not logged — no user-identity leak by construction. `decodeWrap`/`decodeWrapCached` are left unmodified — see Deferred.
- **D-07:** Namespace split for verbosity. Aggregate summary counts (D-05) emit under the sync namespace (e.g. `applesauce:concord:…:sync`); the per-failed-wrap detail (D-06) emits under a finer child namespace (e.g. `…:sync:decode`) that can be enabled independently.
- **D-08:** Sync decode-failure logging (and all new diagnostics) is **opt-in debug** — silent unless the matching `DEBUG=applesauce:concord*` namespace is enabled. No new always-on output for the diagnostic path.
- **D-09:** Genuine operational failures (the existing `console.warn`/`console.error` sites) fire **BOTH** the namespaced debug logger AND the existing `console.warn`/`console.error`. They stay visible by default and are also captured under an injected logger. Only *pure diagnostics* become debug-namespace-only.
- **D-10:** concord is unreleased — **no changeset** for this work, and breaking-change caution does not apply.

### Claude's Discretion

- Exact namespace tree under `applesauce:concord` (working proposal: `:client`, `:community`, `:channel`, `:sync`, `:sync:decode`, `:auth`, `:invite`) and per-instance extension convention (wallet uses `.extend(pubkey.slice(0,8))`; a community-id / channel-id slice is the natural analog here).
- Exact per-wrap line fields and message format (must include enough to locate the wrap: id + plane + epoch), and the precise aggregate message wording.
- Exact set of "notable operations" that get light operational tracing in the sweep, and where the injectable `logger` needs threading (`SyncContext` / `ChannelSyncContext` must gain a logger handle for the sync-path logging).
- Whether a tiny test asserts the aggregate/per-wrap logging fires (a spy on an injected `Debugger`); logging is observability, so a light regression guard is optional but welcome.

## Summary

This phase threads an injectable, namespaced `debug`-based logger through Concord's client/community/channel hierarchy and the free sync functions, so a sync that returns zero events is distinguishable — under `DEBUG=applesauce:concord:*`— from a sync where N wraps arrived but all N failed to decrypt. Every canonical reference in CONTEXT.md (file paths, line numbers, the wallet's injectable-logger idiom, the console.* site inventory) was re-verified against the current tree and is accurate to within 1-2 lines (comment vs. code start). The wallet's `nut-wallet.ts` pattern — `this.log = options.logger ?? baseLogger.extend("nut-wallet").extend(this.pubkey.slice(0, 8))` — is a directly copyable template, and the plumbing chain for concord mirrors it exactly at four levels: `ConcordClient` → `ConcordCommunity` (via `addCommunity()`) → `ConcordPrivateChannel` (via `spawnPrivateChannel()`) → the free `SyncContext`/`ChannelSyncContext` objects built by each instance's private `syncContext()` method.

**One correction to CONTEXT.md is required before planning:** D-01 asserts "no new `debug` dependency is added — the `Debugger` type rides in transitively." This is **false** for `packages/concord` as currently configured. Empirically confirmed with `tsc --noEmit`: a scratch file in `packages/concord/src/` doing `import type { Debugger } from "debug"` fails with `TS2307: Cannot find module 'debug' or its corresponding type declarations` because concord's `package.json` does not declare `debug`/`@types/debug` and this monorepo's pnpm install produces per-package `node_modules` with no `debug` present in `packages/concord/node_modules`. `applesauce-wallet`, which CONTEXT.md cites as doing exactly this "transitively," in fact declares `debug`/`@types/debug` as its *own* direct dependencies (confirmed in `packages/wallet/package.json`) — it never actually relied on transitivity from `applesauce-core`. The planner must add `"debug": "^4.4.3"` to concord's `dependencies` and `"@types/debug": "^4.1.13"` to its `devDependencies` (matching the single version already resolved in `pnpm-lock.yaml`, so this adds zero new downloads — the package is already in the pnpm store). See Package Legitimacy Audit below.

**Primary recommendation:** Add `logger?: Debugger` to `ConcordClientOptions` and derive `this.log` there (`options.logger ?? logger.extend("concord")`), matching nut-wallet's `??` fallback idiom exactly. Thread `this.log.extend("community").extend(communityId.slice(0,8))` into each `ConcordCommunity` via the existing `addCommunity()` call site (`client.ts:567`), and `this.log.extend("channel").extend(channelId.slice(0,8))` into each `ConcordPrivateChannel` via `spawnPrivateChannel()` (`community.ts:690`). Add a required `logger: Debugger` field to `SyncContext`/`ChannelSyncContext` (built internally by each instance's own `syncContext()` — always has a real value, no need for `?`), and update the 5 test-literal `SyncContext`/`ChannelSyncContext` constructions across `sync.test.ts`/`channel-sync.test.ts` to supply one. Add `debug`/`@types/debug` as direct concord dependencies.

### Pattern 1: Injectable logger with module fallback (the wallet template)
**What:** An options field `logger?: Debugger`; the constructor resolves it against a module-level base with `??`, then extends per-instance.
**When to use:** Every class in the injection chain (`ConcordClient`, `ConcordCommunity`, `ConcordPrivateChannel`, and optionally `ConcordInviteManager`/`InviteWatcher`).
**Example:**
```typescript
// Source: packages/wallet/src/wallet/nut-wallet.ts:245 (verified current)
this.log = options.logger ?? baseLogger.extend("nut-wallet").extend(this.pubkey.slice(0, 8));
```
Concord's `ConcordClient` constructor equivalent (new code):
```typescript
// packages/concord/src/client/client.ts constructor, after existing option reads (~line 243)
this.log = options.logger ?? logger.extend("concord");
```
Note the asymmetry vs. wallet: `ConcordClient` has no natural per-instance slice at the top level (one client per logged-in user's whole session, not per-community), so it extends the module base directly with no id suffix; the id-slice pattern (`.extend(id.slice(0,8))`) belongs one level down, at `ConcordCommunity`/`ConcordPrivateChannel`, where `communityId`/`channelId` getters already exist (`community.ts:413`, `private-channel.ts:121`).

### Pattern 2: Threading the logger into a per-instance sub-object
**What:** The owning instance builds the child's options object (already the established pattern for every other cross-cutting concern in this codebase — `pool`, `relayAuth`, `eventStore` are all threaded exactly this way).
**When to use:** `ConcordClient.addCommunity()` → `ConcordCommunity`; `ConcordCommunity.spawnPrivateChannel()` → `ConcordPrivateChannel`.
**Example (verified current call site, `client.ts:567`):**
```typescript
const community = new ConcordCommunity({
  material,
  signer: this.signer,
  pubkey: this.pubkey,
  pool: this.pool,
  relayAuth: this.relayAuth,
  eventStore: this.eventStore,
  // NEW:
  logger: this.log.extend("community").extend(material.community_id.slice(0, 8)),
  // ...existing fields unchanged...
});
```
`ConcordPrivateChannel`'s equivalent call site is `community.ts:690` (`spawnPrivateChannel`), which already has `channelKey.id` in scope for the id-slice.

### Pattern 3: Decode-boundary aggregate + per-wrap logging (the crux, D-05/D-06/D-07)
**What:** At each `syncAuthors(ctx, authors)` consumption loop, count fetched vs. decoded vs. dropped, then log one aggregate line; for each dropped wrap, log one per-wrap line under a child namespace.
**When to use:** `sync.ts` `syncEpoch` step 2 (core planes, lines 136-145) and step 3 (public channel planes, lines 159-169); `channel-sync.ts` `syncMessagePlanes` (lines 50-58) and `syncRekeyAndAdvance` (lines 74-85).
**Example (illustrative, matching the existing loop shape verified at `sync.ts:136-145`):**
```typescript
// Existing loop (sync.ts:136-145), annotated with the D-05/D-06 additions:
const fetched = await syncAuthors(ctx, coreAuthors);
let decoded = 0, dropped = 0;
for (const ev of fetched) {
  const info = keys.planes.get(ev.pubkey);
  if (!info) continue;
  const d = decodeWrapCached(ev, info.convKey);
  if (!d) {
    dropped++;
    // D-06: per-wrap line, finer namespace (D-07) — no author (unavailable, D-06 by construction)
    ctx.logger.extend("decode")("dropped wrap %s plane=%s epoch=%d", ev.id.slice(0, 8), info.type, epochMaterial.root_epoch);
    continue;
  }
  decoded++;
  if (info.type === "control") (control.push(d), emit(info, d));
  else if (info.type === "guestbook") (guestbook.push(d), emit(info, d));
  else if (info.type === "dissolved") (dissolved.push(d), emit(info, d));
  else if (info.type === "rekey") (rekey.push(d), emit(info, d));
}
// D-05: aggregate summary, sync namespace
ctx.logger("core planes: fetched=%d decoded=%d dropped=%d epoch=%d", fetched.length, decoded, dropped, epochMaterial.root_epoch);
```
**Important gotcha found during verification:** `info.epoch` (from `PlaneInfo`) is `undefined` for `control`/`dissolved`/`rekey` plane types (only `channel`/`guestbook` set it — confirmed against `keys.ts:48-53`'s `PlaneInfo` shape). The per-wrap and aggregate epoch field must come from the enclosing loop's `epochMaterial.root_epoch` (always defined, already in scope in every one of the four target loops), not from `info.epoch`, or channel-scope-loop wraps will silently log `epoch=undefined`.

### Anti-Patterns to Avoid
- **Logging inside `decodeWrap`/`decodeWrapCached` (`gift-wrap.ts`):** D-06 explicitly keeps these pure/unmodified — logging belongs at each call site, not inside the shared helper, because the helper has no epoch/plane context and mixing concerns there would touch a memoized hot path shared by both sync and live routing.
- **Making `SyncContext.logger`/`ChannelSyncContext.logger` optional with `?.()` calls everywhere:** Since both contexts are only ever constructed internally by `syncContext()` methods that always resolve a real `Debugger` (never `undefined`, thanks to the `??` fallback one level up), making the field required and non-optional is more type-safe and avoids `logger?.()` calls sprinkled through 4 decode loops. The tradeoff is the 5 test-literal `SyncContext`/`ChannelSyncContext` constructions in `sync.test.ts`/`channel-sync.test.ts` need a `logger` value added (trivial: `debug("test")` or the real `logger.extend("test")`).

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Namespace-filtered opt-in logging | A custom `if (process.env.DEBUG_SYNC) console.log(...)` gate | `debug`'s `.extend()` chain + `DEBUG=applesauce:concord:*` env var | `debug` already gives glob-pattern namespace filtering, color-coded output, and %-format interpolation for free; the whole monorepo already standardizes on it |

## Common Pitfalls

### Pitfall 1: Assuming the `Debugger` type resolves transitively
**What goes wrong:** `import type { Debugger } from "debug"` fails to compile in `packages/concord` with `TS2307: Cannot find module 'debug'`.
**Why it happens:** This monorepo's pnpm install produces isolated per-package `node_modules`; a package that does not declare `debug` in its own `package.json` cannot resolve it, even for a type-only import, even though `applesauce-core` (a concord dependency) itself depends on `debug`. `applesauce-core`'s own re-exported `logger` value works fine without this (its type is inferred structurally, no explicit `Debugger` import needed at the call site) — but the moment concord code wants to *declare* a field typed `Debugger` (D-02's `logger?: Debugger` on `ConcordClientOptions` etc.), the explicit type import is required and fails.
**How to avoid:** Add `"debug": "^4.4.3"` to `packages/concord/package.json`'s `dependencies` and `"@types/debug": "^4.1.13"` to `devDependencies` before writing any `import type { Debugger } from "debug"` line, then run `pnpm install` so the symlink appears in `packages/concord/node_modules`.
**Warning signs:** `tsc --noEmit -p packages/concord/tsconfig.json` reporting `TS2307` on the `debug` import; this is a hard compile error, not a lint warning, so it will surface immediately in the phase's first task.

### Pitfall 2: Breaking the 5 existing `SyncContext`/`ChannelSyncContext` test literals
**What goes wrong:** Adding a required `logger: Debugger` field to `SyncContext`/`ChannelSyncContext` without updating `sync.test.ts:144,238,272` and `channel-sync.test.ts:122,183` causes 5 TypeScript errors (object literal missing property).
**Why it happens:** These tests construct the context interfaces directly (dependency-injected, no real `ConcordCommunity`/`ConcordPrivateChannel` instance involved) — they bypass the `??`-fallback constructor logic entirely.
**How to avoid:** Add `logger: logger.extend("test")` (or any `Debugger` instance, e.g. `debug("test")`) to each of the 5 literals in the same task that changes the interface.
**Warning signs:** `pnpm exec tsc --noEmit -p tsconfig.json` failing inside `packages/concord` after the interface change but before the test files are touched.

### Pitfall 3: `info.epoch` being `undefined` for non-channel/guestbook plane types
**What goes wrong:** A per-wrap or aggregate log line reads `epoch=undefined` for control/dissolved/rekey plane drops.
**Why it happens:** `PlaneInfo.epoch` (`keys.ts:48-53`) is only set for `channel` and `guestbook` plane types; control/dissolved/rekey planes don't carry a per-wrap epoch on `info`.
**How to avoid:** Always source the epoch for a log line from the enclosing loop's known epoch value (`epochMaterial.root_epoch` in `sync.ts`'s `syncEpoch`, or `channel.epoch`/`keys.current.epoch` in `channel-sync.ts`), never from `info.epoch` alone.
**Warning signs:** A manual test run with `DEBUG=applesauce:concord:*:sync:decode` showing `epoch=undefined` in output for a dropped control-plane wrap.

### Pitfall 4: Threading `logger` but forgetting `ConcordRelayAuth`'s positional constructor
**What goes wrong:** `relay-auth.ts`'s existing `console.warn` sites (lines 148, 195) are in scope for D-09's dual-emit sweep, but `ConcordRelayAuth`'s constructor is `constructor(private readonly pool: RelayPool) {}` — positional, no options object, constructed once by `ConcordClient` (`client.ts:243`, `new ConcordRelayAuth(options.pool)`) and shared across every community.
**Why it happens:** Unlike `ConcordCommunityOptions`/`ConcordPrivateChannelOptions`/`ConcordInviteManagerOptions`, `ConcordRelayAuth` was never given an options-object constructor, so there's no natural injection point without either (a) changing its constructor signature (adds a second positional param, a small breaking-shape change — fine per D-10, concord is unreleased) or (b) using the module-level `logger.extend("concord").extend("auth")` directly inside `relay-auth.ts` without threading the injected instance logger at all (simpler, but means an app that injects a custom `logger` into `ConcordClientOptions` won't see relay-auth's traces routed through it).
**How to avoid:** This is a genuine open design fork for the planner — see Open Questions below. Given D-03 only asks for "light operational tracing" here (not the deep D-05/D-06/D-07 treatment), the module-level-only approach is the lower-risk/lower-diff option and is what this research recommends, with the injected-logger threading noted as a possible follow-up if this proves insufficient.

## Open Questions

1. **Should `ConcordRelayAuth` gain a logger parameter, or use the module-level default only?**
   - What we know: D-03 asks for "light operational tracing" on `relay-auth.ts`'s two `console.warn` sites (lines 148, 195); `ConcordRelayAuth` is constructed once per `ConcordClient` (shared across every community) with a positional-only constructor (`constructor(private readonly pool: RelayPool) {}`).
   - What's unclear: Whether an app that injects a custom `logger` into `ConcordClientOptions` expects `relay-auth.ts`'s traces to be captured by that injected instance, or whether the module-default `logger.extend("concord").extend("auth")` is acceptable since this is only "light" tracing, not the deep sync/decode treatment.
   - Recommendation: Use the module-level default (`logger.extend("concord").extend("auth")`) directly inside `relay-auth.ts`, no constructor change. This is the smaller, lower-risk diff, consistent with D-03's "lighter operational tracing" framing for non-sync areas, and NIP-42 auth-lifecycle deep instrumentation is explicitly deferred to backlog Phase 999.4 anyway.

2. **Should `SyncContext.logger`/`ChannelSyncContext.logger` be required or optional?**
   - What we know: Both contexts are only ever constructed internally by each instance's own `syncContext()` method (`community.ts:565`, `private-channel.ts:220`), which always resolves a real `Debugger` via the `??` chain one level up — so `undefined` is never a real runtime possibility.
   - What's unclear: Whether the planner prefers the type-safety of a required field (touches 5 test literals) or the zero-test-diff convenience of an optional field with `ctx.logger?.(...)` calls sprinkled through 4 decode loops.
   - Recommendation: Required (see Pattern 3's Anti-Patterns note) — it's more type-safe, the call-site diff is smaller (no `?.` scattered through hot loops), and the test-literal fix is a trivial one-line addition per site.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|----------------|---------|-------------------|
| V7 Error Handling and Logging | yes | Sensitive data (decrypted rumor plaintext, private/symmetric keys, seals) MUST NOT appear in any new log line — D-06 already enforces this for the per-wrap line by construction (the real author is unavailable for a failed decode, so it can't leak); the sweep (D-03/D-04) must extend the same discipline to every *successful* decode's light operational tracing too (no log line prints `decoded.rumor.content`, `keys.*.pk`/private material, or `convKey` bytes) |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|----------------------|
| Sensitive-data-in-logs (a debug line accidentally prints a decrypted rumor's content, a group symmetric key, or a signer's private key) | Information Disclosure | Log only identifying-but-non-secret fields: wrap id (public, already on the relay), plane type, epoch, pubkey/stream address (already public — it's the outer wrap's `pubkey`, not the real author). Never log `decoded.rumor.content`, `decoded.seal`, `info.convKey`, or any `ConcordKeys`/`ChannelKeys` field. Since `debug` output is opt-in (D-08) but can still land in a shipped app's console/log aggregator if a dev enables it in production, treat every new log line as if it *will* be captured somewhere persistent |
| Log-based DoS / unbounded volume (an attacker floods a public relay with malformed wraps addressed to a community's public stream key, each triggering a per-wrap `:sync:decode` log line) | Denial of Service | Out of scope for this phase's mitigation (D-08 already gates this behind an explicit opt-in namespace a developer must deliberately enable — it's diagnostic tooling, not always-on production logging), but worth a one-line planner note: the per-wrap line (D-06) is O(dropped-wrap-count) per sync batch, same order as the existing (unlogged) `continue` loop, so this phase doesn't change the *cost* of a decode-flood attack, only whether a developer watching `:sync:decode` sees it |
