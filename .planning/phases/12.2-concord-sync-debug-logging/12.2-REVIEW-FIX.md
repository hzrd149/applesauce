## Fixed Issues

### CR-01: Rekey-plane decode drops are logged with the wrong plane name

**Files modified:** `packages/concord/src/client/channel-sync.ts`
**Commit:** `87501c3a`
**Applied fix:** Both dropped-wrap `decodeLogger` sites now pass `info.type`
instead of the hardcoded `"channel"` literal. In `syncRekeyAndAdvance` the loop
is narrowed to `info.type === "rekey"`, so drops there now correctly read
`plane=rekey`; `syncMessagePlanes` still emits `plane=channel` but derives it,
so the two sites cannot drift again.

### CR-02: Full invite unlock token printed to the console

**Files modified:** `packages/concord/src/client/community.ts`
**Commit:** `bfcb2bec`
**Applied fix:** `refreshInviteBundles`'s per-link failure `console.warn` now
interpolates `link.signerPubkey` (public, on relays) rather than `link.token`
(the 128-bit secret keying the kind-33301 bundle). Added a comment at the site
stating the token must never be logged.

### WR-01: `fetched` is not reconcilable against `decoded + dropped`

**Files modified:** `packages/concord/src/client/sync.ts`,
`packages/concord/src/client/channel-sync.ts`
**Commit:** `1629c110`
**Applied fix:** All four decode loops now increment a `skipped` counter on the
pre-decode `continue` (plane-lookup miss / type mismatch) instead of falling
through silently, and every aggregate format string gained `skipped=%d`. The
four affected lines are `core planes`, `public channels`, `message planes`, and
`rekey plane`; each now satisfies `fetched = decoded + dropped + skipped`. The
existing `sync-logging.test.ts` guard asserts on substrings, so it continues to
pass unchanged.

### WR-02: Channel/rekey-plane drops log the root epoch, not the plane's own epoch

**Files modified:** `packages/concord/src/client/sync.ts`,
`packages/concord/src/client/private-channel.ts`
**Commit:** `0ace826b`
**Applied fix:** Both sites now use `info.epoch ?? <enclosing epoch>` — the same
rule `community.ts:563-565` and `private-channel.ts`'s own `route()` already
apply — so the same dropped wrap reports an identical `epoch=` whether it
arrived via sync or via the live subscription.
**Status: fixed — requires human verification** (changes a logged *value*; no
test asserts the epoch a dropped channel wrap reports).

### WR-03: `.extend("sync")` called inline in `syncContext()`

**Files modified:** `packages/concord/src/client/community.ts`,
`packages/concord/src/client/private-channel.ts`
**Commit:** `d6332327`
**Applied fix:** Added a `private readonly syncLog: Debugger` field to both
classes, derived once in the constructor; `decodeLog` now chains off it
(`this.syncLog.extend("decode")`, same resulting namespace) and `syncContext()`
returns the stored field. No `.extend("sync")` call remains outside a
constructor, which restores the accuracy of `SyncContext`'s doc comment and
honors the mid-phase "avoid inline debug.extend()" convention.

### WR-04: "user auth succeeded" logged even when the relay rejected the AUTH

**Files modified:** `packages/concord/src/client/relay-auth.ts`
**Commit:** `77e18190`
**Applied fix:** `autoAuthenticate` now branches on `res?.ok`, emitting
`user auth rejected` on an `OK false` — matching the stream-key path's existing
behavior. Also converted `.then().catch()` into the two-argument
`.then(onFulfilled, onRejected)` form so a throw inside the success handler is
no longer swallowed and misreported as an AUTH failure. `.finally` still clears
the in-flight set.
**Status: fixed — requires human verification** (alters promise-handler
composition, not just log text).

### WR-05: A prefix of the secret invite token is logged where a public identifier exists

**Files modified:** `packages/concord/src/client/invite-manager.ts`,
`packages/concord/src/client/community.ts`
**Commit:** `99c4ca7a`
**Applied fix:** Both sites switched from `token.slice(0, 8)` to
`signerPubkey.slice(0, 8)`, and the `invite-manager` line's key was renamed
`token=` → `link=` to match `community.ts:1215`'s convention. A grep confirms no
token prefix is logged anywhere in `packages/concord/src` outside tests.

### WR-06: Community name is logged verbatim as user content

**Files modified:** `packages/concord/src/client/client.ts`
**Commit:** `2e084aba`
**Applied fix:** `createNewCommunity` now logs
`"creating community relays=%d"` with the resolved relay count instead of the
unbounded user-supplied `name`. The community id is already traced on the
success line a few statements later.

### WR-07: The new base logger is unreachable from outside the package

**Files modified:** `packages/concord/src/index.ts`,
`packages/concord/src/__tests__/exports.test.ts`
**Commit:** `f9d7589f`
**Applied fix:** Added `export * from "./logger.js";` to the package index,
mirroring `applesauce-core`. This surfaces `logger` so consumers can derive
their own namespace to pass as `ConcordClientOptions.logger` without
hand-constructing the `applesauce:concord` string. The `exports.test.ts` inline
snapshot was regenerated to include the new `logger` symbol (the only test
affected). No `./logger` subpath was added to `package.json`'s `exports` map —
the root re-export is the supported access path, matching core.

### WR-08: `rekey plane epoch=` reports the source epoch for an epoch+1 address

**Files modified:** `packages/concord/src/client/channel-sync.ts`
**Commit:** `3be218f9`
**Applied fix:** The aggregate format string became
`"rekey plane from_epoch=%d to_epoch=%d …"`, emitting `channel.epoch` and
`channel.epoch + 1`. The values are unchanged in meaning; the line is now
self-describing during `reReadHeldChannelEpochs`'s backward pass.
