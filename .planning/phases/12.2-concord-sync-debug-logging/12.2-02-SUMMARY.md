# Phase 12.2 Plan 02: Concord Sync/Decode Debug Logging Summary

Instrumented all 4 sync-walk decode batches (`sync.ts`'s `syncEpoch` steps 2/3, `channel-sync.ts`'s `syncMessagePlanes`/`syncRekeyAndAdvance`) and the 2 live `onWrap` decode sites (`community.ts`, `private-channel.ts`) with always-on aggregate `fetched/decoded/dropped/epoch` logging under `:sync` and per-dropped-wrap detail under `:sync:decode` — the litmus test that makes a zero-event sync (`fetched=0`) read differently from an arrived-but-undecryptable sync (`fetched=N decoded=0 dropped=N` plus N per-wrap lines).

## Performance

- **Duration:** ~25 min
- **Tasks:** 3 (plus 1 orchestrator-requested refactor commit)
- **Files modified:** 6 (4 source, 2 test)

## Accomplishments

- `sync.ts`'s `syncEpoch` step 2 (core planes: control/guestbook/dissolved/rekey) and step 3 (public channels) each emit an always-on aggregate line — fires even when `fetched.length === 0`, the D-05 litmus signal — plus one per-dropped-wrap `:sync:decode` line (wrap id, plane type, epoch) before the pre-existing `continue`.
- `channel-sync.ts`'s `syncMessagePlanes` and `syncRekeyAndAdvance` mirror the same aggregate + per-wrap shape, sourcing epoch from `channel.epoch` (the enclosing loop's known value) rather than `info.epoch`.
- `community.ts`'s and `private-channel.ts`'s `onWrap` live-decode sites (single-event, no aggregate — per RESEARCH) each emit one per-wrap `:sync:decode` line on a null decode, logging only the same non-secret identifying fields as the walk path (never `decoded.rumor.content`, `convKey`, or seal bytes).
- Zero decode/routing/sync semantics changed — every new line wraps an existing `continue`/`if (decoded)` branch; the branches themselves, their order, and their side effects are untouched.

## Decisions Made

- **Derive-once logger discipline** adopted as the standing pattern for this phase's remaining plans (03/04): a namespace extended for repeated use inside a loop or hot path must be derived exactly once (a class field set in the constructor, or once per `syncContext()`/context-builder call) — never `.extend()`d again at the individual log call site. One-time `.extend()` at true construction time (deriving `this.log`, or `logger: this.log.extend("sync")` passed once into a child options object) remains fine and untouched.
- **Epoch source discipline**: every new log line's epoch comes from the enclosing scope's already-known value, never `info.epoch` — confirmed against `keys.ts`'s `deriveConcordKeys`: control/dissolved planes carry no `epoch` field at all, and the base-rekey (`rekey`) plane's `info.epoch` is the *next* epoch (the one being rotated *to*), not the current epoch being synced. Using `info.epoch` for the aggregate/per-wrap lines would silently misreport or `undefined`-out the epoch for exactly the planes this phase most needs correct.

## Deviations from Plan

### Auto-fixed Issues

**1. [Course correction — orchestrator-directed, applied as Rule-1-style bugfix] Inline `.extend()` at per-wrap log call sites**
- **Found during:** Post-Task-3, orchestrator relayed a code-review finding against the wave-1 pattern this plan's Tasks 1–3 followed literally from the plan text (`ctx.logger.extend("decode")(...)` inside decode loops; `this.log.extend("sync").extend("decode")(...)` at the two live `onWrap` sites).
- **Issue:** `debug`'s `Debugger.extend()` allocates a new `Debugger` instance and re-runs namespace enable-matching (`debug.enabled(namespace)`) on every call — calling it once per dropped wrap inside a hot loop is wasteful, and repeats the `"decode"` string literal at every call site instead of naming the derived logger once.
- **Fix:** Added a required `decodeLogger: Debugger` field to `SyncContext` (inherited by `ChannelSyncContext`), derived once per `syncContext()` build in `community.ts`/`private-channel.ts` from a new `private readonly decodeLog` class field set once in each constructor (`this.log.extend("sync").extend("decode")`). `sync.ts`/`channel-sync.ts`'s decode loops now call `ctx.decodeLogger(...)` directly; the two live `onWrap` sites call `this.decodeLog(...)`. Same namespaces, same message strings, same output — this changes call *frequency* of `.extend()`, not logging behavior.
- **Files modified:** `packages/concord/src/client/sync.ts`, `packages/concord/src/client/channel-sync.ts`, `packages/concord/src/client/community.ts`, `packages/concord/src/client/private-channel.ts`, `packages/concord/src/client/__tests__/sync.test.ts`, `packages/concord/src/client/__tests__/channel-sync.test.ts`
- **Verification:** `tsc --noEmit` exits 0; `pnpm --filter applesauce-concord test` — 47 files / 289 tests, all green; `pnpm run build` clean; `grep -rn '\.extend(' <the 4 source files>` shows every remaining `.extend()` call is one-time (constructor or per-`syncContext()`-build), none inside a loop.
- **Committed in:** `2f43cf45` (separate `refactor(12.2-02)` commit, per orchestrator instruction — not folded into the 3 task commits)

**Note on the plan's literal acceptance-criteria greps:** Tasks 1–3's `<acceptance_criteria>` blocks specify literal greps like `grep -c 'ctx.logger.extend("decode")'` returning 2 (sync.ts) — these were true immediately after each task's own commit, but the refactor commit (above) intentionally changes that literal string to `ctx.decodeLogger(` / `this.decodeLog(`, so re-running those exact greps against the final tree now returns 0. The underlying truths those criteria were checking for — one per-wrap detail line per decode loop, on the correct `:sync:decode` namespace, epoch never sourced from `info.epoch` — all still hold; verified via the equivalent `decodeLogger`/`decodeLog` call-count greps recorded above instead.

**Note on the `info.epoch` acceptance-criteria greps:** `grep -c 'info.epoch'` across `sync.ts` and `community.ts`/`private-channel.ts` returns non-zero (2 in `sync.ts`, 1 each in `community.ts`/`private-channel.ts`) — every hit is **pre-existing, unrelated code** this plan does not touch: `sync.ts`'s `planeStoreKey` function (keys the epoch-scoped guestbook store, Phase 6 origin) and `community.ts`/`private-channel.ts`'s `route()` method (the CORD-03 channel/epoch binding check, both predate this plan). None of this plan's *new* logging lines source epoch from `info.epoch` — confirmed by inspecting each new line's epoch argument directly (`epochMaterial.root_epoch`, `channel.epoch`, `channelEpochOf(...)`/`this.keys.material.root_epoch`, `this.channelKey.epoch`).
