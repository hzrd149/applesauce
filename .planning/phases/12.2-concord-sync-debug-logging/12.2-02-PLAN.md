<objective>
The crux of phase 12.2: make a decryption failure during sync observable. At each sync
decode batch, count wraps fetched / decoded / dropped and emit one aggregate summary (D-05)
under the `:sync` namespace, plus one per-failed-wrap line (D-06) under the finer
`:sync:decode` namespace (D-07). Apply the same shape to the two live-subscription
`onWrap` decode sites. All logging is opt-in debug (D-08) and adds ZERO semantic change —
it wraps the existing `if (!d) continue;` / `if (decoded) route(...)` branches.

Purpose: This directly delivers the litmus test — `DEBUG=applesauce:concord:*:sync` makes a
zero-event sync (`fetched=0`) read differently from an arrived-but-undecryptable sync
(`fetched=N decoded=0 dropped=N` plus N `:sync:decode` lines).
Output: Instrumented decode loops in sync.ts, channel-sync.ts, and the onWrap sites in
community.ts / private-channel.ts.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| relay -> sync/onWrap decode | Untrusted wraps arrive; a dropped (undecryptable) wrap's contents stay opaque and must never be reconstructed into a log line. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-12.2-01 | Information Disclosure | per-wrap / aggregate log lines | medium | mitigate | Log only wrap id, plane type, epoch, and the outer public `ev.pubkey` — never `decoded.rumor.content`, `info.convKey`, seal bytes, or any ConcordKeys/ChannelKeys field. For dropped wraps the real author is inside the unopened seal and is unavailable by construction (D-06), so identity cannot leak. |
| T-12.2-02 | Denial of Service | per-wrap :sync:decode line | low | accept | O(dropped) lines per batch — same order as the pre-existing (unlogged) continue loop; gated behind an opt-in namespace (D-08), so this phase does not change the cost of a decode-flood, only its visibility. |
</threat_model>
