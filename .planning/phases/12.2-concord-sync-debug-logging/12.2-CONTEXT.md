# Phase 12.2: Concord Sync Debug Logging - Context

**Gathered:** 2026-07-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Concord's community and private-channel sync walks decode gift wraps via
`decodeWrapCached(ev, info.convKey)`, which returns `null` on any decode failure
and is silently `continue`/skipped (`sync.ts:140,166`, `channel-sync.ts:57,81`,
`community.ts:534`, `private-channel.ts:197`). That collapses "relay returned
nothing" and "wraps arrived but wouldn't decrypt" into the same observable
outcome. This phase makes concord emit **namespaced debug logging across the
board — most specifically for syncing** so a decryption failure during sync is
observable rather than silent, letting a developer tell "no events" apart from
"events arrived but could not be decrypted."

**In scope:** a shared namespaced logger for the concord package; a full sweep of
notable operations (sync/decode, auth, publish, invite lifecycle, folds) with
debug logging; and — the crux — decode-boundary logging in the sync paths that
reports fetched/decoded/dropped counts plus per-failed-wrap detail.

**Out of scope:** any change to sync *semantics* (barrier, synced-status,
rekey/fold logic) — this is purely observability. Do not alter decode/routing
behavior; only add logging around it. No changeset (concord unreleased).
</domain>

<decisions>
## Implementation Decisions

### Logging mechanism & injectability
- **D-01:** Build on `applesauce-core`'s exported `logger` (`debug("applesauce")`,
  already a concord dependency). Create a concord base logger
  `logger.extend("concord")` and derive per-area / per-instance child loggers
  from it (see D-08). No new `debug` dependency is added — the `Debugger` type
  rides in transitively (import the type from `debug` as `import type`, matching
  `applesauce-wallet`).
- **D-02:** Add an optional injectable `logger?: Debugger` to the relevant
  options (client-level `ConcordClientOptions`; thread down so community/channel
  instances can be given / derive one). When injected, the client uses it as the
  base (extending it per-instance); when omitted, it falls back to the module
  `logger.extend("concord")` base. Mirrors `applesauce-wallet`'s nut-wallet
  `logger?: Debugger` pattern (`this.log = options.logger ?? baseLogger.extend(...)`).

### Breadth — "across the board" (full sweep)
- **D-03:** Full sweep. Add namespaced debug logging at every notable concord
  operation, not just sync: relay auth (NIP-42 lifecycle), publishes, invite
  lifecycle (bundle build/refresh/revoke, direct-invite watching), control/member
  folds, and the epoch/channel walk progression. Sync/decode is the deepest and
  most detailed (D-05..D-07); the rest gets lighter operational tracing.
- **D-04:** Convert the existing scattered diagnostic `console.*` calls to the
  namespaced logger as part of the sweep — BUT see D-09 for how genuine failures
  are treated (they are not silently downgraded).

### Decode-boundary observability (the crux)
- **D-05:** At each sync decode batch (per plane group, per epoch — i.e. the loops
  in `syncEpoch` step 2 & 3, `syncMessagePlanes`, `syncRekeyAndAdvance`), emit an
  **aggregate summary**: number of wraps fetched, number decoded successfully, and
  number dropped as undecryptable. This aggregate is what directly answers "no
  events vs arrived-but-failed-decrypt."
- **D-06:** ALSO emit a **per-failed-wrap line** for each wrap that `decodeWrap`
  drops, carrying identifying detail: wrap id, plane type + plane/stream address
  (the outer `ev.pubkey` / `info`), and epoch. Note: the *real* author is inside
  the seal we couldn't open, so it is unavailable for undecryptable wraps and is
  not logged — no user-identity leak by construction. (We chose per-wrap
  *pinpointing* over cracking `decodeWrap`'s collapsed `null` into distinct
  reasons; `decodeWrap`/`decodeWrapCached` are left unmodified — see Deferred.)
- **D-07:** Namespace split for verbosity. Aggregate summary counts (D-05) emit
  under the sync namespace (e.g. `applesauce:concord:…:sync`); the per-failed-wrap
  detail (D-06) emits under a finer child namespace (e.g. `…:sync:decode`) that
  can be enabled independently, so counts stay readable and per-wrap detail is
  on-demand rather than flooding whenever sync logging is on.

### Level / visibility
- **D-08:** Sync decode-failure logging (and all new diagnostics) is **opt-in
  debug** — silent unless the matching `DEBUG=applesauce:concord*` namespace is
  enabled. "Observable" here means "available when you look," consistent with
  `applesauce-core`/`signers`. No new always-on output for the diagnostic path.
- **D-09:** Genuine operational failures (the existing `console.warn`/
  `console.error` sites: publish failed, "invite list exceeds the NIP-44 byte
  cap; not publishing", "failed to persist direct invites", "Failed to add rumor
  to … store", etc.) fire **BOTH** the namespaced debug logger AND the existing
  `console.warn`/`console.error`. They stay visible by default (the `debug`
  package has no level tiers, so routing them through it alone would silence
  them) and are also captured under an injected logger. Only *pure diagnostics*
  become debug-namespace-only.

### Milestone conventions carried forward
- **D-10:** concord is unreleased — **no changeset** for this work, and
  breaking-change caution does not apply (this is additive/observability-only
  anyway). See [[concord-unreleased-no-changesets]].

### Claude's Discretion
- Exact namespace tree under `applesauce:concord` (working proposal:
  `:client`, `:community`, `:channel`, `:sync`, `:sync:decode`, `:auth`,
  `:invite`) and per-instance extension convention (wallet uses
  `.extend(pubkey.slice(0,8))`; a community-id / channel-id slice is the natural
  analog here).
- Exact per-wrap line fields and message format (must include enough to locate
  the wrap: id + plane + epoch), and the precise aggregate message wording.
- Exact set of "notable operations" that get light operational tracing in the
  sweep, and where the injectable `logger` needs threading (`SyncContext` /
  `ChannelSyncContext` must gain a logger handle for the sync-path logging).
- Whether a tiny test asserts the aggregate/per-wrap logging fires (a spy on an
  injected `Debugger`); logging is observability, so a light regression guard is
  optional but welcome.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `logger` from `applesauce-core` (`logger.extend(...)`) — the base logger; no new
  dependency needed since `applesauce-core` is already a concord dep.
- `applesauce-wallet` nut-wallet is a working template for the whole shape:
  injectable `logger?: Debugger`, `?? baseLogger.extend(...)` fallback,
  per-instance `.extend(id.slice(0,8))`, and passing the logger into a sync
  loader.
- `decodeWrapCached`'s `null` return is the exact silent-drop signal the counts
  are computed from — no helper change required; count at the call site.

### Established Patterns
- `syncAuthors` centralizes backfill fetch; the decode loops that consume its
  results (`syncEpoch`, `syncMessagePlanes`, `syncRekeyAndAdvance`) are the
  natural aggregation boundaries for fetched/decoded/dropped counts.
- Existing concord error reporting is always-on `console.warn`/`console.error`;
  D-09 keeps that visible while adding logger capture — no regression in what a
  developer currently sees.
- The `debug` package is single-level (a namespace is on or off) — this drives
  D-09 (dual-emit for real failures) and D-07 (namespace split for verbosity)
  rather than log-level tiers.

### Integration Points
- `SyncContext` / `ChannelSyncContext` must gain a logger handle so the free
  sync functions can log; the community/channel instances construct the child
  logger and pass it in (alongside the existing `route`, `ensureAuth`, etc.).
- `ConcordClientOptions` is the single injection point; per-community and
  per-channel loggers extend from there (or from the module base when omitted).

</code_context>

<specifics>
## Specific Ideas

The observable litmus test: with `DEBUG=applesauce:concord:*:sync` set, a sync
that returns zero events must read differently from a sync where N wraps arrived
but all N failed to decrypt — the first shows "fetched 0", the second shows
"fetched N, decoded 0, dropped N" plus N per-wrap lines (under `:sync:decode`).
That contrast is the whole point of the phase.

</specifics>

<deferred>
## Deferred Ideas

- **Cracking `decodeWrap`'s collapsed `null` into distinct failure reasons**
  (not-a-wrap / undecryptable-under-convKey / bad-seal-signature / author-binding
  mismatch) so the *why* of a decode failure is logged, not just the *that*. We
  chose per-wrap pinpointing (D-06) over reason-cracking this phase; surfacing
  structured reasons would touch `gift-wrap.ts`'s pure helpers and is a separate,
  larger change.
- **Synced-status / sync-barrier semantics rework** — reporting how "synced" a
  plane is. Carried over from Phase 12.1's deferred list; still out of scope
  (this phase is observability only, not semantics).
- **NIP-42 lifecycle debug logging** as its own dedicated effort — ROADMAP
  backlog Phase 999.4. The full sweep (D-03) touches `relay-auth.ts` for
  operational tracing, but a deep auth-lifecycle instrumentation remains its own
  backlog item.

</deferred>
