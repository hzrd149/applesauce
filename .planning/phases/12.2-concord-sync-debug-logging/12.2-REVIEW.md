# Phase 12.2: Code Review Report

**Reviewed:** 2026-07-22
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 12.2 added namespaced `debug` logging across `packages/concord`. The control-flow
audit is clean: every `console.*` call in the diff range is preserved byte-for-byte
(verified with `git diff 5a024bc5^..HEAD | grep '^[-+].*console\.'` — 14 removals, 14
matching re-additions inside the widened arrow bodies), no loop was short-circuited, no
`continue`/`return` was moved, and every format string's `%s`/`%d` placeholder count
matches its argument count at all ~75 call sites. `debug` is correctly declared as a
runtime dependency with `@types/debug` in devDependencies.

The defects are in the *content* of the logs, not the surrounding control flow — which
matters, because this phase's entire deliverable is the trustworthiness of those lines.
Two are blocking: the private-channel rekey-plane decode-drop line reports a **provably
wrong plane name**, and a pre-existing-but-in-range `console.warn` prints a **complete
invite unlock token** (the only secret guarding an invite bundle that carries
`community_root`). Beyond that, the aggregate `fetched/decoded/dropped` triple silently
omits a whole category of skipped wraps, so `fetched` is not reconcilable against
`decoded + dropped` — a debugger reading `fetched=10 decoded=0 dropped=0` is told nothing
about where the other ten went.

Hot-path cost is fine: every logged argument is a `.slice(0, 8)`, a `Map` lookup, or an
integer counter. No `JSON.stringify`, no array map/filter at a call site.

## Critical Issues

### CR-01: Rekey-plane decode drops are logged with the wrong plane name

**File:** `packages/concord/src/client/channel-sync.ts:107`
**Issue:** `syncRekeyAndAdvance` filters its loop to rekey-plane wraps only
(`if (!info || info.type !== "rekey") continue;` at :99), yet its dropped-wrap log
hardcodes the string literal `"channel"` for `plane=%s`:

```ts
if (!info || info.type !== "rekey") continue;
const decoded = decodeWrapCached(ev, info.convKey);
if (decoded) { ... } else {
  dropped++;
  ctx.decodeLogger("dropped wrap=%s plane=%s epoch=%d", ev.id.slice(0, 8), "channel", channel.epoch);
}
```

Every dropped **rekey** wrap is emitted as `plane=channel`, indistinguishable from a
dropped message wrap logged by `syncMessagePlanes` at :65. This is the single
highest-value drop in the whole engine: an undecryptable rekey wrap is why a private
channel silently stops advancing epochs, and the phase exists specifically to make that
observable. Instead the line actively misdirects. The correct value is in scope as
`info.type` (which is narrowed to `"rekey"` at that point), exactly as `sync.ts:154`/`:197`
and `private-channel.ts:217` already do.

Line 65 is not wrong (that loop is narrowed to `"channel"`), but it hardcodes the same
literal and should use `info.type` so the two sites cannot drift again.

**Fix:**
```ts
// channel-sync.ts:65
ctx.decodeLogger("dropped wrap=%s plane=%s epoch=%d", ev.id.slice(0, 8), info.type, channel.epoch);

// channel-sync.ts:107
ctx.decodeLogger("dropped wrap=%s plane=%s epoch=%d", ev.id.slice(0, 8), info.type, channel.epoch);
```

### CR-02: Full invite unlock token printed to the console

**File:** `packages/concord/src/client/community.ts:1209`
**Issue:** The per-link failure path in `refreshInviteBundles` interpolates the entire
token into a `console.warn`:

```ts
console.warn(`invite refresh skipped for link ${link.token}`, err);
```

`link.token` is `bytesToHex(newInviteToken())` — the full 16 random bytes
(`helpers/invite-bundle.ts:134-136`) that key `inviteBundleKey(token)` and are the **only**
secret protecting the kind-33301 bundle. The bundle event itself is public on relays, and
its plaintext carries `community_root`, `owner_salt`, `root_epoch`, and any granted
private-channel keys. Anyone who reads this console line (a pasted bug report, a shipped
browser console, a log aggregator) can decrypt that community's join material for the life
of the link. Note the sibling `publishLog` line immediately above it (:1204-1208) already
truncates to 8 chars, so the two co-located emissions disagree on how secret this value is.

This line predates 12.2 (introduced by `10c46b29 fix(10-04)`), but it is inside the review
diff range, is co-located with a 12.2 addition, and is a live credential disclosure.

**Fix:** Log the public link identity instead — `invite.signerPubkey` is already the
convention at `community.ts:1215` and uniquely identifies the link.
```ts
this.publishLog("invite refresh skipped for link=%s: %s", link.signerPubkey.slice(0, 8), (err as Error)?.message ?? err);
console.warn(`invite refresh skipped for link ${link.signerPubkey}`, err);
```

## Warnings

### WR-01: `fetched` is not reconcilable against `decoded + dropped` — skipped wraps are invisible

**File:** `packages/concord/src/client/sync.ts:149-150, 192-193`; `packages/concord/src/client/channel-sync.ts:57-58, 98-99`
**Issue:** All four decode loops `continue` *before* incrementing either counter when the
plane lookup misses or the type doesn't match:

```ts
const info = keys.planes.get(ev.pubkey);
if (!info) continue;                                  // sync.ts:149-150
if (!info || info.type !== "channel") continue;       // sync.ts:192-193
```

So `coreFetched.length` counts wraps that were never even attempted, and the emitted
aggregate can read `fetched=10 decoded=0 dropped=0` — which a reader will interpret as
"ten wraps arrived and were all fine", when in fact ten wraps were discarded before the
decode boundary. This is precisely the ambiguity the D-05 litmus (zero-event sync vs.
arrived-but-undecryptable sync) was meant to remove, and it leaves the third failure mode —
"arrived at an address we don't have a plane for" — completely unobservable.

**Fix:** Count the skips and report them, so the triple always sums to `fetched`:
```ts
let coreSkipped = 0;
...
if (!info) { coreSkipped++; continue; }
...
ctx.logger(
  "core planes epoch=%d fetched=%d decoded=%d dropped=%d skipped=%d",
  epochMaterial.root_epoch, coreFetched.length, coreDecoded, coreDropped, coreSkipped,
);
```

### WR-02: Channel/rekey-plane drops log the root epoch, not the plane's own epoch

**File:** `packages/concord/src/client/sync.ts:197`; `packages/concord/src/client/private-channel.ts:217`
**Issue:** `sync.ts:197` sits inside the public-channel loop (narrowed to
`info.type === "channel"`) but logs `epochMaterial.root_epoch`. `private-channel.ts:217`
logs `this.channelKey.epoch` for *every* plane type including `"rekey"`, whose addresses
are derived at `epoch + 1`. Meanwhile `community.ts:563-565` deliberately does the
opposite for the same event class, with a comment citing "RESEARCH Pitfall 3":

```ts
const epoch = info.type === "channel" ? channelEpochOf(this.keys, info.channelId!) : this.keys.material.root_epoch;
```

The same dropped channel wrap therefore reports a different `epoch=` depending on whether
it arrived via sync or via the live subscription, which defeats correlating the two.
`PlaneInfo.epoch` (`helpers/keys.ts:52`) is populated for channel planes and available at
both sites.

**Fix:** Prefer the plane's own epoch with the enclosing epoch as fallback:
```ts
// sync.ts:197
ctx.decodeLogger("dropped wrap=%s plane=%s epoch=%d", ev.id.slice(0, 8), info.type, info.epoch ?? epochMaterial.root_epoch);
// private-channel.ts:217
this.decodeLog("dropped wrap=%s plane=%s epoch=%d", canonical.id.slice(0, 8), info.type, info.epoch ?? this.channelKey.epoch);
```

### WR-03: `.extend("sync")` called inline in `syncContext()`, contradicting its own doc comment

**File:** `packages/concord/src/client/community.ts:609`; `packages/concord/src/client/private-channel.ts:256`
**Issue:** `SyncContext.decodeLogger`'s doc comment (`sync.ts:88-92`) states both loggers
are "derived ONCE by `syncContext()`... never re-`.extend()`d ... since `Debugger.extend()`
allocates and re-runs namespace enable-matching on every call". `decodeLogger` honors that
(it reads the stored `this.decodeLog` field), but `logger` does not:

```ts
logger: this.log.extend("sync"),   // community.ts:609, private-channel.ts:256
decodeLogger: this.decodeLog,
```

`syncContext()` is not a one-shot: `community.ts` calls it in `start()` (:450) **and** in
`reconcileLive()` (:677, once per batch of newly revealed public channels);
`private-channel.ts` calls it in `walk()` (:171) **and** in `catchUpCurrent()` (:329, once
per adopted rekey). Each call allocates a fresh `Debugger` and re-runs namespace matching.
This is the exact regression the mid-phase convention (`9137de81 docs: plant seed — avoid
inline debug.extend()`) was created to prevent, and it makes the doc comment inaccurate.

**Fix:** Store it beside `decodeLog` in both constructors:
```ts
private readonly syncLog: Debugger;
// constructor
this.syncLog = this.log.extend("sync");
this.decodeLog = this.syncLog.extend("decode");
// syncContext()
logger: this.syncLog,
```

### WR-04: "user auth succeeded" is logged even when the relay rejected the AUTH

**File:** `packages/concord/src/client/relay-auth.ts:211-213`
**Issue:** The user-auth path logs success from a bare `.then()`, which fires for any
resolved promise regardless of the result payload:

```ts
relay.authenticate(signer)
  .then(() => log("user auth succeeded pubkey=%s relay=%s", pubkey.slice(0, 8), url))
```

`relay.authenticate` resolves with an `{ ok }` result — the stream-key path 60 lines above
(:152-157) correctly branches on `res.ok` and emits either `auth succeeded` or
`auth rejected`. Here a relay that answers `OK false` ("restricted: ...") is reported as a
success, so the one log line an operator would grep to explain "why am I not authenticated"
lies. Secondary: `.then()` now sits ahead of `.catch()`, so a throw from inside the log
call would be swallowed and re-reported as an AUTH failure.

**Fix:**
```ts
.then((res) =>
  log(res?.ok ? "user auth succeeded pubkey=%s relay=%s" : "user auth rejected pubkey=%s relay=%s",
      pubkey.slice(0, 8), url),
)
```

### WR-05: A prefix of the secret invite token is logged where a public identifier exists

**File:** `packages/concord/src/client/invite-manager.ts:186`; `packages/concord/src/client/community.ts:1204-1208`
**Issue:** Both 12.2-added lines log `invite.token.slice(0, 8)` / `link.token.slice(0, 8)`.
The token is 128 bits of secret key material (`randomBytes(16)`), not an event id. The
existing 8-char truncation convention exists for *public* identifiers (event ids, pubkeys,
community ids) to keep lines readable — applying it to a secret borrows the form without
the justification. The residual 96 bits means this is not directly exploitable, which is
why it is a warning rather than a blocker, but there is no reason to put any of it in a log:
`invite.signerPubkey` uniquely identifies the same link and is already public on relays
(and is what `community.ts:1215` uses).

**Fix:**
```ts
// invite-manager.ts:186
this.log("revoking invite link=%s community=%s", invite.signerPubkey.slice(0, 8), invite.communityId.slice(0, 8));
```

### WR-06: Community name is logged verbatim as user content

**File:** `packages/concord/src/client/client.ts:456`
**Issue:** `this.log("creating community name=%s", name);` emits an unbounded,
user-controlled string into the log. Every other identifier in this package is a
truncated hex id; this is the only place raw user content crosses the logging boundary, in
a package whose whole premise is that community content is end-to-end encrypted. It is
also unbounded — a long name produces an arbitrarily long log line. The community id is
available two lines later (:471) and is the stable handle a developer would actually
correlate on.

**Fix:**
```ts
this.log("creating community relays=%d", relays.length);
// the id is already traced at :471 once `createCommunity` resolves
```

### WR-07: The new base logger is unreachable from outside the package

**File:** `packages/concord/src/logger.ts:6`; `packages/concord/src/index.ts`; `packages/concord/package.json:25-91`
**Issue:** `export const logger = coreLogger.extend("concord")` is exported from the module
but `src/index.ts` never re-exports it, and `package.json`'s `exports` map has no
`./logger` subpath. `applesauce-core` does the opposite (`packages/core/src/index.ts:7`:
`export * from "./logger.js"`), which is how `logger.ts:1` is able to import it. The result
is that a consumer wanting to derive their own namespace to pass as
`ConcordClientOptions.logger` — the injection point this phase added and documented on
every options interface — has no supported way to obtain the base. They must either
hand-construct `debug("applesauce:concord")` (duplicating the namespace string) or reach
into `applesauce-concord/dist/logger.js`, which is not in the exports map.

**Fix:** Mirror core — add `export * from "./logger.js";` to `packages/concord/src/index.ts`.

### WR-08: `rekey plane epoch=` reports the source epoch for an epoch+1 address

**File:** `packages/concord/src/client/channel-sync.ts:111-117`
**Issue:** The aggregate reads
`"rekey plane epoch=%d fetched=%d decoded=%d dropped=%d"` with `channel.epoch`, but the
authors it just synced are `keys.nextRekey` — the addresses for `channel.epoch + 1`
(see the function's own doc comment at :79-81, "the channel-rekey address(es) for
`channel`'s next epoch"). During the `reReadHeldChannelEpochs` backward pass this function
is invoked repeatedly with different held epochs, so a reader tracing a stalled rotation
across those lines cannot tell whether `epoch=3` means "read the rekey plane at 3" or "read
the rekey plane that advances 3→4".

**Fix:** Disambiguate in the format string rather than changing the value:
```ts
ctx.logger(
  "rekey plane from_epoch=%d to_epoch=%d fetched=%d decoded=%d dropped=%d",
  channel.epoch, channel.epoch + 1, fetched.length, decodedCount, dropped,
);
```
