## Decisions Made

- Hoisted `client.ts`'s two repeated `this.log.extend("invite")` derivations into a single `this.inviteLog` field, per this run's mandatory-convention "ADDITIONAL CLEANUP" instruction; the `addCommunity()` per-community `.extend("community").extend(id.slice(0,8))` chain was deliberately left as-is, since the same instruction explicitly carves out derivations keyed by a varying instance id.
- `sync-logging.test.ts`'s spy fixture supplies `SyncContext.logger`/`decodeLogger` as two independent spies rather than one spy exposing `.extend("decode")` — the plan's Task 3 action text describes the pre-12.2-02 shape (`ctx.logger.extend("decode")`), which the plan-02 course-correction already superseded with a required `decodeLogger` field. The fixture matches the actual current `SyncContext` interface.
- Deliberately-undecryptable test wraps are sealed under the control plane's real `convKey` (so the seal step succeeds) but wrapped under a different, WRONG `convKey` (so the outer wrap decrypt step — the one `decodeWrapCached` actually performs first — fails). This keeps the wrap's pubkey/kind correct (so it's genuinely fetched and routed to the right plane), matching VALIDATION.md's Wave-0 "sealed under a different convKey than the loop derives" note precisely.
- Chose light, transition-only tracing (walk start/complete, fold adopt/removed, newly-revealed-channel catch-up) rather than per-event tracing inside hot subscription callbacks (e.g. `rewireState`'s fold subscribe), to keep the new lines operationally useful without adding noise to a namespace that fires on every control/guestbook emission.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - missing critical / convention-driven] Hoisted client.ts's two repeated `this.log.extend("invite")` call sites into a single field**
- **Found during:** Task 1
- **Issue:** The mandatory logging convention reiterated in this run's prompt explicitly called out `client.ts`'s constructor (`ConcordInviteManager` options) and `ensureInviteWatcher()` as two separate `this.log.extend("invite")` derivations for the same fixed namespace — a duplication the convention forbids, since `ensureInviteWatcher()` can run again after a `stop()`/restart re-derives the same namespace.
- **Fix:** Added a `private readonly inviteLog: Debugger` field, derived once in the constructor (`this.log.extend("invite")`), and pointed both sites at it.
- **Files modified:** `packages/concord/src/client/client.ts`
- **Verification:** `tsc --noEmit` exits 0; `client.test.ts` (27 tests) passes unchanged; `grep -n '.extend(' client.ts` shows only the one-time constructor derivations plus the per-community-id chain the convention explicitly says to leave alone.
- **Committed in:** `fbdd183f` (Task 1 commit)

**2. [Rule 1-adjacent — interface-drift correction, not a plan literal-text following issue] Test fixture built against the CURRENT `SyncContext.decodeLogger` field, not the plan's stale `.extend("decode")` description**
- **Found during:** Task 3
- **Issue:** The plan's Task 3 action text describes building a spy Debugger whose `.extend("decode")` call is captured (`ctx.logger.extend("decode")(...)`) — the shape `SyncContext` had BEFORE 12.2-02's course-correction. The actual current interface (since 12.2-02) already carries a separate, required `decodeLogger: Debugger` field precisely so nothing calls `.extend()` inline at a decode call site.
- **Fix:** The test fixture supplies two independent spy `Debugger`s (`aggregate.log` for `ctx.logger`, `decode.log` for `ctx.decodeLogger`) rather than one spy exposing a working `.extend()`. This matches production code exactly and still fully exercises both the aggregate and per-wrap logging paths the plan's acceptance criteria describe.
- **Files modified:** `packages/concord/src/client/__tests__/sync-logging.test.ts`
- **Verification:** `tsc --noEmit` exits 0; the test passes and, per the non-vacuity check below, correctly fails when the wraps are actually decryptable.
- **Committed in:** `b669a886` (Task 3 commit)
